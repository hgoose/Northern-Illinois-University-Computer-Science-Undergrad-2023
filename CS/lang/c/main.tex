    \unsect{Input / output}
    \begin{itemize}
        \item \textbf{stdio.h}: In C, input/output is mostly handled through the standard I/O library, declared in:
            \bigbreak \noindent 
            \begin{cppcode}
            #include <std.io>
            \end{cppcode}
        \item \textbf{Standard streams}: When a C program starts, it automatically has three standard streams:
            \bigbreak \noindent 
            \begin{cppcode}
            stdin
            stdout
            stderr
            \end{cppcode}
        \item \textbf{printf}: Printf prints formatted text to stdout.
            \bigbreak \noindent 
            \begin{cppcode}
            printf("%s %s", "Hello, ", "world!");
            \end{cppcode}
            \bigbreak \noindent 
            The placeholders are called format specifies. Common ones include
            \bigbreak \noindent 
            \begin{cppcode}
                %d      int
                %u      unsigned int
                %ld     long
                %lu     unsigned long
                %lld    long long
                %llu    unsigned long long

                %f      double
                %lf     double in scanf, but also accepted in printf
                %c      char
                %s      string / char array
                %p      pointer address
                %%      literal percent sign
            \end{cppcode}
            \bigbreak \noindent 
            For \%p, cast the pointer to void*. Note that the language itself does not make printf typesafe. Wrong format specifiers cause \textbf{undefined behavior}.
        \item \textbf{fprintf}: The fprintf function in C is used to write formatted data directly to an output stream, which is most commonly a file. It works identically to printf, except it requires a destination file pointer as its very first argument. 
            \bigbreak \noindent 
            \begin{cppcode}
            int fprintf(FILE *stream, const char *format, ...);
            \end{cppcode}
        \item \textbf{sprintf}: The sprintf function in C formats and stores a series of characters and values into a string buffer instead of printing them directly to the console. It is defined in the <stdio.h> standard library.
            \bigbreak \noindent 
            \begin{cppcode}
            int sprintf(char *str, const char *format, ...); 
            \end{cppcode}
            \bigbreak \noindent 
            Returns the total number of characters successfully written, excluding the null terminator (\textbackslash0). If an error occurs, it returns a negative value. 
        \item \textbf{snprintf}: 
            \bigbreak \noindent 
            \begin{cppcode}
            int snprintf ( char * s, size_t n, const char * format, ... );
            \end{cppcode}
            \bigbreak \noindent 
            Composes a string with the same text that would be printed if format was used on printf, but instead of being printed, the content is stored as a C string in the buffer pointed by s (taking n as the maximum buffer capacity to fill).
            \bigbreak \noindent 
            If the resulting string would be longer than n-1 characters, the remaining characters are discarded and not stored, but counted for the value returned by the function.
            \bigbreak \noindent 
            A terminating null character is automatically appended after the content written.
            \bigbreak \noindent 
            Returns the number of characters that would have been written if $n$ had been sufficiently large, not counting the terminating null character. If an encoding error occurs, a negative number is returned.
            \bigbreak \noindent 
            The key rule is \textbf{size} is the total buffer size, including space for the null terminator.
            \bigbreak \noindent 
            \begin{cppcode}
                char buf[10];
                snprintf(buf, sizeof buf, "hello");
            \end{cppcode}
            \bigbreak \noindent 
            sizeof buf is 10, so snprintf may write at most 9 visible characters plus the final '\textbackslash0'.
            \bigbreak \noindent 
            \begin{cppcode}
                char buf[6];
                snprintf(buf, sizeof buf, "hello");
            \end{cppcode}
            \bigbreak \noindent 
            This fits exactly. So, this is safe. 
            \bigbreak \noindent 
            \begin{cppcode}
                char buf[5];

                int n = snprintf(buf, sizeof buf, "hello");
            \end{cppcode}
            \bigbreak \noindent 
            snprintf writes
            \bigbreak \noindent 
            \begin{cppcode}
            'h' 'e' 'l' 'l' '\0'
            \end{cppcode}
            \bigbreak \noindent 
            in this case the return value is still $5$, because "hello" would have required 5 visible characters, not counting the null terminator. snprintf returns the number of characters that would have been written, excluding the null terminator.
            \bigbreak \noindent 
            \begin{cppcode}
                char buf[5];

                int n = snprintf(buf, sizeof buf, "hello");

                if (n < 0) {
                    // encoding or formatting error
                } else if ((size_t)n >= sizeof buf) {
                    // output was truncated
                } else {
                    // output fit
                }
            \end{cppcode}
            \bigbreak \noindent 
            You can call snprintf once to find out how much space is needed:
            \bigbreak \noindent 
            \begin{cppcode}
                int n = snprintf(NULL, 0, "value = %d", 123);
            \end{cppcode}
        \item \textbf{scanf}: scanf reads formatted input from stdin.
            \bigbreak \noindent 
            \begin{cppcode}
            int x;
            scanf("%d", &x)
            \end{cppcode}
            \bigbreak \noindent 
            Scanf needs the address because it has to modify the variable. Scanf returns the number of successful conversions. You should usually check the return value
            \bigbreak \noindent 
            \begin{cppcode}
            int x;
            if (scanf("%d", &x) != 1) {
                fprintf(stderr, "Invalid integer\n");
            }
            \end{cppcode}
        \item \textbf{puts}: Writes a string to the console
            \bigbreak \noindent 
            \begin{cppcode}
            puts("Hello, world!");
            \end{cppcode}
        \item \textbf{getchar}: reads one character from stdin
            \bigbreak \noindent 
            \begin{cppcode}
            int c;
            c = getchar();
            \end{cppcode}
            \bigbreak \noindent 
            Notice that getchar returns int, not char. That is because it may return EOF, which is a special value indicating end-of-file or error.
            \bigbreak \noindent 
            \begin{cppcode}
            int c;
            while ((c=getchar()) != EOF) putchar(c);
            \end{cppcode}
        \item \textbf{putchar}: Writes one character to stdout
            \bigbreak \noindent 
            \begin{cppcode}
            putchar('A');
            \end{cppcode}
        \item \textbf{fgets for line input}:  Avoid
            \bigbreak \noindent 
            \begin{cppcode}
                char name[32];
                scanf("%s", name);
            \end{cppcode}
            \bigbreak \noindent 
            If the user types more than 31 characters, you can overflow the buffer. A safer approach is fgets:
            \bigbreak \noindent 
            \begin{cppcode}
                char name[32];

                printf("Enter your name: ");

                if (fgets(name, sizeof name, stdin) == NULL) {
                    fprintf(stderr, "Input error\n");
                    return 1;
                }

                printf("Hello, %s", name);
            \end{cppcode}
            \bigbreak \noindent 
            Fgets has signature
            \bigbreak \noindent 
            \begin{cppcode}
            char* fgets( char* str, int count, FILE* stream );
            \end{cppcode}
            \bigbreak \noindent 
            and returns \textbf{str} on success, null pointer on failure.
        \item \textbf{sscanf}: The sscanf function in C extracts and parses formatted data directly from a character string rather than reading from standard keyboard input (stdin). It is defined in the <stdio.h> library and stands for "String Scan Formatted". 
            \bigbreak \noindent 
            \begin{cppcode}
            int sscanf(const char *str, const char *format, ...);
            \end{cppcode}
            \bigbreak \noindent 
            \begin{cppcode}
                const char *info = "ID:4023 Alice 28";
                int id;
                char name[20];
                int age;

                // Parse data by explicitly matching the literal "ID:" prefix
                int items = sscanf(info, "ID:%d %19s %d", &id, name, &age);
            \end{cppcode}
            \begin{itemize}
                \item \textbf{\%[set]:} A scanset. For instance, \%[A-Za-z] reads only alphabetic characters.
                \item \textbf{\%[$\land$set]:} An inverted scanset. For instance, \%[$\land$,] reads everything up until it hits a comma.
                \item \textbf{* Modifier:} Suppresses assignment. For example, \%*d matches and skips an entire integer without storing it.
            \end{itemize}


    \end{itemize}

    \subsection{Working with files}
    \begin{itemize}
        \item \textbf{Header}: File I /O is mainly done through the same header
            \bigbreak \noindent 
            \begin{cppcode}
            #include <stdio.h>
            \end{cppcode}
        \item \textbf{FILE type}: The central type is
            \bigbreak \noindent 
            \begin{cppcode}
            FILE*
            \end{cppcode}
            \bigbreak \noindent 
            A FILE* is a handle to an open file stream. You do not manipulate the file directly. You open it, get a FILE *, use functions such as fgetc, fgets, fprintf, fread, and finally close it with fclose.
        \item \textbf{fopen}: 
            \bigbreak \noindent 
            \begin{cppcode}
            FILE *fopen(const char *restrict filename, const char *restrict mode);
            \end{cppcode}
            \bigbreak \noindent 
            Upon successful completion fopen(), fdopen(), and freopen() return a FILE pointer.  Otherwise, NULL is returned and errno is set to indicate the error.
            \bigbreak \noindent 
            The mode string tells C whether you want to read, write, append, or work in binary mode. 
            \bigbreak \noindent 
            \begin{center}
                \begin{tabularx}{\textwidth}{@{}lX@{}}
                    \toprule
                    \textbf{Mode}	&\textbf{Meaning} \\
                    \midrule
                    "r"	    &Open existing file for reading\\[2ex]
                    "w"	    &Open file for writing; destroys old contents\\[2ex]
                    "a"	    &Open file for appending\\[2ex]
                    "r+"	&Open existing file for reading and writing\\[2ex]
                    "w+"	&Open for reading and writing; destroys old contents\\[2ex]
                    "a+"	&Open for reading and appending\\[2ex]
                    "rb"	&Read binary\\[2ex]
                    "wb"	&Write binary\\[2ex]
                    "ab"	&Append binary \\
                    \bottomrule
                \end{tabularx}
            \end{center}
            \bigbreak \noindent 
            The primary difference between r+ and a+ modes in C file handling lies in how they handle file creation and where the data is written. While both allow you to read and write without erasing (truncating) the existing data, they behave very differently under the hood. 
            \bigbreak \noindent 
            \begin{center}
                \begin{tabularx}{\textwidth}{@{}XXX@{}}
                    \toprule
                    \textbf{Feature}& \textbf{r+ (Read / Update)}&	\textbf{a+ (Append / Update)} \\
                    \midrule
                    File Must Exist?	&Yes (returns NULL if missing)	&No (creates a new file if missing)\\[2ex]
                    Initial Pointer Position	&Beginning of the file	&End of the file\\[2ex]
                    Overwrites Content?	&Yes (overwrites at current cursor)	&No (always forces writes to EOF)\\[2ex]
                    Effect of fseek() on Writes	&Moves the cursor for the next write	&Ignored (writes always go to the end) \\ 
                    \bottomrule
                \end{tabularx}
            \end{center}
            \bigbreak \noindent 
            \begin{cppcode}
                #include <stdio.h>

                int main(void) {
                    FILE *fp = fopen("data.txt", "r");

                    if (fp == NULL) {
                        perror("fopen");
                        return 1;
                    }

                    fclose(fp);
                    return 0;
                }
            \end{cppcode}
        \item \textbf{fclose}: 
            \bigbreak \noindent 
            \begin{cppcode}
            int fclose(FILE* stream);
            \end{cppcode}
            \bigbreak \noindent 
            fclose closes an open file stream. It also flushes any buffered output before closing the file.
            \bigbreak \noindent 
            You should call fclose on every file you successfully opened with fopen.
            \bigbreak \noindent 
            Upon successful completion, 0 is returned.  Otherwise, EOF is returned and errno is set to indicate the error.
            \bigbreak \noindent 
            EOF is a negative integer constant defined in <stdio.h>.
            \bigbreak \noindent 
            \begin{cppcode}
            #include <stdio.h>

            int main(void) {
                FILE *fp = fopen("output.txt", "w");

                if (fp == NULL) {
                    perror("fopen");
                    return 1;
                }

                fprintf(fp, "Hello file\n");

                if (fclose(fp) == EOF) {
                    perror("fclose");
                    return 1;
                }

                return 0;
            }
            \end{cppcode}
        \item \textbf{The restrict keyword}: restrict is a type qualifier used on pointers to promise the compiler that the pointer is the only means to access its underlying memory object
            \bigbreak \noindent 
            When you qualify a pointer with restrict, you guarantee that for the lifetime of that pointer, no other pointer or variable inside the same scope will read or write to that specific chunk of memory.
        \item \textbf{fprintf}: 
            \bigbreak \noindent 
            \begin{cppcode}
            int fprintf(FILE *restrict stream, const char *restrict format, ...);
            \end{cppcode}
            \bigbreak \noindent 
            fprintf works like printf, but it writes to a file stream instead of standard output.
            \bigbreak \noindent 
            Returns the number of characters written on success.
            \bigbreak \noindent 
            Returns a negative value on failure.
            \bigbreak \noindent 
            \begin{cppcode}
                #include <stdio.h>

                int main(void) {
                    FILE *fp = fopen("numbers.txt", "w");

                    if (fp == NULL) {
                        perror("fopen");
                        return 1;
                    }

                    int x = 10;
                    double y = 3.14;

                    fprintf(fp, "x = %d\n", x);
                    fprintf(fp, "y = %.2f\n", y);

                    fclose(fp);
                    return 0;
                }
            \end{cppcode}
        \item \textbf{fputs}: 
            \bigbreak \noindent 
            \begin{cppcode}
                int fputs(const char *restrict s, FILE *restrict stream);
            \end{cppcode}
            \bigbreak \noindent 
            fputs writes a string to a file. Unlike puts, it does not automatically add a newline.
            \bigbreak \noindent 
            Returns a non-negative value on success. Returns EOF on failure.
        \item \textbf{fputc}
            \bigbreak \noindent 
            \begin{cppcode}
            int fputc(int c, FILE *stream);
            \end{cppcode}
            \bigbreak \noindent 
            fputc writes a single character to a file. The character is passed as an int, but it usually represents a char.
            \bigbreak \noindent 
            Returns the character written on success. Returns EOF on failure.
        \item \textbf{fgetc}: 
            \bigbreak \noindent 
            \begin{cppcode}
            int fgetc(FILE *stream);
            \end{cppcode}
            \bigbreak \noindent 
            fgetc reads one character from a file.
            \bigbreak \noindent 
            It returns an int, not a char, because it needs to represent every possible character plus the special value EOF.
            \bigbreak \noindent 
            Returns the character read on success. Returns EOF if the end of the file is reached or if an error occurs.
            \bigbreak \noindent 
            \begin{cppcode}
                #include <stdio.h>

                int main(void) {
                    FILE *fp = fopen("data.txt", "r");

                    if (fp == NULL) {
                        perror("fopen");
                        return 1;
                    }

                    int ch;

                    while ((ch = fgetc(fp)) != EOF) {
                        putchar(ch);
                    }

                    fclose(fp);
                    return 0;
                }
            \end{cppcode}
        \item \textbf{fgets}: 
            \begin{cppcode}
                char *fgets(char *restrict s, int n, FILE *restrict stream);
            \end{cppcode}
            \bigbreak \noindent 
            fgets reads a line, or part of a line, from a file into a character buffer.
            \bigbreak \noindent 
            It reads at most $n - 1$ characters and then adds the null terminator '\textbackslash 0'.
            \bigbreak \noindent 
            If it sees a newline before the limit, it includes the newline in the buffer.
            \bigbreak \noindent 
            \begin{cppcode}
                #include <stdio.h>

                int main(void) {
                    FILE *fp = fopen("data.txt", "r");

                    if (fp == NULL) {
                        perror("fopen");
                        return 1;
                    }

                    char line[256];

                    while (fgets(line, sizeof line, fp) != NULL) {
                        printf("%s", line);
                    }

                    fclose(fp);
                    return 0;
                }
            \end{cppcode}
            \bigbreak \noindent 
            This is the normal way to read a text file line by line in C.
        \item \textbf{fscanf}: 
            \bigbreak \noindent 
            \begin{cppcode}
                int fscanf(FILE *restrict stream, const char *restrict format, ...);
            \end{cppcode}
            \bigbreak \noindent 
            fscanf reads formatted input from a file, similar to how scanf reads from standard input.
            \bigbreak \noindent 
            It is useful for simple structured input, such as integers, floats, and words.
            \bigbreak \noindent 
            However, fscanf can be fragile for complicated input. For many real programs, reading a full line with fgets and then parsing it manually is safer.
            \bigbreak \noindent 
            Returns the number of input items successfully matched and assigned.
            \bigbreak \noindent 
            Returns EOF if the end of the file is reached before any conversion happens, or if a read error occurs.
        \item \textbf{fputs}


    \end{itemize}

    \subsection{Binary I/O}
    \begin{itemize}
        \item \textbf{fwrite}
            \bigbreak \noindent 
            \begin{cppcode}
                size_t fwrite(const void *restrict data_ptr, size_t element_size, size_t n_elements, FILE *restrict stream); 
            \end{cppcode}
            \bigbreak \noindent 
            Writes raw binary data from memory into a file. To write one \texttt{int}:
            \bigbreak \noindent 
            \begin{cppcode}
            fwrite(&x, sizeof x, 1, fp);
            \end{cppcode}
            \bigbreak \noindent 
            To write an array:
            \bigbreak \noindent 
            \begin{cppcode}
                fwrite(arr, sizeof arr[0], count, fp)
            \end{cppcode}
            \bigbreak \noindent 
            Returns the number of characters written on success.
            \bigbreak \noindent 
            Returns a negative value on failure.
        \item \textbf{fread}:
            \bigbreak \noindent 
            \begin{cppcode}
            size_t fread(void ptr[restrict size * n], size_t size, size_t n, FILE *restrict stream);
            \end{cppcode}
            \bigbreak \noindent 
            The function fread() reads n items of data, each size bytes long, from the stream pointed to by stream, storing them at the location given by ptr.
            \bigbreak \noindent 
            On success, fread() and fwrite() return the number of items read or written.  This number equals the number of bytes transferred only when size is 1.  If an error occurs, or the end of the file is reached, the return value is a short item count (or zero).


    \end{itemize}

    \subsection{I/O helpers}
    \subsubsection{Flushing}
    \begin{itemize}
        \item \textbf{fflush}
            \bigbreak \noindent 
            \begin{cppcode}
            int fflush(FILE* stream);
            \end{cppcode}
            \bigbreak \noindent 
            C file output is often buffered. That means data may sit in memory temporarily before being physically written to the file. fflush forces buffered output to be written immediately. It is mainly used for output streams.
            \bigbreak \noindent 
            Returns 0 on success. Returns EOF on failure.
    \end{itemize}

    \subsubsection{File positioning}
    \begin{itemize}
        \item \textbf{Intro}: Every open file stream has a current position. Reading or writing usually advances that position.
            \bigbreak \noindent 
            You can move around in a file using fseek, ftell, and rewind.        
        \item \textbf{fseek}: 
            \bigbreak \noindent 
            \begin{cppcode}
                int fseek(FILE* stream, long offset, int whence);
            \end{cppcode}
            \bigbreak \noindent 
            fseek moves file position. Whence controls where the offset is measured from:
            \begin{itemize}
                \item \textbf{SEEK\_SET}: Beginning of file
                \item \textbf{SEEK\_CUR}: Current position
                \item \textbf{SEEK\_END}: End of file
            \end{itemize}
            Returns zero on success, nonzero on failure.
            \bigbreak \noindent 
            \begin{cppcode}
                #include <stdio.h>

                int main(void) {
                    FILE *fp = fopen("data.txt", "r");

                    if (fp == NULL) {
                        perror("fopen");
                        return 1;
                    }

                    if (fseek(fp, 0, SEEK_END) != 0) {
                        perror("fseek");
                        fclose(fp);
                        return 1;
                    }

                    fclose(fp);
                    return 0;
                }
            \end{cppcode}
            \bigbreak \noindent 
            Moves position to EOF
        \item \textbf{ftell}: 
            \bigbreak \noindent 
            \begin{cppcode}
            long ftell(FILE* stream);
            \end{cppcode}
            \bigbreak \noindent 
            returns the current file position. For binary files, this is commonly used as a byte offset.
            \bigbreak \noindent 
            Returns current position on success, -1L on failure.
            \bigbreak \noindent 
            \begin{cppcode}
            #include <stdio.h>

            int main(void) {
                FILE *fp = fopen("data.txt", "r");

                if (fp == NULL) {
                    perror("fopen");
                    return 1;
                }

                fseek(fp, 0, SEEK_END);

                long size = ftell(fp);

                if (size == -1L) {
                    perror("ftell");
                    fclose(fp);
                    return 1;
                }

                printf("File size: %ld bytes\n", size);

                fclose(fp);
                return 0;
            }
            \end{cppcode}
        \item \textbf{rewind}:
            \bigbreak \noindent 
            \begin{cppcode}
            void rewind(FILE* stream);
            \end{cppcode}
            \bigbreak \noindent 
            reset the file position indicator in a stream




    \end{itemize}

    \subsubsection{EOF and Errors}
    \begin{itemize}
        \item \textbf{feof}: 
            \bigbreak \noindent 
            \begin{cppcode}
               int feof(FILE *stream);
            \end{cppcode}
            \bigbreak \noindent 
            The feof() function shall test the end-of-file indicator for the stream pointed to by stream.
            \bigbreak \noindent 
            The feof() function shall not change the setting of errno if stream is valid.
            \bigbreak \noindent 
            The feof() function shall return non-zero if and only if the end-of-file indicator is set for stream, otherwise returns zero.
        \item \textbf{ferror}
            \bigbreak \noindent 
            \begin{cppcode}
            int ferror(FILE* stream);
            \end{cppcode}
            \bigbreak \noindent 
            ferror checks whether an error occurred on a file stream.
            \bigbreak \noindent 
            This is useful because some functions, such as fgetc, return EOF both for real end-of-file and for errors.
            \bigbreak \noindent 
            Returns nonzero if the error indicator is set, zero otherwise.
        \item \textbf{clearerr}
            \bigbreak \noindent 
            \begin{cppcode}
                void clearerr(FILE* stream);
            \end{cppcode}
            \bigbreak \noindent 
            clearerr clears both the error indicator and the EOF indicator for a stream.
            \bigbreak \noindent 
            You usually do not need it in simple programs, but it matters if you want to continue using a stream after EOF or after handling an error.
        \item \textbf{perror}: 
            \bigbreak \noindent 
            \begin{cppcode}
            void perror(const char* s);
            \end{cppcode}
            \bigbreak \noindent 
            perror prints an error message describing the most recent library/system error.
            \bigbreak \noindent 
            It is commonly used after functions such as fopen, fread, fwrite, fflush, or fclose fail.

    \end{itemize}

    \unsect{Strings}
    \begin{itemize}
        \item \textbf{String literals}: Consider
            \bigbreak \noindent 
            \begin{cppcode}
            char* s1 = "Hello, world!";
            const char* s2 = "Hello, world!";
            \end{cppcode}
            \bigbreak \noindent 
            These create read only string literals. You should always make string literals const. Attempting to modify an non-const string literal  will cause a crash.
            \bigbreak \noindent 
            To make them mutable, attach them to a char array
            \bigbreak \noindent 
            \begin{cppcode}
                char arr[] = "Hello, world!";
            \end{cppcode}
            \bigbreak \noindent 
            The compiler allocates a fresh block of writable memory on the stack (or global scope) and copies the contents of "Hello" into it. This memory is completely safe to change.


        \item \textbf{string.h}: The main C header for working with strings. Contains the standard C functions for working with null-terminated strings and raw memory blocks.
        \item \textbf{strlen}
            \bigbreak \noindent 
            \begin{cppcode}
                size_t strlen(const char* s);
            \end{cppcode}
            \bigbreak \noindent 
            Returns the length of a C string, not counting the null terminator.
            \bigbreak \noindent 
            \begin{cppcode}
                char s[] = "Hello";

                sizeof s // 6, includes \0
                strlen(s) // 5, excludes \0
            \end{cppcode}
        \item \textbf{strcmp}: 
            \bigbreak \noindent 
            \begin{cppcode}
            int strcmp(const char *s1, const char *s2);
            \end{cppcode}
            \bigbreak \noindent 
            Compares two strings lexicographically.
            \bigbreak \noindent 
            A negative return value implies s1 is less than s2. A return value of zero implies s1 equals s2. A positive return value implies s1 is greater than s2.
            \bigbreak \noindent 
            \textbf{Note:} Strings \textbf{cannot} be compared with ==. Using this operator simply compares addresses.
            \bigbreak \noindent 
            Because strcmp does not accept a maximum length argument, it evaluates characters sequentially until it detects a null-terminator (\textbackslash 0). If a maliciously crafted or malformed string lacks a null-terminator,
            \begin{itemize}
                \item Keep reading past the assigned memory buffer.
                \item Trigger undefined program behavior.
                \item Cause a segmentation fault (SIGSEGV) or application crash if it tries to read protected memory spaces.
                \item Potentially read adjacent data from the stack or heap, giving attackers a mechanism to bleed sensitive information
            \end{itemize}
        \item \textbf{strncmp}: 
            \bigbreak \noindent 
            \begin{cppcode}
            int strncmp(const char* s1, const char* s2, size_t n);
            \end{cppcode}
            \bigbreak \noindent 
            Compares at most the first $n$ characters of two strings. 
            \bigbreak \noindent 
            Has same return behavior as \texttt{strcmp}.
        \item \textbf{strcpy}: 
            \bigbreak \noindent 
            \begin{cppcode}
            char* strcpy(char* dest, const char* src);
            \end{cppcode}
            \bigbreak \noindent 
            Copies the string src into dest, including the null terminator.
            \bigbreak \noindent 
            Returns dest.
        \item \textbf{strncpy}
            \bigbreak \noindent 
            \begin{cppcode}
            char *strncpy(char *dest, const char *src, size_t n);
            \end{cppcode}
            \bigbreak \noindent 
            Copies at most $n$ characters from src to dest.
            \bigbreak \noindent 
            Returns dest.
            \bigbreak \noindent 
            This function is \textbf{not} simply a safer strcpy. If src has length greater than or equal to $n$, strncpy does \textbf{not} append a null-terminator.
            \bigbreak \noindent 
            \begin{cppcode}
                #include <stdio.h>
                #include <string.h>

                int main(void) {
                    char dest[6];

                    strncpy(dest, "hello world", sizeof dest - 1);
                    dest[sizeof dest - 1] = '\0';

                    printf("%s\n", dest);

                    return 0;
                }
            \end{cppcode}
        \item \textbf{strcat}: 
            \bigbreak \noindent 
            \begin{cppcode}
            char *strcat(char *dest, const char *src);
            \end{cppcode}
            \bigbreak \noindent 
            Appends src to the end of dest. Starts at the null-terminator of dest. Dest must have a null-terminator and enough room for the result.
            \bigbreak \noindent 
            Returns dest.
        \item \textbf{strncat}: 
            \bigbreak \noindent 
            \begin{cppcode}
            char *strncat(char *dest, const char *src, size_t n);
            \end{cppcode}
            \bigbreak \noindent 
            Appends at most $n$ characters from src to dest, then appends a null-terminator.
            \bigbreak \noindent 
            Returns dest
        \item \textbf{strchr}: 
            \bigbreak \noindent 
            \begin{cppcode}
            char* strchr(const char* s, int c);
            \end{cppcode}
            \bigbreak \noindent 
            Finds the first occurrence of character $c$ is string $s$.
            \bigbreak \noindent 
            Returns pointer to first occurrence of character $c$ in string $s$, or NULL if not found.
        \item \textbf{strrchr}
            \bigbreak \noindent 
            \begin{cppcode}
            char* strrchr(const char* s, int c);
            \end{cppcode}
            \bigbreak \noindent 
            Finds the last occurrence of character $c$ is string $s$. 
            \bigbreak \noindent 
            Same return value as strchr
        \item \textbf{strstr}
            \bigbreak \noindent 
            \begin{cppcode}
            char* strstr(const char* haystack, const char* needle);
            \end{cppcode}
            \bigbreak \noindent 
            Finds the first occurrence of string \textbf{needle} inside string \textbf{haystack}.
            \bigbreak \noindent 
            Returns pointer on success, NULL if not found.
        \item \textbf{strtok}: 
            \bigbreak \noindent 
            \begin{cppcode}
            char* strtok(char* str, const char* delim);
            \end{cppcode}
            \bigbreak \noindent 
            Splits a string into tokens using delimiter characters.
            \bigbreak \noindent 
            \begin{cppcode}
                #include <stdio.h>
                #include <string.h>

                int main(void) {
                    char s[] = "red,green,blue";

                    char *token = strtok(s, ",");

                    while (token != NULL) {
                        printf("%s\n", token);
                        token = strtok(NULL, ",");
                    }

                    return 0;
                }
            \end{cppcode}
            \bigbreak \noindent 
            strtok modifies the original string by writing '\textbackslash 0' characters into it. Also notice that strtok uses a hidden internal static state. Essentially, you can pass NULL to it and it will pick up where it left off.
            \bigbreak \noindent 
            \begin{cppcode}
                char s[] = "red,green,blue";
                size_t n = sizeof s;

                char *token = strtok(s, ",");

                for (size_t i = 0; i < n; ++i) {
                    printf("%c", s[i]);
                    if (s[i] == '\0') {
                        fputs("__null-term__", stdout);
                    }
                }

                // red__null-term__green,blue__null-term__
            \end{cppcode}

            \bigbreak \noindent 
            \begin{cppcode}
                char s[] = "red,green,blue";
                size_t n = sizeof s;

                char *token = strtok(s, ",");

                while (token != NULL) {
                    token = strtok(NULL, ",");
                }

                for (size_t i = 0; i < n; ++i) {
                    printf("%c", s[i]);
                    if (s[i] == '\0') {
                        // Using fputs instead of puts for no newline.
                        fputs("\\0", stdout);
                    }
                }

                // red\0green\0blue\0
            \end{cppcode}
        \item \textbf{strspn}: 
            \bigbreak \noindent 
            \begin{cppcode}
            size_t strspn(const char* s, const char* accept);
            \end{cppcode}
            \bigbreak \noindent 
            Returns the length of the initial segment of $s$ containing only characters from \texttt{accept}.
        \item \textbf{strcspn}: 
            \bigbreak \noindent 
            \begin{cppcode}
            size_t strcspn(const char *s, const char *reject);
            \end{cppcode}
            \bigbreak \noindent 
            Returns the length of the initial segment of $s$ containing no characters from reject.
        \item \textbf{strpbrk}:
            \bigbreak \noindent 
            \begin{cppcode}
            char *strpbrk(const char *s, const char *accept);
            \end{cppcode}
            \bigbreak \noindent 
            Finds the first character in $s$ that matches any character in accept.
        \item \textbf{strerror}: 
            \bigbreak \noindent 
            \begin{cppcode}
            char* strerror(int errnum);
            \end{cppcode}
            \bigbreak \noindent 
            Returns a human-readable error message for an error number.


    \end{itemize}

    \begin{itemize}
        \item \textbf{malloc} 
            \bigbreak \noindent 
            \begin{cppcode}
            void* malloc(size_t size);
            \end{cppcode}
            \bigbreak \noindent 
            Allocates \texttt{size} bytes of uninitialized memory.
            \bigbreak \noindent 
            Returns a pointer to the allocated memory, or NULL if allocation fails.
            \bigbreak \noindent 
            \begin{cppcode}
            int* arr = malloc(10 * sizeof *arr);
            \end{cppcode}
            \bigbreak \noindent 
            The memory contains indeterminate values. Reading before writing is undefined behavior.
        \item \textbf{calloc}: 
            \bigbreak \noindent 
            \begin{cppcode}
            void* calloc(size_t n, size_t size);
            \end{cppcode}
            \bigbreak \noindent 
            Allocates space for $n$ elements, each \texttt{size} bytes, and initializes all bytes to zero.
            \bigbreak \noindent 
            Returns a pointer to the allocated memory, or NULL if allocation fails.
            \bigbreak \noindent 
            \begin{cppcode}
            int* arr = calloc(10, sizeof *arr);
            \end{cppcode}
            \bigbreak \noindent 
            For integer types, the values become 0. For pointer types, on normal modern systems this gives null pointers, but the standard technically describes byte zeroing.
        \item \textbf{realloc}: 
            \bigbreak \noindent 
            \begin{cppcode}
            void* realloc(void* ptr, size_t size);
            \end{cppcode}
            \bigbreak \noindent 
            Changes the size of a previously allocated block.
            \bigbreak \noindent 
            Returns a pointer to the resized block, or NULL if resizing fails.
            \bigbreak \noindent 
            \begin{cppcode}
                int *arr = malloc(5 * sizeof *arr);

                int *tmp = realloc(arr, 10 * sizeof *arr);
                if (!tmp) {
                    free(arr);
                    /* allocation failed */
                } else {
                    arr = tmp;
                }

                free(arr);
            \end{cppcode}
            \bigbreak \noindent 
            If you assign the return value of realloc to the original memory pointer, you loose it if realloc fails and returns NULL.
        \item \textbf{free}
            \bigbreak \noindent 
            \begin{cppcode}
            void free(void* ptr);
            \end{cppcode}
            \bigbreak \noindent 
            Releases memory allocated by malloc, calloc, or realloc.
            \bigbreak \noindent 
            Returns nothing.
            \bigbreak \noindent 
            \begin{cppcode}
            free(NULL); // safe
            free(nullptr) // safe
            \end{cppcode}
            \bigbreak \noindent 
            Both of these are safe, but freeing the same pointer twice is undefined behavior.


    \end{itemize}

    \subsection{Program termination and process control}
    \begin{itemize}
        \item \textbf{exit} 
            \bigbreak \noindent 
            \begin{cppcode}
            void exit(int status); 
            \end{cppcode}
            \bigbreak \noindent 
            Terminates the program with exit code \texttt{status}. 
            \bigbreak \noindent 
            Returns nothing.
        \item \textbf{abort}: 
            \bigbreak \noindent 
            \begin{cppcode}
            void abort(void)
            \end{cppcode}
            \bigbreak \noindent 
            Abnormally terminates the program.
        \item \textbf{atexit}: 
            \bigbreak \noindent 
            \begin{cppcode}
            int atexit(void (*func)(void));
            \end{cppcode}
            \bigbreak \noindent 
            Registers a function to be called when the program exists normally.
            \bigbreak \noindent 
            Returns zero on success, nonzero on failure.
            \bigbreak \noindent 
            Functions registered with atexit are called in reverse order of registration.
        \item \textbf{Normal program exit cleanup}: A normal program exit (triggered by returning from main() or calling exit()) automatically performs standard runtime and stream cleanup, while application-level resource cleanup remains the responsibility of the code.
            \bigbreak \noindent 
            When a program terminates normally, the C standard library (stdlib.h) executes the following operations automatically in order: 
            \begin{itemize}
                \item \textbf{Calls Termination Functions:} Executes functions registered via the atexit() function in the reverse order of their registration.
                \item \textbf{Flushes and Closes Streams:} Flushes all unwritten buffered data to open C streams (like stdout or stderr) and closes all open files.
                \item \textbf{Deletes Temporary Files:} Removes any temporary files created using tmpfile().
                \item \textbf{Returns Control:} Passes the final exit status code back to the host environment (e.g., to the operating system or parent process). 
            \end{itemize}
        \item \textbf{\_Exit}:
            \bigbreak \noindent 
            \begin{cppcode}
            void _Exit(int status);
            \end{cppcode}
            \bigbreak \noindent 
            Terminates the program immediately. No \texttt{atexit} handlers are called and normal cleanup is not performed.
        \item \textbf{system}
            \bigbreak \noindent 
            \begin{cppcode}
            int system(const char* string);
            \end{cppcode}
            \bigbreak \noindent 
            Passes a command string to the host environment’s command processor.
            \bigbreak \noindent 
            Returns implementation-defined status information.
            \bigbreak \noindent 
            You can also check whether a command processor exists:
            \bigbreak \noindent 
            \begin{cppcode}
                if (system(NULL)) {
                    puts("command processor available");
                }
            \end{cppcode}
            The return value is 
            \begin{itemize}
                \item If command is NULL, then a nonzero value if a shell is available, or 0 if no shell is available.
                \item  If a child process could not be created, or its status could not be retrieved, the return value is -1 and errno is set to indicate the error.
                \item If a shell could not be executed in the child process, then the return value is as though the child shell terminated by calling \_exit(2) with the status 127.
                \item  If all system calls succeed, then the return value is the termination status of the child shell used to execute command. (The termination status of a shell is the termination status of the last command it executes.)
            \end{itemize}
        \item \textbf{getenv}: 
            \bigbreak \noindent 
            \begin{cppcode}
            char* getenv(const char* name);
            \end{cppcode}
            \bigbreak \noindent 
            Gets the value of an environment variable.
            \bigbreak \noindent 
            Returns a pointer to the value string, or NULL if the variable does not exist.
            \bigbreak \noindent 
            Do not modify the returned string unless your platform specifically allows it. Treat it as read-only.

    \end{itemize}

    \unsect{Concepts}
    \begin{itemize}
        \item \textbf{Tag names}: In C, a tag name is the name that belongs to a struct, union, or enum declaration. For example,
            \bigbreak \noindent 
            \begin{cppcode}
                struct Node {
                    int value;
                };
            \end{cppcode}
            \bigbreak \noindent 
            Here, \texttt{Node} is the \textbf{tag name}. The actual type is \texttt{struct Node}. So, to declare a variable
            \bigbreak \noindent 
            \begin{cppcode}
                struct Node n; 
            \end{cppcode}
            \bigbreak \noindent 
            This is why we see
            \bigbreak \noindent 
            \begin{cppcode}
                typedef struct Node {...} node;
                typedef union _U {...} U;
            \end{cppcode}
            \bigbreak \noindent 
            Then,
            \bigbreak \noindent 
            \begin{cppcode}
                U u;  
                Node n;
            \end{cppcode}
            \bigbreak \noindent 
            Are both legal. If we don't care about writing
            \bigbreak \noindent 
            \begin{cppcode}
                Union _U u;
                Struct Node n;
            \end{cppcode}
            \bigbreak \noindent 
            We can omit the tag name,
            \bigbreak \noindent 
            \begin{cppcode}
                typedef struct {...} Node;
                typedef union {...} U;
            \end{cppcode}
            \bigbreak \noindent 
            Then, only 
            \bigbreak \noindent 
            \begin{cppcode}
                Node n;
                U u;
            \end{cppcode}
            \bigbreak \noindent 
            Are legal.
        \item \textbf{Functions that return function pointers}: Consider a function $f$ that accepts a single integer.
            \bigbreak \noindent 
            \begin{cppcode}
                f(int x) {...}
            \end{cppcode}
            \bigbreak \noindent 
            Suppose $f$ returns a pointer a function that returns void and accepts a single double.
            \bigbreak \noindent 
            \begin{cppcode}
                typedef void (*fp)(double);
                fp f(int x) {...}
            \end{cppcode}
            \bigbreak \noindent 
            But, what if we didn't want to use typedef. Then, we wrap the function $f$ inside the function pointer type.
            \bigbreak \noindent 
            \begin{cppcode}
                void (*f(int x))(double y) { ... }
            \end{cppcode}
            \bigbreak \noindent 
            Notice how $fp$ became the signature (excluding a return value) of $f$. Read as: "$f$" is a function that takes an int and returns a pointer to a function that takes a double and returns void.
        \item \textbf{Unions}: C code uses union more often because C has fewer built-in abstraction tools. In C++, many uses of union are replaced by safer language/library features such as classes, inheritance, templates, std::variant, std::optional, and RAII-managed types.
            \bigbreak \noindent 
            A union is useful when you want several different members to occupy the same memory location, but only one of them is valid at a time
            \bigbreak \noindent 
            \begin{cppcode}
                union Value {
                    int i;
                    double d;
                    char* s;
                }
            \end{cppcode}
            \bigbreak \noindent 
            The size of Value is large enough to hold the largest member. The fields overlap in memory.
            \bigbreak \noindent 
            A classic C pattern is a tagged union:
            \bigbreak \noindent 
            \begin{cppcode}
                enum ValueKind {
                    VALUE_INT,
                    VALUE_DOUBLE,
                    VALUE_STRING
                };

                struct Value {
                    enum ValueKind kind;

                    union {
                        int i;
                        double d;
                        char* s;
                    } as;
                };
            \end{cppcode}
            \bigbreak \noindent 
            The kind field tells you which union member is currently valid.
            \item \textbf{SIZE\_MAX}: SIZE\_MAX is a macro constant in the C programming language that represents the maximum possible value of the size\_t data type. It defines the absolute upper limit for the size of any object, array, or memory allocation that the hosting architecture can handle.
                \bigbreak \noindent 
                Defined in
                \bigbreak \noindent 
                \begin{cppcode}
                #include <stdint.h>
                \end{cppcode}
        \end{itemize}

        \subsection{Linkage}
        \begin{itemize}
            \item \textbf{Linkage}: Linkage is about whether declarations of the same name refer to the same object/function across scopes or translation units.
                \begin{itemize}
                    \item \textbf{external linkage}:   same entity can be referred to from other translation units
                    \item \textbf{internal linkage}:   same entity only within this translation unit
                    \item \textbf{no linkage}:         name is purely local; cannot be linked to another declaration
                \end{itemize}
                C has mostly the same linkage / scope rules as C++, with one major difference. At file scope const variable have \textbf{external linkage}.
                \bigbreak \noindent 
                C23 adds a C version of constexpr for objects. A WG14 proposal states that a file-scope constexpr object has static storage duration and internal linkage, meaning each translation unit gets a separate object. So in C23, file-scope constexpr also does not need static for header safety.
        \end{itemize}

        \subsection{stdarg.h}
        \begin{itemize}
            \item \textbf{Header}:
                \bigbreak \noindent

                    va_arg
                    va_end
                    va_copy
                \end{cppcode}
            \item \textbf{Variadic functions}: 
                A variadic function must have at least one named parameter before ....
                \bigbreak \noindent 
                \begin{cppcode}
                    void f(int x, ...) {...}
                \end{cppcode}
            \item \textbf{va\_list}: va\_list is an object that tracks the current position in the unnamed argument list.
                \bigbreak \noindent 
                You do not manually index into the stack or registers. The ABI decides where arguments live. Some may be in registers, some on the stack. va\_list hides those details.
                \bigbreak \noindent 
                \begin{cppcode}
                va_list args;
                \end{cppcode}
                \bigbreak \noindent 
                the arguments are split conceptually like this:
                \bigbreak \noindent 
                \begin{cppcode}
                    named/fixed parameters 
                    unnamed variadic args 
                \end{cppcode}

                \bigbreak \noindent

                    void f(int x, ...) {

                        va_start(args, x); 
                    }
                \end{cppcode}
                \bigbreak \noindent 
                The second argument must be the last named parameter before ....
            \item \textbf{va\_arg}: Reads the next variadic argument as the type specified, then advances the cursor.
                \bigbreak \noindent 
                \begin{cppcode}
                    int value = va_arg(args, int)
                \end{cppcode}
                \bigbreak \noindent 
                This is dangerous because C does not store type information for variadic arguments.
            \item \textbf{va\_end}: Cleans up the va\_list.
            \item \textbf{Count parameter}: A variadic function has no built-in way to know how many extra arguments were passed. You must design a protocol.
                \bigbreak \noindent 
                \begin{cppcode}
                int sum(int count, ...);
                \end{cppcode}
                \bigbreak \noindent 
                The first argument tells the function to read \texttt{count} values.
            \item \textbf{Sentinel value}: We can mark the end of the parameter pack by passing NULL as the last argument.
                \bigbreak \noindent 
                \begin{cppcode}
                print_strings("hello", "world", "test", NULL);
                \end{cppcode}
            \item \textbf{Examples}: 
                \bigbreak \noindent 
                \begin{cppcode}
                    int acc(size_t count, int x, ...) {
                        int acc = x;

                        va_list args;
                        va_start(args, x); 
                        int arg = va_arg(args, int);

                        for (size_t i = 0; i < count; ++i) {
                            acc += arg; 
                            arg = va_arg(args, int);
                        }

                        va_end(args);

                        return acc;
                    }
                \end{cppcode}
                \bigbreak \noindent 
                \begin{cppcode}
                    int acc2(size_t count, ...) {
                        int acc = 0;

                        va_list args;
                        va_start(args, count); 
                        int arg = va_arg(args, int);

                        for (int i = 0; i < count; ++i) {
                            acc += arg; 
                            arg = va_arg(args, int);
                        }

                        va_end(args);

                        return acc;
                    }
                \end{cppcode}
                \bigbreak \noindent 
                \begin{cppcode}
                    size_t ppack_len_count(const char* s1, va_list args) {
                        if (s1 == NULL) return 0;
                        size_t total_len = strlen(s1);

                        const char* arg = NULL;
                        while ((arg = va_arg(args, const char*)) != NULL) {
                            size_t len = strlen(arg);
                            // Rearranged total_len + len + 1 to avoid overflow
                            if (total_len > SIZE_MAX - len - 1) {
                                return 0;
                            }
                            total_len += len;
                        }

                        return total_len;
                    }
                \end{cppcode}
                \bigbreak \noindent 
                \begin{cppcode}
                    char* str_concat(const char* s1, ...) {
                        if (s1 == NULL) {
                            return NULL;
                        }

                        va_list args;
                        va_start(args, s1);

                        size_t tlen = ppack_len_count(s1,args);

                        va_end(args);

                        char* final = malloc(tlen + 1);
                        if (final == NULL) {
                            return NULL;
                        }

                        // Moving write pointer. Final stays fixed to base
                        char* mv_write = final;

                        size_t len = strlen(s1);
                        memcpy(mv_write, s1, len);
                        mv_write += len;

                        va_start(args, s1);

                        const char* arg = NULL;
                        while ((arg = va_arg(args, const char*)) != NULL) {
                            len = strlen(arg);
                            memcpy(mv_write, arg, len);
                            mv_write += len;
                        }

                        va_end(args);

                        *mv_write = '\0';

                        return final;
                    }
                \end{cppcode}

        \end{itemize}

        \subsection{C99}
        \begin{itemize}
            \item \textbf{Designated initializers}: Designated initializers allow initializing struct fields by name.
                \bigbreak \noindent 
                \begin{cppcode}
                    struct S {
                        int x,y,z;
                    };

                    struct S s  = {.x = 10 };
                    struct S s2  = {.x = 10, .y = 20, .z = 30};

                    printf("%d, %d, %d\n", s.x, s.y, s.z); // 10, 0, 0
                    printf("%d, %d, %d\n", s2.x, s2.y, s2.z); // 10, 20, 30
                \end{cppcode}
            \item \textbf{Designate array indices}: You can also designate array indices:
                \bigbreak \noindent 
                \begin{cppcode}
                    int arr[10] = {
                        [5] = 6,
                        [2] = 1
                    };

                    // 0, 0, 1, 0, 0, 6, 0, 0, 0, 0, 
                    for (__auto_type i = 0; i < 10; ++i) 
                    printf("%d, ", arr[i]);
                \end{cppcode}
            \item \textbf{\_\_auto\_type}: \_\_auto\_type is a GNU C compiler extension supported by GCC and Clang that enables automatic type inference in the C programming language. It allows the compiler to automatically deduce the data type of a local variable from its initialization expression, providing functionality similar to the auto keyword in C++11
                \begin{itemize}
                    \item \textbf{Immediate Initialization Required:} You cannot declare a variable with \_\_auto\_type without assigning a value immediately.
                    \item \textbf{Plain Identifiers Only:} The declarator must be a single, plain variable name. You cannot mix it with pointer or array syntax during declaration (e.g., \_\_auto\_type *ptr = &x; is syntax error).
                    \item \textbf{Single Variable Limit:} You can only declare one variable per statement
                \end{itemize}
                Before \_\_auto\_type, type-generic macros relied on the typeof extension. However, typeof evaluates expressions with side effects twice. Using \_\_auto\_type safely captures values exactly once, preventing double-evaluation bugs:
            \item \textbf{auto}: The behavior of the auto keyword in C depends entirely on the version of the C standard you are using. In traditional C (up to C17), auto is a storage class specifier used to declare variables with automatic storage duration, which means they are created when entering a block and destroyed upon exiting it. However, starting with the C23 standard, auto has been redefined to perform type inference, matching its modern behavior in C++
            \item \textbf{Compound literals}: Compound literals create unnamed objects inline.
                \bigbreak \noindent 
                \begin{cppcode}
                    struct Point {
                        int x;
                        int y;
                    };

                    draw_point((struct Point){ .x = 10, .y = 20 });
                \end{cppcode}
                \bigbreak \noindent 
                They are also useful for arrays:
                \bigbreak \noindent 
                \begin{cppcode}
                    int *p = (int[]){1, 2, 3, 4};
                \end{cppcode}
                \bigbreak \noindent 
                Inside a function, a compound literal has automatic storage duration. At file scope, it has static storage duration.
            \item \textbf{Variadic macros}: C99 added macros that accept a variable number of arguments.
                \bigbreak \noindent 
                \begin{cppcode}
                    #define LOG(fmt, ...) \
                    fprintf(stderr, "[log] " fmt "\n", __VA_ARGS__)
                \end{cppcode}
            \item \textbf{String concatenation at compile time}: This is allowed because C concatenates adjacent string literals at compile time.
                \bigbreak \noindent 
                \begin{cppcode}
                    printf("hello, " "world!");
                \end{cppcode}
                \bigbreak \noindent 
                Is translated as if you had written
                \bigbreak \noindent 
                \begin{cppcode}
                    printf("Hello, world!");
                \end{cppcode}
                \bigbreak \noindent 
                The two string literals are not separate runtime strings. The compiler combines them during translation, before the program runs.
                \bigbreak \noindent 
                This rule exists mainly to make long strings easier to write:
                \bigbreak \noindent 
                \begin{cppcode}
                    printf("This is a very long message "
                    "split across multiple source lines "
                    "but still one string.\n");
                \end{cppcode}
            \item \textbf{Inline functions}: C99 added the inline keyword.
                \bigbreak \noindent 
                \begin{cppcode}
                    inline int square(int x) {
                        return x * x;
                    }
                \end{cppcode}
                \bigbreak \noindent 
                The purpose is to allow function-like abstractions without the problems of macros.
                \bigbreak \noindent 
                \begin{cppcode}
                #define SQUARE(x) ((x) * (x))
                SQUARE(i++) // Increments i twice
                \end{cppcode}
                \bigbreak \noindent 
                An inline function would avoid the issue above.
                \bigbreak \noindent 
                C99 inline rules are slightly subtle, especially across translation units. For header-defined functions, many projects use static inline. This gives each translation unit its own internal-linkage copy.
                \bigbreak \noindent 
                \begin{cppcode}
                    static inline int square(int x) {
                        return x*x;
                    }
                \end{cppcode}
            \item \textbf{Variable Length Arrays (VLAs)}: C99 added variable length arrays, or VLAs.
                \bigbreak \noindent 
                \begin{cppcode}
                    void f(int n) {
                        int arr[n];

                        for (int i = 0; i < n; ++i) {
                            arr[i] = i;
                        }
                    }
                \end{cppcode}
                \bigbreak \noindent 
                The array size is determined at runtime, but the object usually has automatic storage duration, typically stack-like storage.
                \bigbreak \noindent 
                VLAs are controversial. They are convenient, but they can be dangerous if n is large or untrusted:
                \bigbreak \noindent 
                C11 later made VLAs optional, so they are less universally portable than other C99 features.
            \item \textbf{Flexible array members}: C99 added flexible array members for structs with variable-sized trailing data.
                \bigbreak \noindent 
                \begin{cppcode}
                    struct Buffer {
                        size_t len;
                        unsigned char data[];
                    };
                \end{cppcode}
                \bigbreak \noindent 
                You allocate enough memory for the struct plus the extra array data:
                \bigbreak \noindent 
                \begin{cppcode}
                    struct Buffer *b = malloc(sizeof *b + 1024);

                    b->len = 1024;
                    b->data[0] = 42;
                \end{cppcode}
                \bigbreak \noindent 
                The flexible array member must be the last member of the struct.
            \item \textbf{\_\_func\_\_}: C99 added \_\_func\_\_, which expands to the current function name as a string.
                \bigbreak \noindent 
                \begin{cppcode}
                    void parse_expr(void) {
                        printf("entered %s\n", __func__);
                    }
                \end{cppcode}
        \end{itemize}

        \unsect{Memory concepts}
        \begin{itemize}
                        \item \textbf{uintptr\_t, intptr\_t}: uintptr\_t and intptr\_t are integer types from:
                \bigbreak \noindent 
                \begin{cppcode}
                    #include <stdint.h>
                \end{cppcode}
                \bigbreak \noindent 
                They are used to store pointer values as integers.
                \bigbreak \noindent 
                \begin{cppcode}
                    uintptr_t  // unsigned integer type capable of holding a pointer value
                    intptr_t   // signed integer type capable of holding a pointer value
                \end{cppcode}
                \bigbreak \noindent 
                if these types exist on your platform, converting a void * pointer to uintptr\_t or intptr\_t and then back to the same pointer type preserves the original pointer value.
                \bigbreak \noindent 
                \begin{cppcode}
                    int x = 42;

                    uintptr_t p_as_int = (uintptr_t)&x;

                    int *p = (int *)p_as_int;

                    printf("%d\n", *p); // 42
                \end{cppcode}
                \bigbreak \noindent 
                Pointers are not necessarily the same size as int, long, or long long. On some platforms, unsigned long might not be large enough to store a pointer. uintptr\_t exists specifically for this purpose.
                \bigbreak \noindent 
                Usually, uintptr\_t is the one you want. Pointer values are normally treated like raw addresses, and addresses are better represented as unsigned integers.
                \bigbreak \noindent 
                Use intptr\_t only if you specifically need a signed integer representation of a pointer value.
            \item \textbf{Printing a pointer as an integer}: Normally, use \%p for pointers
                \bigbreak \noindent 
                \begin{cppcode}
                printf("%p\n", (void*)p);
                \end{cppcode}
                \bigbreak \noindent 
                But, if you want the numeric address
                \bigbreak \noindent 
                \begin{cppcode}
                #include <stdint.h>
                #include <inttypes.h>

                printf("%p\n", (void*)p);
                printf("%" PRIuPTR "\n", (uintptr_t)p);
                printf("%" PRIxPTR "\n", (uintptr_t)p);

                // 0x7ffdceb0ff5c
                // 140728071159644
                // 7ffdceb0ff5c
                \end{cppcode}
                \bigbreak \noindent 
                On my machine, \texttt{PRIuPTR} expands to \texttt{lu}, and \texttt{PRIxPTR} expands to \texttt{lx}
                \bigbreak \noindent 
                Do not assume the format is \%lu, because uintptr\_t might not be unsigned long.
            \item \textbf{Alignment checks}: 
                \bigbreak \noindent 
                \begin{cppcode}
                    if (((uintptr_t)p % 16) == 0) {
                    // p is 16-byte aligned
                }
                \end{cppcode}
                \bigbreak \noindent 
                \begin{cppcode}
                    if (((uintptr_t)p & 15) == 0) {
                        // p is 16-byte aligned
                    }
                \end{cppcode}
                \bigbreak \noindent 
                For an alignment $A$, where $A$ is a power of two, the addr is aligned if
                \bigbreak \noindent 
                \begin{cppcode}
                addr % A == 0
                addr & (A-1) == 0 // Equivalent
                \end{cppcode}
            \item \textbf{Aligning an address downward}: For power-of-two alignment, the formula is
                \bigbreak \noindent 
                \begin{cppcode}
                aligned = addr & ~(align - 1); 
                \end{cppcode}
                \bigbreak \noindent 
                Consider the case where we want to round an address down to the previous 16-byte boundary. In that case, we know that the address is aligned to a 16-byte boundary if its bottom four bites are cleared, since $16-1=15=0xF=0b1111$.
                \bigbreak \noindent 
                If we take an address and divide it by $16$, then its remainder $r$ must obey $ 0 \leq r < 16$. That is, the remainder is between zero and $16-1$ inclusive.
                \bigbreak \noindent 
                Consider a number $a$ and an alignment $A$. In base two,
                \begin{align*}
                    a = &\lambda_{n}2^{n} + \lambda_{n-1}2^{n-1} + \ldots + \lambda_{\ell}2^{\ell} + \ldots  \\
                        &+ \lambda_{\log_{2}(A)}2^{\log_{2}(A)} + \lambda_{\log_{2}(A)-1}2^{\log_{2}(A)-1} \\
                        &+\lambda_{\log_{2}(A)-2}2^{\log_{2}(A)-2} + \ldots + \lambda_{1}2^{1} + \lambda_{0}2^{0}
                .\end{align*}
                Note that $\lambda_{i} \in \left\{0,1\right\}$. If $a$ has alignment $A$, then $a$ has no remainder when divided by $A$. Observe that
                \begin{align*}
                    A = \lambda_{\log_{2}(A)}2^{\log_{2}(A)}
                \end{align*}
                when $\lambda_{\log_{2}(A)} = 1$, and all terms that come before this term are divisible by $A$. All terms that come before this term are in the form
                \begin{align*}
                    \lambda_{\log_{2}(A)+k}2^{\log_{2}(A) + k}
                .\end{align*}
                for $k \in \mathbb{Z}_{>0}$, and $\log_{2}(A)+k \leq n$. Observe that if $\lambda_{\log_{2}{(A)}+k} = 1$,
                \begin{align*}
                    \frac{2^{\log_{2}{(A)}+k}}{2^{\log_{2}{(A)}}} = 2^{k} \in \mathbb{Z}
                ,\end{align*}
                and if $\lambda_{\log_{2}{(A) + k}} = 0$,
                \begin{align*}
                    \frac{0}{2^{\log_{2}{(A)}}} = 0 
                .\end{align*}
                Thus, $a$ will only have a remainder after being divided by $A$  when all terms of the form
                \begin{align*}
                    \lambda_{\log_{2}{(A) - j}}2^{\log_{2}{(A)-j}} 
                \end{align*}
                have $\lambda_{\log_{2}{(A)-j}} =0$, since none of these terms are divisible by $A$. Note that $j \in \mathbb{Z}_{>0}$ and $\log_{2}{(A) - j} \geq 0$
                \bigbreak \noindent 
                This is precisely why the check
                \bigbreak \noindent 
                \begin{cppcode}
                    a & (A-1) == 0
                \end{cppcode}
                \bigbreak \noindent 
                Checks if $a$ has remainder zero when divided by $A$. Now, consider
                \bigbreak \noindent 
                \begin{cppcode}
                ~(A-1) 
                \end{cppcode}
                \bigbreak \noindent 
                This is the number with all terms greater than or equal to $2^{\log_{2}{(A)}}$ non zero, and all terms less than $2^{\log_{2}{(A)}}$ zero. In essence, all bits left of  and including the $\log_{2}{(A)}$th one, and all bits to the right zero. For example, take $A = 16$, then 
                \begin{align*}
                    \sim(A - 1) = \sim(15) = 0b1111\ldots11110000
                .\end{align*}
                So, the bitwise and of a number $a$ with this mask turns off those bottom bits. Thus, it rounds $a$ down to the nearest boundary such that it is $A$ aligned.
            \item \textbf{Aligning an address upward}:  For power-of-two alignment, the formula is
                \bigbreak \noindent 
                \begin{cppcode}
                    aligned = (addr + align - 1) & ~(align - 1);
                \end{cppcode}
                \bigbreak \noindent 
                Adding align - 1 moves the address such that it is inside the next highest boundary, then we take the bitwise and of the negation of the alignment -1 (see above). This rounds it down to the bottom of that next highest boundary. Thus, a round up of the original boundary.
            \item \textbf{Pointer tagging}: Some low-level code stores flags in unused low bits of aligned pointers.
                \bigbreak \noindent 
                \begin{cppcode}
                    #define FLAG 1u

                    uintptr_t tagged = ((uintptr_t)p) | FLAG;

                    int *real_ptr = (int *)(tagged & ~(uintptr_t)FLAG);
                    int flag_set = tagged & FLAG;
                \end{cppcode}
                \bigbreak \noindent 
                This only works when you know the pointer is aligned enough that the low bit is unused.
            \item \textbf{Dangers with uinptr\_t}: This type does not make pointer arithmetic magically safe. The following is okay 
                \bigbreak \noindent 
                \begin{cppcode}
                uintptr_t n = (uintptr_t)p;
                p = (int*)n;
                \end{cppcode}
                \bigbreak \noindent 
                This is much more dangerous
                \bigbreak \noindent 
                \begin{cppcode}
                    uintptr_t n = (uintptr_t)p;
                    n += 100;
                    p = (int *)n;
                \end{cppcode}
                \bigbreak \noindent 
                That may produce a pointer that does not point to a valid object. Dereferencing it would be undefined behavior unless you know exactly what memory layout you are working with.

        \end{itemize}

        \unsect{Idioms}
        \begin{itemize}
            \item \textbf{(void*)-1}: Casting -1 to void* in C is a common idiom used to represent an error, sentinel, or special map value.
                \bigbreak \noindent 
                While NULL (typically (void *)0) signifies "nothing" or "not found," APIs often need a distinct value to indicate an actual failure or error state.
                \bigbreak \noindent 
                When you cast the signed integer -1 to a pointer type, the compiler converts its bit pattern into an address.
                \bigbreak \noindent 
                On modern architectures utilizing two's complement arithmetic, -1 translates to all bits set to 1 (e.g., 0xFFFFFFFF on a 32-bit system or 0xFFFFFFFFFFFFFFFF on a 64-bit system).
                \bigbreak \noindent 
                This effectively places the pointer at the very top of the addressable memory space, a region generally reserved for the operating system kernel. As a result, this pointer is safely outside the range of normal user-space application memory.
                \bigbreak \noindent 
                Attempting to read or write to (void *)-1 (e.g., *(int *)ptr) will immediately trigger a segmentation fault or an access violation because user applications do not have permission to access that extreme address boundary.
                \bigbreak \noindent 
                Converting integers directly to pointers is technically implementation-defined by the C standard. While it works predictably across standard compilers (GCC, Clang, MSVC), you should use the integer type uintptr\_t or intptr\_t from <stdint.h> if you are passing custom integer IDs inside pointer variables:
                \bigbreak \noindent 
                \begin{cppcode}
                    void* ptr = (void*)(uintptr_t)int_id;
                \end{cppcode}

        \end{itemize}


        \pagebreak 
        \unsect{Memory allocator}
        \begin{itemize}
            \item \textbf{sbrk and brk}:
                \begin{cppcode}
                    int brk(void *addr);
                    void *sbrk(intptr_t increment);
                \end{cppcode}
                \bigbreak \noindent 
                brk() and sbrk() change the location of the program break, which defines the end of the process's data segment (i.e., the program break is the first location after the end of the uninitialized data segment).  Increasing the program break has the effect of allocating memory to the process; decreasing the break deallocates memory.
                \bigbreak \noindent 
                brk() sets the end of the data segment to the value specified by addr, when that value is reasonable, the system has enough memory, and the process does not exceed its maximum data size
                \bigbreak \noindent 
                sbrk() increments the program's data space by increment bytes. Calling sbrk() with an increment of 0 can be used to find the current location of the program break.
                \bigbreak \noindent 
                On success, brk() returns zero.  On error, -1 is returned, and errno is set to ENOMEM.
                \bigbreak \noindent 
                On success, sbrk() returns the previous program break.  (If the break was increased, then this value is a pointer to the start of the newly allocated memory).  On error, (void *) -1 is returned, and errno is set to ENOMEM.

            \item \textbf{System calls for obtaining memory}: A custom allocator needs to get large chunks of memory from the operating system. 
                \bigbreak \noindent 
                Historically, allocators use
                \bigbreak \noindent 
                \begin{cppcode}
                sbrk()
                brk
                \end{cppcode}
                \bigbreak \noindent 
                Modern allocators often use:
                \bigbreak \noindent 
                \begin{cppcode}
                mmap();
                munmap();
                \end{cppcode}
            \item \textbf{Process}: 
                \begin{itemize}
                    \item OS gives allocator a large memory region.
                    \item Allocator splits that region into smaller blocks.
                    \item User calls malloc/free implementation that uses the allocator.
                    \item Allocator hands out and recycles blocks.
                \end{itemize}
            \item \textbf{Block metadata}: Every allocated or free block usually has a hidden header.
                \bigbreak \noindent 
                \begin{cppcode}
                typedef struct block {
                    size_t size;
                    int free;
                    struct block *next;
                } block_t;
                \end{cppcode}
                \bigbreak \noindent 
                The real memory may look like this:
                \begin{verbatim}
                    +----------------------+----------------------+
                    | allocator metadata   | user memory          |
                    | size, free, next     | returned by malloc   |
                    +----------------------+----------------------+
                    ^                      ^
                    block header           pointer returned to user
                \end{verbatim}
                If the user requests
                \bigbreak \noindent
                \begin{cppcode}
                int *p = malloc(100);
                \end{cppcode}
                \bigbreak \noindent 
                Your allocator may actually reserve
                \bigbreak \noindent 
                \begin{cppcode}
                sizeof(block_t) + 100 bytes
                \end{cppcode}
                \bigbreak \noindent 
                Then it returns the address just after the metadata. 
                \bigbreak \noindent 
                \textbf{Note:} \textbf{free} is a flag that records whether that block is currently available for reuse.
                \bigbreak \noindent 
                We call \texttt{free} an integer here because historically $C$ did not have a boolean type. However, now that it does, you are free to mark \texttt{free} as a boolean.
                \bigbreak \noindent 
                The \texttt{size} member usually records the size of the usable memory area after the header.
                \bigbreak \noindent 
                \begin{cppcode}
                    void *a = malloc(16);
                    void *b = malloc(64);
                    void *c = malloc(200);
                \end{cppcode}
                \begin{verbatim}
                    +--------+-------------+
                    | header | 16 bytes    |
                    +--------+-------------+

                    +--------+-------------+
                    | header | 64 bytes    |
                    +--------+-------------+

                    +--------+-------------+
                    | header | 208 bytes   |
                    +--------+-------------+
                \end{verbatim}
                If we call the size after the header the \textbf{payload}, then total block size is header plus payload.
                \bigbreak \noindent 
                \begin{cppcode}
                sizeof(block_t) + block->size
                \end{cppcode}
            \item \textbf{alignof and max\_align\_t}: In C, alignof is an operator that returns the CPU-enforced memory alignment requirement of a type. 
                \bigbreak \noindent 
                Note that C11 introduced alignof as
                \bigbreak \noindent 
                \begin{cppcode}
                size_t _Alignof()
                \end{cppcode}
                \bigbreak \noindent 
                with a macro
                \bigbreak \noindent 
                \begin{cppcode}
                #define _Alignof(x) alignof(x)
                \end{cppcode}
                \bigbreak \noindent 
                found in 
                \bigbreak \noindent 
                \begin{cppcode}
                #include <stdalign.h>
                \end{cppcode}
                \bigbreak \noindent 
                While C23 made it available as \texttt{alignof} without the header \texttt{stdalign.h}.
                \bigbreak \noindent 
                \begin{cppcode}
                size_t alignof()
                \end{cppcode}
                \bigbreak \noindent 
                Defined in <stddef.h>, max\_align\_t is a built-in type that guarantees an alignment boundary at least as large as any basic C data type (like double, long double, long long, or pointers).
                \bigbreak \noindent 
                \begin{cppcode}
                #include <stddef.h>
                \end{cppcode}
            \item \textbf{\_Alignas / alignas}: The alignas specifier in C and C++ dictates the custom memory alignment for variables or user-defined types.
                \bigbreak \noindent 
                Values must be an integral constant expression that evaluates to zero or a power of 2 (e.g., 1, 2, 4, 8, 16). If multiple alignas specifiers are applied, the compiler respects the largest value.
                \bigbreak \noindent 
                In C, it was introduced as \_Alignas (in C11). As of C23, it is available as the alignas keyword.
                \bigbreak \noindent 
                \begin{cppcode}
                alignas(8) int my_variable;
                \end{cppcode}
                \bigbreak \noindent 
                 If the type naturally requires a larger alignment (e.g., a hardware type demanding 8 bytes), using alignas(4) is ignored.
            \item \textbf{Alignment}: malloc must return memory aligned enough for any normal object type. For example, this should be valid
                \bigbreak \noindent 
                \begin{cppcode}
                long double *x = malloc(sizeof(long double));
                \end{cppcode}
                \bigbreak \noindent 
                So your allocator cannot return arbitrary byte addresses. It must return properly aligned addresses.
                \bigbreak \noindent 
                If the allocator uses 8-byte alignment, then every returned block size should be a multiple of 8. So, if a user requests 13 bytes, the allocator must return 16 bytes.
                \bigbreak \noindent 
                For a general-purpose malloc-style allocator, choose an alignment that is valid for any ordinary C object the user might store in the returned memory.
                \bigbreak \noindent 
                \begin{cppcode}
                alignment = _Alignof(max_align_t)
                \end{cppcode}
                \bigbreak \noindent 
                A portable alignment helper would be 
                \bigbreak \noindent 
                \begin{cppcode}
                #define ALIGNMENT _Alignof(max_align_t)

                static size_t align_up(size_t n) {
                    return (n + ALIGNMENT - 1) & ~(ALIGNMENT - 1);
                }
                \end{cppcode}
                \bigbreak \noindent 
                This assumes ALIGNMENT is a power of two, which it normally is for C object alignments.
            \item \textbf{Free lists}: A simple allocator usually keeps a linked list of blocks. When malloc(64) is called, the allocator searches the list for a free block large enough.
                \bigbreak \noindent 
                when \texttt{free(p)} is called, the allocator marks the block as free.
                \begin{itemize}
                    \item \textbf{malloc}:
                        \begin{itemize}
                            \item search free list
                            \item find a big enough free block
                            \item mark it used
                            \item return pointer to user memory
                        \end{itemize}
                    \item \textbf{free}:
                        \begin{itemize}
                            \item find block header
                            \item mark block free
                        \end{itemize}
                \end{itemize}
                If the allocator cannot find a free block large enough to satisfy the request, it usually asks the OS for more memory and creates a new block from that memory.
                \bigbreak \noindent 
                The important idea is that malloc does not create memory from nothing. It manages memory it already has, and when that is insufficient, it asks the operating system for a larger region.
            \item \textbf{Two models}:
                \begin{enumerate}
                    \item \textbf{One linked list of all blocks}: Whether free or in use. This is the simpler beginner allocator design.
                    \item \textbf{Separate free list}: More realistic allocators often maintain a list containing only free blocks.
                        \bigbreak \noindent 
                        In this model, when a block is allocated, it is removed from the free list.
                        \bigbreak \noindent 
                        When a block is freed, it is inserted into the free list
                        \bigbreak \noindent 
                        But usually there is not also a normal “in-use list” for allocated blocks. The allocator does not need one for ordinary operation. When the user calls 
                        \bigbreak \noindent 
                        \begin{cppcode}
                        free(ptr)
                        \end{cppcode}
                        \bigbreak \noindent 
                        The allocator can find the block header directly
                        \bigbreak \noindent 
                        \begin{cppcode}
                        block_t* block = (block_t*)ptr-1;
                        \end{cppcode}
                \end{enumerate}
            \item \textbf{Finding the block header from the user pointer}: This is a core trick. If malloc returns a pointer after the header
                \begin{verbatim}
                    +---------+----------------+
                    | header  | user memory    |
                    +---------+----------------+
                              ^
                              returned pointer
                \end{verbatim}
                Then free(ptr) must recover the header by moving backward:
                \bigbreak \noindent 
                \begin{cppcode}
                block_t* block = (block_t* ptr) - 1;
                \end{cppcode}
                \bigbreak \noindent 
                This works if the returned pointer was created like this:
                \bigbreak \noindent 
                \begin{cppcode}
                return block + 1
                \end{cppcode}
                \bigbreak \noindent 
                Thus,
                \bigbreak \noindent 
                \begin{cppcode}
                    void my_free(void *ptr) {
                        block_t *block = (block_t *)ptr - 1;
                        block->free = 1;
                    }
                \end{cppcode}
                \bigbreak \noindent 
                Recall that arithmetic on pointers moves the pointer using the size of the type pointed to.
            \item \textbf{Splitting blocks}: Suppose the allocator has this free block:
                \begin{verbatim}
                    +-----------------------+
                    | free block: 1024 bytes|
                    +-----------------------+
                \end{verbatim}
                Then the user requests 100 bytes. A simple allocator could give the entire 1024-byte block to the user, but that wastes memory. 
                \bigbreak \noindent 
                Instead, it can split the block:
                \begin{verbatim}
                    before:

                    [ free 1024 ]

                    after malloc(100):

                    [ used 100 ][ free 900-ish ]
                \end{verbatim}
                You need to understand how to split one block into two valid blocks with separate headers.
            \item \textbf{Coalescing adjacent free blocks}: When blocks are freed, memory can become fragmented:
                \begin{verbatim}
                    [ used ][ free ][ free ][ used ]
                \end{verbatim}
                The two adjacent free blocks should often be merged:
                \begin{verbatim}
                    [ used ][ larger free ][ used ]
                \end{verbatim}
                This is called \textbf{coalescing}.
                \bigbreak \noindent 
                Without coalescing, your allocator may fail large allocations even though there is enough total free memory.
            \item \textbf{Fragmentation}: There are two major kinds.
                \begin{enumerate}
                    \item \textbf{Internal fragmentation}: The allocator gives more memory than requested.
                    \item \textbf{External fragmentation}: There is enough total free memory, but it is split into small pieces.
                \end{enumerate}
            \item \textbf{Allocation strategies}: When searching the free list, the allocator needs a policy.
                \begin{itemize}
                    \item \textbf{First fit}:	Use the first free block large enough
                    \item \textbf{Best fit}:	Use the smallest free block large enough
                    \item \textbf{Worst fit}:	Use the largest free block
                    \item \textbf{Next fit}:	Continue searching from the previous search position
                \end{itemize}
            \item \textbf{Boundary tags}: More advanced allocators store metadata at both the beginning and end of a block.
                \begin{verbatim}
                    +--------+-------------+--------+
                    | header | user memory | footer |
                    +--------+-------------+--------+
                \end{verbatim}
                This makes it easier to find neighboring blocks and coalesce them efficiently.
                \bigbreak \noindent 
                Without boundary tags, coalescing can require scanning the whole list.
            \item \textbf{Bins and segregated free lists}: A simple allocator may keep one global free list.
                \bigbreak \noindent 
                More advanced allocators keep separate lists for different block sizes:
                \begin{itemize}
                    \item 8-byte blocks    -> list
                    \item 16-byte blocks   -> list
                    \item 32-byte blocks   -> list
                    \item 64-byte blocks   -> list
                    \item large blocks     -> list
                \end{itemize}
                These are often called bins, size classes, or segregated free lists. This improves performance because malloc(16) does not need to search through large unrelated blocks.
            \item \textbf{Thread safety}: A basic allocator is not thread-safe. If two threads call malloc at the same time, they may corrupt the allocator’s internal structures. 
                \bigbreak \noindent 
                You need to eventually understand:
                \begin{enumerate}
                    \item mutexes
                    \item atomic operations
                    \item per-thread arenas
                    \item lock contention
                    \item reentrancy
                \end{enumerate}
            \item \textbf{Debugging allocator bugs}: Allocator bugs are often severe because they corrupt memory silently.
                \bigbreak \noindent 
                You need to understand common failure cases:
                \begin{enumerate}
                    \item double free
                    \item use after free
                    \item buffer overflow
                    \item freeing invalid pointers
                    \item metadata corruption
                    \item returning unaligned pointers
                    \item overlapping allocations
                    \item losing blocks from the free list
                \end{enumerate}
                You should also write internal consistency checks.
                \bigbreak \noindent 
                \begin{cppcode}
                void heap_check(void);
                \end{cppcode}
                \bigbreak \noindent 
                This can verify that block sizes, free-list links, and address ranges make sense.
            \item \textbf{Minimum concepts for a first allocator}
                \begin{enumerate}
                    \item void *, char *, and pointer arithmetic
                    \item Process heap and sbrk()
                    \item Block headers
                    \item Alignment
                    \item Linked-list free list
                    \item First-fit allocation
                    \item Finding the header from a user pointer
                    \item Splitting free blocks
                    \item Marking blocks free
                    \item Coalescing adjacent free blocks
                \end{enumerate}




        \end{itemize}
