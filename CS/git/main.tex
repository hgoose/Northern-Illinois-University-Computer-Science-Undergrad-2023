    \unsect{Brief overview / introduction}
    \begin{itemize}
        \item \textbf{Local vs remote repos}
            \begin{itemize}
                \item A \textbf{local repository} is a repository that is stored on a computer.
                \item A \textbf{remote repository} is a repository that is hosted on a hosting service.
            \end{itemize}
        \item \textbf{Local repositories}: A local repository is represented by a hidden directory called .git that exists within a project directory. It contains all the data on the changes that have been made to the files in a project.
            \bigbreak \noindent 
            You should never touch any files or directories inside your .git directory. Doing this could have undesirable consequences for your repository. You should never delete this directory unless you want to delete your repository.
        \item \textbf{git init}: git init creates a new empty Git repository, or reinitializes an existing one. In the normal case, it creates a .git/ directory containing Git’s object database, refs, tags, config, hooks/templates, and other repository metadata. Running git init again inside an existing repository is safe; Git will not overwrite existing repository data.
            \bigbreak \noindent 
            \begin{bashcode}
                git init [-q | --quiet] 
                         [--bare] 
                         [--template=<template-directory>]
                         [--separate-git-dir <git-dir>] 
                         [--object-format=<format>]
                         [--ref-format=<format>]
                         [-b <branch-name> | --initial-branch=<branch-name>]
                         [--shared[=<permissions>]] 
                         [<directory>]
            \end{bashcode}
            \begin{itemize}
                \item -q Suppress normal output. Git will only print errors and warnings. This is useful in scripts where you do not want the usual “Initialized empty Git repository...” message.
                \item --bare Creates a bare repository. A bare repository has no working tree; it is essentially just the repository database itself. Instead of storing Git metadata inside .git/, the repository directory itself contains things like objects/, refs/, HEAD, and config. Git’s repository-layout documentation describes a bare repository as one “without its own working tree,” typically used for exchanging histories by pushing and fetching.
                \item --template Tells Git where to copy template files from when initializing the repository. Templates can include default hook samples, info files, exclude patterns, and initial repository structure. Git chooses the template directory in this order: the --template argument, then $GIT\_TEMPLATE\_DIR, then init.templateDir, then the default template directory.
                    \bigbreak \noindent 
                    This is useful if you want every new repo to start with the same hooks or config-related files.
                \item --separate-git-dir Stores the actual Git repository data somewhere other than .git/ in the working directory. Instead of creating a real .git/ directory, Git creates a .git text file that points to the actual repository directory. If used while reinitializing, Git moves the repository to the specified path
                    \bigbreak \noindent 
                    Stores the actual Git repository data somewhere other than .git/ in the working directory. Instead of creating a real .git/ directory, Git creates a .git text file that points to the actual repository directory. If used while reinitializing, Git moves the repository to the specified path
                \item --object-format Chooses the hash algorithm used for Git objects. Valid values are currently sha1 and, if supported by your Git build, sha256. sha1 is the default. Git’s documentation also notes that SHA-256 repositories and SHA-1 repositories currently do not interoperate directly.
                \item --ref-format Chooses how Git stores refs, such as branches and tags. Valid values are files and reftable. The default is files, meaning loose ref files plus packed-refs. reftable uses the newer reftable storage format.
                \item -b Sets the name of the initial branch in the new repository. These two forms are equivalent. For example, this creates the repo with main as the initial branch name:
                \item --shared Initializes the repository with permissions suitable for sharing among multiple users. This sets core.sharedRepository, causing files under \$GIT\_DIR to be created with the requested permissions. If --shared is not specified, Git uses the normal permissions determined by your system umask.
                \item Tells Git where to initialize the repository. If the directory does not exist, Git creates it. If it does exist, Git initializes the repository inside it.
            \end{itemize}
        \item \textbf{The areas of Git}: There are four important areas to be aware of when you are working with Git:
            \begin{enumerate}
                \item \textbf{Working directory}: The actual files and folders you see and edit in your project directory.
                \item \textbf{Staging area}: The intermediate area where you choose which changes will go into the next commit.
                    \bigbreak \noindent 
                    The staging area is also called the index. When you run
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git add main.c
                    \end{bashcode}
                    \bigbreak \noindent 
                    you copy the current version of main.c from the working directory into the staging area.
                    \bigbreak \noindent 
                    This lets you prepare a commit carefully. For example, you might edit three files but stage only one of them.
                    \bigbreak \noindent 
                    You can inspect staged changes with:
                    \bigbreak \noindent 
                    \begin{cppcode}
                        git diff --staged             
                    \end{cppcode}
                \item \textbf{Commit history}: The sequence of commits that record the project’s past states. A commit is a snapshot of the staged files, plus metadata such as author, date, commit message, and a pointer to the previous commit.
                    \bigbreak \noindent 
                    \begin{center}
                        A --- B --- C
                    \end{center}
                    Here, $A$, $B$, and $C$ are commits. The latest commit on your branch is usually what HEAD points to.
                    \bigbreak \noindent 
                    You can view the commit history with
                    \bigbreak \noindent 
                    \begin{bashcode}
                        git log 
                    \end{bashcode}
                    \bigbreak \noindent 
                    And
                    \bigbreak \noindent 
                    \begin{cppcode}
                    git log --graph
                    \end{cppcode}
                \item \textbf{Local repository}: The Git repository stored on your own machine. In a normal project, the local repository is mainly the .git directory
                    \bigbreak \noindent 
                    The .git/ directory contains Git’s internal database: commits, branches, tags, object files, configuration, refs, and the staging area.
            \end{enumerate}
        \item \textbf{Working tree}: The working tree is the set of actual project files that git has checked out for you to edit. For
            \bigbreak \noindent 
            \begin{bashcode}
            myproject/
              main.c
              README.md
              src/
              .git/
            \end{bashcode}
            \bigbreak \noindent 
            The working tree is 
            \bigbreak \noindent 
            \begin{bashcode}
            main.c 
            README.md
            src/
            \end{bashcode}
        \item \textbf{git diff}: Git can show the difference between the working tree and the staging area with
            \bigbreak \noindent 
            \begin{bashcode}
            git diff
            \end{bashcode}
            \bigbreak \noindent 
            shows changes in your working tree that are not staged. Answers: 
            \begin{quote}
                "\textit{What have I changed but not staged yet?}"
            \end{quote}
            \bigbreak \noindent 
            \begin{bashcode}
            git diff --staged
            \end{bashcode}
            \bigbreak \noindent 
            Shows staged changes, meaning what would go into the next commit.
            \bigbreak \noindent 
            \begin{bashcode}
            git diff HEAD
            \end{bashcode}
            \bigbreak \noindent 
            Shows all changes compared to the last commit, both staged and unstaged.
            \bigbreak \noindent 
            \begin{bashcode}
            git diff commit1 commit2
            \end{bashcode}
            \bigbreak \noindent 
            Shows what changed between two commits
            \bigbreak \noindent 
            \begin{bashcode}
            git diff main feature
            \end{bashcode}
            \bigbreak \noindent 
            Shows the difference between the snapshots pointed to by main and feature.
            
        \item \textbf{Tracked files}: A tracked file is a file Git already knows about.
            \bigbreak \noindent 
            More precisely, a file is tracked if it is already in Git’s repository history or currently in the index.
            \bigbreak \noindent 
            A tracked file is under Git’s management. It can be in states such as:
            \begin{enumerate}
                \item unmodified
                \item modified
                \item staged
                \item deleted
            \end{enumerate}
        \item \textbf{git add -A}: Stages all additions, modifications, and deletions in the whole repository.
            \bigbreak \noindent 
            \begin{bashcode}
            git add -A
            \end{bashcode}
            \bigbreak \noindent 
            stages all changes in the entire repository, regardless of what directory you are currently in.
            \bigbreak \noindent 
            \begin{bashcode}
            git add .
            \end{bashcode}
            \bigbreak \noindent 
            stages changes only in the current directory and below.
        \item \textbf{git restore, revert, and reset}: 
            \begin{itemize}
                \item \textbf{Restore}: git restore is mainly for undoing changes in your working tree or staging area. Use it when you want to throw away uncommitted changes.
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git restore file.txt
                    \end{bashcode}
                    \bigbreak \noindent 
                    This discards unstaged changes to file.txt in your working tree and restores it to the version currently in the staging area.
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git restore --staged file.txt
                    \end{bashcode}
                    \bigbreak \noindent 
                    removes file.txt from the index, --staged can undo a git add
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git restore --worktree file.txt
                    \end{bashcode}
                    \bigbreak \noindent 
                    discards unstaged changes in the file and restores it from the index. Note that this is equivalent to 
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git restore file.txt
                    \end{bashcode}
                    \bigbreak \noindent 
                    because --worktree is the default target for git restore.
                \item \textbf{revert}: git revert undoes a commit by creating a new commit that applies the opposite changes.
                    \bigbreak \noindent 
                    Use it when the bad commit is already part of shared history, such as a branch you pushed to GitHub.
                    \bigbreak \noindent 
                    If commit \textbf{abc123} adds a line "\textit{hello}", then
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git revert abc123
                    \end{bashcode}
                    \bigbreak \noindent 
                    creates a new commit that removes that line. The original commit still exists in history.
                \item \textbf{Reset}: git reset moves the current branch pointer to another commit. Depending on the option, it may also change the staging area and working tree.
                    \bigbreak \noindent 
                    Use it when you want to move your branch backward or rewrite local history.
            \end{itemize}
        \item \textbf{Branches}: A branch in Git is a movable name that points to a commit. That is the core idea. Suppose history looks like this:
            \begin{center}
                A \texttt{--} B \texttt{--} C 
            \end{center}
            Suppose main is a branch, and points to commit $C$. When you make a new commit while on main, Git creates a new commit and moves main forward:
            \bigbreak \noindent 
            Branches let you work on a separate line of development without disturbing another line.
            \bigbreak \noindent 
            \begin{bashcode}
            git switch -c feature-login
            \end{bashcode}
            \bigbreak \noindent 
            This creates a new branch called feature-login and switches to it. Now, both main and feature-login point to commit $C$. If you commit on feature-login, only that branch moves forward.
            \begin{verbatim}
                A -- B -- C
                          ^
                        main
                            \
                            D -- E
                                 ^
                           feature-login
            \end{verbatim}
        \item \textbf{HEAD}: HEAD means “where you currently are.” Usually, head points to a branch. If you make a commit, git moves the branch that HEAD is attached to 
            \bigbreak \noindent 
            \begin{bashcode}
                git branch feature-login
            \end{bashcode}
            \bigbreak \noindent 
            Creates a branch but does not switch to it.
            \bigbreak \noindent 
            \begin{bashcode}
            git switch -c feature-login
            git checkout -b feature-login
            \end{bashcode}
            \bigbreak \noindent 
            Both create the branch and switch to it
        \item \textbf{Merging branches}: Eventually, you usually want to bring the changes from one branch into another. Suppose you want to merge feature-login into main. First, switch to main.
            \bigbreak \noindent 
            \begin{bashcode}
            git switch main
            \end{bashcode}
            \bigbreak \noindent 
            Then, merge
            \bigbreak \noindent 
            \begin{bashcode}
            git merge feature-login.
            \end{bashcode}
            \bigbreak \noindent 
            \begin{verbatim}
               A -- B -- C -------- M
                          \        /
                           D -- E 
            \end{verbatim}
            \bigbreak \noindent 
            $M$ is a merge commit. It combines the histories of main and feature-login. Sometimes Git can do a \textbf{fast-forward merge} instead:
            \bigbreak \noindent 
            \begin{verbatim}
                A -- B -- C -- D -- E
                                    ^
                                  main
            \end{verbatim}
            \bigbreak \noindent 
            Note that we can pass \texttt{--no-ff} to force a merge commit. 
            \bigbreak \noindent 
            \begin{bashcode}
            git merge --no-ff feature-login
            \end{bashcode}
            \bigbreak \noindent 
            After merging a branch, you can delete the local branch name:
            \bigbreak \noindent 
            \begin{bashcode}
            git branch -d feature-login
            \end{bashcode}
            \bigbreak \noindent 
            This deletes the branch pointer, not necessarily the commits. If the commits are already reachable from another branch, such as main, they are still part of history.
            \bigbreak \noindent 
            \begin{bashcode}
            git branch -D feature-login
            \end{bashcode}
            \bigbreak \noindent 
            This deletes the branch even if Git thinks it has unmerged work.
        \item \textbf{Squash merge vs regular merge}: A regular merge preserves the branch history. A squash merge takes the final changes from a branch but does not preserve that branch’s individual commits in the target branch history. 
            \bigbreak \noindent 
            Suppose we have the history
            \bigbreak \noindent 
            \begin{verbatim}
                A---B---C  main
                    \
                    D---E---F  feature 
            \end{verbatim}
            \bigbreak \noindent 
            A regular merge creates a new merge commit that has two parents: the current main commit and the tip of feature.
            \bigbreak \noindent 
            \begin{bashcode}
                git switch main
                git merge feature
            \end{bashcode}
            \bigbreak \noindent 
            \begin{verbatim}
                A---B---C-------M  main
                    \         /
                    D---E---F  feature
            \end{verbatim}
            \bigbreak \noindent 
            The commits D, E, and F remain visible as separate commits in the project history. Git records that feature was merged into main.
            \bigbreak \noindent 
            Use a regular merge when you want to preserve the real development history, including all intermediate commits and the fact that a branch existed.
            \bigbreak \noindent 
            A squash merge takes the combined changes from D, E, and F, applies them to main, and then lets you commit them as one new commit.
            \bigbreak \noindent 
            \begin{bashcode}
                git switch main
                git merge --squash feature
                git commit
            \end{bashcode}
            \bigbreak \noindent 
            \begin{verbatim}
                A---B---C---S  main

                    D---E---F  feature
            \end{verbatim}
            \bigbreak \noindent 
            Here, $S$ is a single new commit containing the same final changes as the whole feature branch.
            \bigbreak \noindent 
            But importantly, S is not a merge commit. It has only one parent: C.
            \bigbreak \noindent 
            Git does not record that feature was merged into main. The commits D, E, and F are not part of main’s history.
            \bigbreak \noindent 
            After a squash merge, Git does not consider the feature branch “merged” in the same structural way.
            \bigbreak \noindent 
            \begin{bashcode}
            git branch --merged
            \end{bashcode}
            \bigbreak \noindent 
            Will not show that the branch was merged.
            
            
            
        \item \textbf{Local vs remote branches}: A local branch exists in your repository. A \textbf{remote-tracking branch} represents the state of a branch on a remote, such as GitHub.
            \bigbreak \noindent 
            origin/main means: My local record of what main looked like on the remote named origin.
            \bigbreak \noindent 
            \begin{bashcode}
            git fetch
            \end{bashcode}
            \bigbreak \noindent 
            updates remote-tracking branches.
            \bigbreak \noindent 
            \begin{bashcode}
            git push origin feature-login
            \end{bashcode}
            \bigbreak \noindent 
            Git sends your local feature-login branch to the remote.
        \item \textbf{git fetch}: The git fetch command downloads the latest commits, files, and references from a remote repository into your local repository, but it does not merge or integrate them into your current working code.
            \bigbreak \noindent 
            It essentially acts as a safe, "read-only" check that updates Git's internal tracking records to mirror what is happening on the server. Because it never alters your local branches or working files, it is completely safe to run at any time without risking merge conflicts.
        \item \textbf{Checkout vs switch}: Git checkout is the older, overloaded command. Switch is the newer, more specific command for switching branches.
            \bigbreak \noindent 
            \begin{bashcode}
            git checkout main
            git switch main
            \end{bashcode}
            \bigbreak \noindent 
            Both switch you to the main branch. But checkout can do multiple unrelated things. It can also restore files (seen below).
        \item \textbf{Hunks}: In Git, a hunk is a contiguous block of changes inside a file. When Git shows a diff, it does not usually show the entire file. It shows only the changed regions, plus a little surrounding context. Each changed region is called a \textbf{hunk}.
            \bigbreak \noindent 
            \begin{bashcode}
                diff --git a/file.txt b/file.txt
                index 83db48f..f735c2d 100644
                --- a/file.txt
                +++ b/file.txt
                @@ -1,5 +1,5 @@
                    line 1
                    line 2
                -old line
                +new line
                    line 4
                    line 5
            \end{bashcode}
            \bigbreak \noindent 
            This part is the hunk:
            \bigbreak \noindent 
            \begin{bashcode}
                @@ -1,5 +1,5 @@
                    line 1
                    line 2
                -old line
                +new line
                    line 4
                    line 5
            \end{bashcode}
            \begin{itemize}
                \item The -old line means that line was removed.
                \item The +new line means that line was added.
            \end{itemize}
            The unchanged lines around it are context, included so Git and humans can understand where the change belongs.
            \bigbreak \noindent 
            \begin{bashcode}
            @@ -1,5 +1,5 @@
            \end{bashcode}
            \bigbreak \noindent 
            is the \textbf{hunk header}. It means:
            \begin{bashcode}
                -1,5   old file: starting at line 1, show 5 lines
                +1,5   new file: starting at line 1, show 5 lines
            \end{bashcode}
            \bigbreak \noindent 
            So the hunk describes a change from one version of the file to another.
        \item \textbf{Patches}: A patch is a text representation of changes. In Git, a patch usually means a \textbf{diff file} that describes how to change one version of files into another version. 
            \bigbreak \noindent 
            A patch is made out of one or more \textbf{hunks}.
            \bigbreak \noindent 
            You can create a patch with
            \bigbreak \noindent 
            \begin{bashcode}
            git diff > changes.patch
            \end{bashcode}
            \bigbreak \noindent 
            Someone else can apply it with
            \bigbreak \noindent 
            \begin{bashcode}
            git apply changes.patch
            \end{bashcode}
            \bigbreak \noindent 
            That modifies their working tree using the changes described in the patch.
        \item \textbf{Merges}: A merge in git combines changes from one branch into another branch.
            \bigbreak \noindent 
            \begin{bashcode}
            git switch main
            git merge feature
            \end{bashcode}
            \bigbreak \noindent 
            This means: Take the changes from feature and integrate them into main.
            \bigbreak \noindent 
            A merge conflict happens when Git cannot automatically combine the changes. Suppose both branches changed the same line differently,
            \bigbreak \noindent 
            \begin{cppcode}
            // main.c
            int x = 10;
            \end{cppcode}
            \bigbreak \noindent 
            \begin{bashcode}
            // feature.c
            int x = 20;
            \end{bashcode}
            \bigbreak \noindent 
            Git cannot possibly know which one you want, so it marks the conflict in the file.
            \bigbreak \noindent 
            \begin{bashcode}
                <<<<<<< HEAD
                int x = 10;
                =======
                int x = 20;
                >>>>>>> feature
            \end{bashcode}
            \bigbreak \noindent 
            This means
            \bigbreak \noindent 
            \begin{bashcode}
                HEAD side:    your current branch, main
                other side:   the branch being merged, feature
            \end{bashcode}
            \bigbreak \noindent 
            You manually edit the file to the final version you want, then you stage the resolved file, and finish the merge with a commit.
        \item \textbf{Cherry pick}:  git revert and git cherry-pick are almost opposites in spirit.
            \bigbreak \noindent 
            git cherry-pick copies a commit’s changes into your current branch. Use cherry-pick when you want to take one specific commit from another branch and apply it to your current branch.
        \item \textbf{Tags}: You can attach a name to a commit using a tag.
            \bigbreak \noindent 
            \begin{bashcode}
            git tag v1.0 a3f4c91
            \end{bashcode}
            \bigbreak \noindent 
            Now, v1.0 refers to that commit.
        \item \textbf{Stashing}: Git stash temporarily saves your uncommited changes somewhere inside git, then cleans your working tree. Use it when you are in the middle of work but need to switch branches, pull changes, or do something else without committing yet.
            \bigbreak \noindent 
            \begin{bashcode}
            git stash
            \end{bashcode}
            \bigbreak \noindent 
            This takes your current uncommitted changes and stores them in a stash entry.
            \bigbreak \noindent 
            Then your working tree goes back to a clean state.
            \bigbreak \noindent 
            \begin{bashcode}
            git stash list
            \end{bashcode}
            \bigbreak \noindent 
            lists the stashes
            \bigbreak \noindent 
            \begin{bashcode}
            git stash apply
            \end{bashcode}
            \bigbreak \noindent 
            Brings back the most recent stash
            \bigbreak \noindent 
            \begin{bashcode}
            git stash pop
            \end{bashcode}
            \bigbreak \noindent 
            Applies and removes from the list
            \bigbreak \noindent 
            \begin{bashcode}
            git stash apply <stash name>
            \end{bashcode}
            \bigbreak \noindent 
            Applies a specific stash from the list.
            \bigbreak \noindent 
            \begin{bashcode}
            git stash drop <name>
            git stash clear
            \end{bashcode}
            \bigbreak \noindent 
            Removes stashes from the list, or clears the list.
            \bigbreak \noindent 
            \begin{bashcode}
            git stash --staged
            \end{bashcode}
            \bigbreak \noindent 
            stashes only the staged changes. That means Git takes the changes currently in the index/staging area and puts them into a stash. Your unstaged working-tree changes are left alone.
            \bigbreak \noindent 
            \begin{bashcode}
            git stash --keep-indexd 
            \end{bashcode}
            \bigbreak \noindent 
            This stashes your unstaged changes, but keeps your staged changes in place.
            \bigbreak \noindent 
            In other words, the index is preserved.
        \item \textbf{Staging area manipulations}: Staging is represented by a directory \textbf{index} instead .git. The commands that 
            \begin{itemize}
                \item \textbf{git add}: Stages file contents. More precisely, it copies the current content of selected working-directory files into the index, preparing that content for the next commit. git add -p stages selected hunks interactively. git add -A stages additions, modifications, and deletions.
                \item \textbf{git rm}: Removes paths from the index. By default, it removes them from both the working tree and the index. With --cached, it removes them from the index only, leaving the file in your working directory 
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git rm file.c
                    git rm --cached file.c
                    \end{bashcode}
                    \bigbreak \noindent 
                    Use git rm --cached when you accidentally started tracking a file but want to keep the local copy.
                \item \textbf{git mv}: Moves or renames a file and updates the index to record that change. The change is staged, but it still needs to be committed.
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git mv oldname.c newname.c
                    \end{bashcode}
                    \bigbreak \noindent 
                    This is roughly equivalent to
                    \bigbreak \noindent 
                    \begin{bashcode}
                        mv oldname.c newname.c
                        git rm oldname.c
                        git add newname.c
                    \end{bashcode}
                \item \textbf{git restore --staged}: Changes the index without necessarily changing the working directory. This is the modern command for “unstaging” something. By default, git restore --staged <path> restores the index version from HEAD.
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git restore --staged file.c
                    \end{bashcode}
                    \bigbreak \noindent 
                    You can also restore both the index and working tree:
                    \bigbreak \noindent 
                    \begin{cppcode}
                    git restore --staged --worktree file.c
                    \end{cppcode}
                \item \textbf{git reset}: git reset can move HEAD, reset the index, and sometimes reset the working tree depending on the mode.
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git reset file.c
                    \end{bashcode}
                    \bigbreak \noindent 
                    Unstages file.c
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git reset 
                    \end{bashcode}
                    \bigbreak \noindent 
                    Resets the whole index to HEAD, leaving the working directory alone
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git reset --mixed HEAD~1
                    \end{bashcode}
                    \bigbreak \noindent 
                    Moves HEAD and resets the index, but keeps working-directory changes. --mixed is the default reset mode.
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git reset --hard HEAD
                    \end{bashcode}
                    \bigbreak \noindent 
                    Resets both the index and working tree to HEAD
                    \bigbreak \noindent 
                    \begin{cppcode}
                    git reset --soft HEAD
                    \end{cppcode}
                    \bigbreak \noindent 
                    Resets the working tree but \textbf{not} the index.
                \item \textbf{git checkout}: git checkout is older and overloaded. It can affect the index in some forms but not others.
                    \bigbreak \noindent 
                    This form updates both the working tree and the index from another commit/tree:
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git checkout main -- file.c
                    \end{bashcode}
                    \bigbreak \noindent 
                    That means: take file.c from main, put it into my working tree, and stage that version in the index.
                    \bigbreak \noindent 
                    Recall that \texttt{--} signals the end of command options. 
                    \bigbreak \noindent 
                    The form
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git checkout -- file.c
                    \end{bashcode}
                    \bigbreak \noindent 
                    Restores the working-tree file from the index, so it affects the working directory, not the staging area.
                \item \textbf{Detached head}: Consider a commit with hash 5f1bc85
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git checkout 5f1bc85
                    \end{bashcode}
                    \bigbreak \noindent 
                    This checks out the commit whose hash starts with 5f1bc85.
                    \bigbreak \noindent 
                    The important effect is that Git puts you in a detached HEAD state.
                    \bigbreak \noindent 
                    Normally, HEAD points to a branch:
                    \bigbreak \noindent 
                    \begin{bashcode}
                    HEAD -> main -> C
                    \end{bashcode}
                    \bigbreak \noindent 
                    After checking out a specific commit, HEAD points directly to that commit instead of to a branch:
                    \bigbreak \noindent 
                    \begin{bashcode}
                    HEAD -> f51bc85
                    \end{bashcode}
                    So your working tree and index are updated to match the snapshot stored in commit 5f1bc85.
                    \bigbreak \noindent 
                    You are not currently “on” a branch.
                    \bigbreak \noindent 
                    You can inspect old files, build the project, run tests, or look around safely.
                    \bigbreak \noindent 
                    But if you make a new commit from here:
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git add .
                    git commit -m "experiment"
                    \end{bashcode}
                    \bigbreak \noindent 
                    you get
                    \bigbreak \noindent 
                    \begin{verbatim}
                       A--B--C--D main 
                          \ 
                           E HEAD
                    \end{verbatim}
                    \bigbreak \noindent 
                    Commit E is not on a named branch unless you create one.
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git switch -c experiment-branch
                    \end{bashcode}
                \item \textbf{Git reset vs git checkout}: 
                    \begin{itemize}
                        \item \textbf{reset --hard:}
                            \begin{enumerate}
                                \item moves the current branch
                                \item resets index
                                \item resets working tree
                                \item discards tracked-file changes
                            \end{enumerate}
                        \item \textbf{checkout:}
                            \begin{enumerate}
                                \item moves HEAD to another commit
                                \item leaves the branch where it is
                                \item usually preserves/protects local changes, unless we pass -f
                            \end{enumerate}
                    \end{itemize}
                \item \textbf{Delete untracked files}: Git clean removes untracked files from the working tree
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git clean -fd # Force (f) and recursive (d)
                    \end{bashcode}
                \item \textbf{git switch}: Switching branches updates the working tree and the index to match the branch being checked out
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git switch main
                    git switch feature
                    \end{bashcode}
                    \bigbreak \noindent 
                    So while git switch is not normally thought of as a staging command, it does replace the index as part of changing branches
                \item \textbf{git commit}: A plain commit reads the current index and creates a commit from it. The Git docs describe git commit as creating a new commit from the current contents of the index
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git commit -a
                    \end{bashcode}
                    \bigbreak \noindent 
                    Automatically stages modifications and deletions for already-tracked files before committing. It does not add brand-new untracked files. 
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git commit -p
                    \end{bashcode}
                    \bigbreak \noindent 
                    Lets you interactively choose hunks to include in the commit.
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git commit file.c
                    \end{bashcode}
                    \bigbreak \noindent 
                    Commits the current content of the named tracked file, bypassing the normal “only what is already staged” behavior for that file.
                \item \textbf{git commit --amend}: Replaces the most recent commit with a new version of that commit.
                    \bigbreak \noindent 
                    It is used when you want to modify the last commit instead of creating an additional commit.
                \item \textbf{git apply --index}: Applies a patch to both the working tree and the index.
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git apply --index patch patch.diff
                    \end{bashcode}
                \item \textbf{git apply --cached}: Applies a patch only to the index, leaving the working tree alone
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git apply --cached patch.diff
                    \end{bashcode}
                    \bigbreak \noindent 
                    Without --index or --cached, git apply applies the patch only to files in the working tree.
                \item \textbf{git merge}: Merging updates the index and working tree when changes apply cleanly. If conflicts occur, the index stores special unmerged entries until you resolve them and stage the resolved files.
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git merge feature
                    \end{bashcode}
                    \bigbreak \noindent 
                    After conflict resolution,
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git add conflicted-file.c
                    git commit
                    \end{bashcode}
                \item \textbf{git cherry-pick}: Applies the changes from an existing commit. If paths apply cleanly, they are updated in both the index and working tree. If conflicts occur, the index records multiple versions for conflicted paths.
                    \bigbreak \noindent 
                    \begin{bashcode}
                        git cherry-pick abc123
                    \end{bashcode}
                    \bigbreak \noindent 
                    After resolving conflicts:
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git add file.c
                    git cherry-pick --continue
                    \end{bashcode}
                \item \textbf{git revert}: Creates a new commit that reverses an earlier commit. With --no-commit/-n, it applies the reverse changes to the working tree and index without committing.
                \item \textbf{git rebase}: Rebase repeatedly applies commits onto a new base. Internally, this affects the working tree and index as commits are replayed. If conflicts occur, you resolve files, stage them with git add, and continue:
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git rebase main
                    git add file.c
                    git rebase --continue
                    \end{bashcode}
                \item \textbf{Conflicts in a rebase}: After a git rebase, conflicts are handled one commit at a time. Git pauses at the first commit that cannot be replayed cleanly.
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git status
                    \end{bashcode}
                    \bigbreak \noindent 
                    Tells you which files have conflicts. Open each conflicted file and look for conflict markers. 
                    \bigbreak \noindent 
                    \begin{bashcode}
                        <<<<<<< HEAD
                        code from the branch you are rebasing onto
                        =======
                        code from the commit currently being replayed
                        >>>>>>> commit-message-or-hash
                    \end{bashcode}
                    \bigbreak \noindent 
                    Edit the file so it contains the final correct version. Remove the conflict markers. Then, stage the resolved files and continue the rebase
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git rebase --continue
                    \end{bashcode}
                    \bigbreak \noindent 
                    Git will then try to replay the next commit. If another conflict occurs, repeat the same process.
                \item \textbf{Rebase escape options}:
                    \begin{enumerate}
                        \item \textbf{Abort}: Cancels the whole rebase and returns your branch to how it was before the rebase started.
                            \bigbreak \noindent 
                            \begin{bashcode}
                            git rebase --abort
                            \end{bashcode}
                        \item \textbf{Skip}:  Skips the commit that caused the conflict. Use this only when you are sure that commit’s changes are no longer needed.
                            \bigbreak \noindent 
                            \begin{bashcode}
                            git rebase --skip
                            \end{bashcode}
                        \item \textbf{Continue}: Continues after you resolve and stage the conflicts
                            \bigbreak \noindent 
                            \begin{bashcode}
                            git rebase --continue
                            \end{bashcode}
                    \end{enumerate}
                \item \textbf{Shortcut for resolving conflicts}: git checkout --ours file and git checkout --theirs file are shortcuts for resolving a conflicted file by choosing one side completely.
                    \bigbreak \noindent 
                    They are used when Git has stopped because of a conflict.
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git checkout --ours file.cpp
                    \end{bashcode}
                    \bigbreak \noindent 
                    Means replace file.cpp with our side of the conflict.
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git checkout --theirs file.cpp
                    \end{bashcode}
                    \bigbreak \noindent 
                    Means replace file.cpp with their side of the conflict. After choosing one, you still need to stage the file:
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git add file.cpp
                    \end{bashcode}
                    \bigbreak \noindent 
                    Then, continue the operation
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git merge --continue
                    \end{bashcode}
                    \bigbreak \noindent 
                    However, during a rebase the meaning feels reversed. Suppose we are replaying feature commits on top of main. So, we are on branch \texttt{feature} and ran
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git rebase main
                    \end{bashcode}
                    \bigbreak \noindent 
                    In this case, 
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git checkout --ours file.cpp
                    \end{bashcode}
                    \bigbreak \noindent 
                    chooses the version from main, the branch we are rebasing onto.
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git checkout --theirs file.cpp
                    \end{bashcode}
                    \bigbreak \noindent 
                    chooses the version from the \texttt{feature} commit currently being replayed.
                    \bigbreak \noindent 
                    \textbf{Note:} Modern git also lets you write this as
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git restore --ours file.cpp
                    git restore --theirs file.cpp
                    \end{bashcode}
                \item \textbf{Interactive rebasing}: git rebase -i means interactive rebase. It lets you rewrite a sequence of commits by editing a temporary “todo list” before Git replays those commits.
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git rebase -i main
                    \end{bashcode}
                    \bigbreak \noindent 
                    Git opens a list like this:
                    \bigbreak \noindent 
                    \begin{bashcode}
                        pick a1b2c3d Add parser
                        pick b2c3d4e Fix parser bug
                        pick c3d4e5f Add tests
                        pick d4e5f6a Rename parse function
                    \end{bashcode}
                    \bigbreak \noindent 
                    You can change \textbf{pick}, reorder lines, delete lines, or combine commits.
                    \bigbreak \noindent 
                    Suppose your history is:
                    \bigbreak \noindent 
                    \begin{verbatim}
                       A -- B -- C -- D   feature 
                    \end{verbatim}
                    \bigbreak \noindent 
                    Maybe $B$, $C$, and $D$ are really one logical change.
                    \bigbreak \noindent 
                    \begin{bashcode}
                        B: Add parser
                        C: Fix typo in parser
                        D: Fix parser edge case
                    \end{bashcode}
                    \bigbreak \noindent 
                    With interactive rebase, you can squash them:
                    \bigbreak \noindent 
                    \begin{bashcode}
                        pick B Add parser
                        squash C Fix typo in parser
                        squash D Fix parser edge case
                    \end{bashcode}
                    \bigbreak \noindent 
                    After the rebase, Git creates one new commit containing the combined changes:
                    \bigbreak \noindent 
                    \begin{verbatim}
                       A -- B' 
                    \end{verbatim}
                    \bigbreak \noindent 
                    Because Git gives you a list of commits, you can literally move lines around. Suppose the original order is 
                    \bigbreak \noindent 
                    \begin{bashcode}
                        pick B Add UI
                        pick C Add database schema
                        pick D Connect UI to database
                    \end{bashcode}
                    \bigbreak \noindent 
                    You might change it to
                    \bigbreak \noindent 
                    \begin{bashcode}
                        pick C Add database schema
                        pick B Add UI
                        pick D Connect UI to database 
                    \end{bashcode}
                    \bigbreak \noindent 
                    This makes Git replay the database commit first, then the UI commit, then the connection commit.
                    \bigbreak \noindent 
                    This is useful when the original order was accidental or confusing. However, the reordered commits still need to make sense. If commit $D$ depends on code from $B$, putting $D$ before $B$ may cause conflicts or broken intermediate commits.
                    \bigbreak \noindent 
                    We can remove a commit from the branch history by replacing \textbf{pick} with \textbf{drop} or simply deleting the line. This is different from git revert. A revert creates a new commit that undoes an old one. Interactive rebase rewrites history as if the bad commit was never there.

                \item \textbf{More specific rebasing}: You can also use --onto for more specific rebasing:                   
                    \bigbreak \noindent 
                    \begin{bashcode}
                    rebase --onto new_base old_base branch
                    \end{bashcode}
                    \bigbreak \noindent 
                    Take the commits after old\_base on branch, and replay them onto new\_base. That lets you move a branch onto almost any commit in the repository.
                \item \textbf{git pull}: git pull is essentially git fetch followed by either git merge or git rebase, depending on configuration/options. Because the merge/rebase part can update the index, git pull can affect the staging area indirectly.
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git pull
                    git pull --rebase
                    \end{bashcode}
                \item \textbf{git stash}: git stash records the current state of both the working directory and the index, then reverts the working directory to match HEAD.
                    \bigbreak \noindent 
                    \begin{bashcode}
                        git stash

                        git stash --staged
                        git stash --keep-index
                        git stash pop
                        git stash apply
                    \end{bashcode}
                    \bigbreak \noindent 
                    git stash pop and git stash apply can also modify the index depending on how the stash was created and applied.
            \end{itemize}
        \item \textbf{Low-level index manipulations}:
            \begin{itemize}
                \item \textbf{git update-index}: This is the direct plumbing command for modifying the index. It can add entries, remove entries, refresh stat information, change executable bits, set assume-unchanged, set skip-worktree, and more. Git’s own documentation says git update-index “modifies the index.”
                    \bigbreak \noindent 
                    \begin{bashcode}
                        git update-index --chmod=+x script.sh
                        git update-index --assume-unchanged config.local
                        git update-index --skip-worktree config.local
                    \end{bashcode}
                \item \textbf{git read-tree}: Low-level command for reading tree objects into the index. It is used internally by commands such as checkout, merge, and reset-like operations.
                    \bigbreak \noindent 
                    \begin{bashcode}
                     git read-tree HEAD
                    \end{bashcode}
            \end{itemize}
        \item \textbf{git ls-files}: git ls-files lists files that Git knows about in the index/staging area, and optionally files in the working tree
            \bigbreak \noindent 
            \begin{bashcode}
            git ls-files
            \end{bashcode}
            \bigbreak \noindent 
            This prints the files that are currently tracked by Git, because tracked files are recorded in the index.
            \bigbreak \noindent 
            \begin{bashcode}
            git ls-files --others
            \end{bashcode}
            \bigbreak \noindent 
            Shows untracked files
            \bigbreak \noindent 
            \begin{bashcode}
            git ls-files --others --exclude-standard
            \end{bashcode}
            \bigbreak \noindent 
            Shows untracked files, but respect .gitignore.
            \bigbreak \noindent 
            \begin{bashcode}
            git ls-files --ignored --others --exclude-standard
            \end{bashcode}
            \bigbreak \noindent 
            \begin{bashcode}
            git ls-files --deleted
            \end{bashcode}
            \bigbreak \noindent 
            Shows deleted tracked files
            \bigbreak \noindent 
            \begin{bashcode}
            git ls-files --modified
            \end{bashcode}
            \bigbreak \noindent 
            Shows modified tracked files
            \bigbreak \noindent 
            \begin{bashcode}
            git ls-files --unmerged
            \end{bashcode}
            \bigbreak \noindent 
            Shows files with merge conflicts.
        \item \textbf{git status}: Shows the current state of your repo relative to the current commit, usually HEAD.
        \item \textbf{git log}: Shows the commit history
            \bigbreak \noindent 
            \begin{bashcode}
            git log
            \end{bashcode}
            \bigbreak \noindent 
            Show recent commits only:
            \bigbreak \noindent 
            \begin{bashcode}
                git log -5
            \end{bashcode}
            \bigbreak \noindent 
            Show commits affecting a specific file:
            \bigbreak \noindent 
            \begin{bashcode}
                git log -- main.c
            \end{bashcode}
            \bigbreak \noindent 
            Show patches introduced by each commit:
            \bigbreak \noindent 
            \begin{bashcode}
                git log -p
            \end{bashcode}
            \bigbreak \noindent 
            Show summary of changed files:
            \begin{bashcode}
                git log --stat
            \end{bashcode}
        \item \textbf{Tracking}: In Git, to \textbf{track} means: A local branch is linked to a remote branch so Git knows which remote branch it should compare, pull from, and push to by default.
        \item \textbf{Upstreams}: An upstream is the remote branch that your local branch is configured to “track.”
            \bigbreak \noindent 
            For example, suppose you have:
            \bigbreak \noindent 
            \begin{bashcode}
            main
            origin/main
            \end{bashcode}
            \bigbreak \noindent 
            If your local \texttt{main} is tracking \texttt{origin/main}, then origin/main is the upstream of main. You can see this with
            \bigbreak \noindent 
            \begin{bashcode}
            git branch -vv
            \end{bashcode}
            \bigbreak \noindent 
            \begin{bashcode}
            * main  abc1234 [origin/main] Add parser notes
            \end{bashcode}
            \bigbreak \noindent 
            So,  local branch main has upstream origin/main. The upstream matters because commands like these use it automatically.
            \bigbreak \noindent 
            \begin{bashcode}
            git pull
            git push 
            git status
            \end{bashcode}
            \bigbreak \noindent 
            If your branch has an upstream, then
            \bigbreak \noindent 
            \begin{bashcode}
            git push
            \end{bashcode}
            \bigbreak \noindent 
            usually means
            \bigbreak \noindent 
            \begin{bashcode}
            git push origin main
            \end{bashcode}
            \bigbreak \noindent 
            Also,
            \bigbreak \noindent 
            \begin{bashcode}
            git pull
            \end{bashcode}
            \bigbreak \noindent 
            usually means
            \bigbreak \noindent 
            \begin{bashcode}
            git fetch origin
            git merge origin/main
            \end{bashcode}
        \item \textbf{Setting an upstream}: When you first push a local branch, you often use
            \bigbreak \noindent 
            \begin{bashcode}
            git push -u origin main
            \end{bashcode}
            \bigbreak \noindent 
            Or, equivalently
            \bigbreak \noindent 
            \begin{bashcode}
            git push --set-upstream origin main
            \end{bashcode}
            \bigbreak \noindent 
            The -u means \textit{set upstream}.
            \bigbreak \noindent 
            You can also manually set an upstream:
            \bigbreak \noindent 
            \begin{bashcode}
            git branch --set-upstream-to=origin/main main
            \end{bashcode}
            \bigbreak \noindent 
            Or, if already on main,
            \bigbreak \noindent 
            \begin{bashcode}
            git branch --set-upstream-to=origin/main
            \end{bashcode}
        \item \textbf{Upstreams and branches}: You do not always need to set an upstream for a branch.
            \bigbreak \noindent 
            You only need an upstream if you want Git to know which remote branch your local branch corresponds to for commands like:
            \bigbreak \noindent 
            \begin{bashcode}
            git pull
            git push
            git status
            \end{bashcode}
            \bigbreak \noindent 
            Suppose you create a local branch
            \bigbreak \noindent 
            \begin{bashcode}
            git switch -c feature
            \end{bashcode}
            \bigbreak \noindent 
            At this point, feature is just local. It does not automatically have an upstream unless you created it from a remote branch in a tracking-aware way.
            \bigbreak \noindent 
            If you only want to work locally, no upstream is needed.
            \bigbreak \noindent 
            But if you want to push it, you usually do
            \bigbreak \noindent 
            \begin{bashcode}
                git push -u origin feature 
            \end{bashcode}
            \bigbreak \noindent 
            This does two things
            \begin{enumerate}
                \item Creates / updates the remote branch
                    \bigbreak \noindent 
                    \begin{bashcode}
                    origin/feature
                    \end{bashcode}
                \item Sets your local branch \textbf{feature} to track \textbf{origin/feature}.
            \end{enumerate}
        \item \textbf{Remote-tracking branch}: A remote-tracking branch is Gits local record of a branch on a remote.
            \bigbreak \noindent 
            \begin{bashcode}
            origin/main
            origin/dev
            origin/feature-x
            \end{bashcode}
            \bigbreak \noindent 
            These are not branches you usually edit directly. They are Git’s local snapshots of what the remote branches looked like the last time you fetched.
            \bigbreak \noindent 
            An \textbf{upstream} is a relationship between your local branch and one of those remote-tracking branches.
        \item \textbf{Rebase}: A rebase moves your commits so that they appear to be based on a different commit. Suppose we have history
            \begin{verbatim}
                A---B---C  origin/main
                    \
                    D---E  feature
            \end{verbatim}
            \bigbreak \noindent 
            You made commits $D$ and $E$ on feature, but meanwhile main advanced to $C$. If you run
            \bigbreak \noindent 
            \begin{bashcode}
            git switch feature
            git rebase origin/main
            \end{bashcode}
            \bigbreak \noindent 
            Git rewrites your branch like this:
            \begin{verbatim}
                A---B---C  origin/main
                         \
                          D'---E'  feature
            \end{verbatim}
            \bigbreak \noindent 
            Your changes are replayed on top of the latest origin/main.
            \bigbreak \noindent 
            Notice the commits are now called $D^{\prime}$ and $E^{\prime}$, not $D$ and $E$. That is because rebasing creates new commits with new commit hashes.
        \item \textbf{Rebase vs merge}: Both merge and rebase can bring one branch up to date with another branch, but they do it differently.
            \begin{itemize}
               \item \textbf{Merge}:  
                   \bigbreak \noindent 
                   \begin{bashcode}
                   git merge origin/main
                   \end{bashcode}
                   \bigbreak \noindent 
                   creates a new merge commit:
                   \begin{verbatim}
                       A---B----C  origin/main
                           \     \
                           D---E--M  feature 
                   \end{verbatim}
                   \bigbreak \noindent 
                   The history preserves the fact that the branches diverged and were merged.
                \item \textbf{Rebase}:  
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git rebase origin/main
                    \end{bashcode}
                    \bigbreak \noindent 
                    rewrites the feature branch
                    \begin{verbatim}
                       A---B---C  origin/main
                                \
                                 D'---E'  feature 
                    \end{verbatim}
                    \bigbreak \noindent 
                    The history looks linear, as if you started your feature branch from the latest main. Rebase is commonly used to keep a clean, linear history.
                    \bigbreak \noindent 
                    For example, before merging a feature branch, you might do:
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git fetch origin
                    git rebase origin/main
                    \end{bashcode}
                    \bigbreak \noindent 
                    This updates your feature branch so your commits are placed after the newest commits on main.
                \end{itemize}

                \item \textbf{Rebasing with upstream}: If your branch has an upstream, you can do:
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git pull --rebase
                    \end{bashcode}
                    \bigbreak \noindent 
                    This means:
                    \bigbreak \noindent 
                    \begin{bashcode}
                    git fetch
                    git rebase <upstream>
                    \end{bashcode}
                    \bigbreak \noindent 
                    So instead of fetching and merging the upstream branch, Git fetches and rebases your local commits on top of it.
                \item \textbf{Blobs}: a blob is the object type that stores the contents of a file.
                    \bigbreak \noindent 
                    A blob does not store the file name, path, permissions, or directory structure. It only stores the raw file data.
                    \bigbreak \noindent 
                    If you have
                    \bigbreak \noindent 
                    \begin{bashcode}
                    src/main.c
                    \end{bashcode}
                    \bigbreak \noindent 
                    Git stores the contents of main.c as a \textbf{blob}. The filename main.c and the path src/main.c are stored separately in a \textbf{tree object}.
                    \bigbreak \noindent 
                    \begin{itemize}
                        \item \textbf{blob}: file contents
                        \item \textbf{tree}:  directory listing: names + permissions + pointers to blobs/trees
                        \item \textbf{commit}:  snapshot metadata + pointer to a tree
                    \end{itemize}
                \item \textbf{Summary}: 
                    \begin{itemize}
                        \item \textbf{repository}: A repository is a collection of commits, each of which is an archive of what the project's working tree looked like at a past date, whether on your machine or someone else's. It also defines HEAD (see below), which identifies the branch or commit the current working tree stemmed from. Lastly, it contains a set of branches and tags, to identify certain commits by name.
                        \item \textbf{index}: Unlike other, similar tools you may have used, Git does not commit changes directly from the working tree into the repository. Instead, changes are first registered in something called the index. Think of it as a way of “confirming” your changes, one by one, before doing a commit (which records all your approved changes at once). Some find it helpful to call it instead as the “staging area”, instead of the index.
                        \item \textbf{working tree}: A working tree is any directory on your filesystem which has a repository associated with it (typically indicated by the presence of a sub-directory within it named .git.). It includes all the files and sub-directories in that directory.
                        \item \textbf{commit}: A commit is a snapshot of your working tree at some point in time. The state of HEAD (see below) at the time your commit is made becomes that commit’s parent. This is what creates the notion of a “revision history”.
                        \item \textbf{branch}: A branch is just a name for a commit (and much more will be said about commits in a moment), also called a reference. It’s the parentage of a commit which defines its history, and thus the typical notion of a “branch of development”.
                        \item \textbf{tag}: A tag is also a name for a commit, similar to a branch, except that it al- ways names the same commit, and can have its own description text.
                        \item \textbf{master}: The mainline of development in most repositories is done on a branch called “master”. Although this is a typical default, it is in no way special.
                        \item \textbf{HEAD}: HEAD is used by your repository to define what is currently checked out:
                            \begin{itemize}
                                \item If you checkout a branch, HEAD symbolically refers to that branch, indicating that the branch name should be updated after the next commit operation.
                                \item If you checkout a specific commit, HEAD refers to that commit only. This is referred to as a detached HEAD, and occurs, for example, if you check out a tag name.
                            \end{itemize}
                    \end{itemize}
                    \bigbreak \noindent 
                    \fig{.6}{./figures/1.png}
    \end{itemize}

    \pagebreak 
    \unsect{Blobs and trees}
    \begin{itemize}
        \item \textbf{Data is immutable}: Once Git stores an object in the repository, Git does not modify that object in place. Instead, if something changes, Git creates a \textbf{new object} with a \textbf{new hash}.
            \bigbreak \noindent 
            Git’s object database stores objects such as:
            \begin{itemize}
                \item \textbf{blob   }: file contents
                \item \textbf{tree   }: directory structure
                \item \textbf{commit }: snapshot metadata + pointer to a tree
                \item \textbf{tag    }: annotated tag object
            \end{itemize}
            Each object is identified by a hash of its contents. For example, a blob’s ID is computed from the blob’s data plus a small header.
            \begin{align*}
                \text{hash(contents)} = \text{object ID}
            .\end{align*}
            So, if the contents change, the hash changes.
            \bigbreak \noindent 
            \begin{bashcode}
                echo "hello" | git hash-object --stdin
                echo "hello world" | git hash-object --stdin
            \end{bashcode}
            \bigbreak \noindent 
            Those produce different object IDs because the contents are different. That means Git cannot just “edit” an existing blob. If it did, the object’s contents would no longer match its hash. The old object ID would become invalid.
            \bigbreak \noindent 
            Git does this
            \bigbreak \noindent 
            \begin{bashcode}
                blob abc123: "hello"
                blob def456: "hello world"
            \end{bashcode}
            \bigbreak \noindent 
            The old object still exists. The new version is stored as a separate object. The same idea applies to commits. A commit contains data like:
            \bigbreak \noindent 
            \begin{bashcode}
                tree pointer
                parent commit pointer
                author
                committer
                message
                timestamp
            \end{bashcode}
            \bigbreak \noindent 
            If any of that changes, the commit hash changes. So when you “amend” a commit, Git does not actually edit the old commit. It creates a new commit object.
        \item \textbf{Blobs and trees}: A blob represents the contents of one file at one particular version.
            \bigbreak \noindent 
            A \textbf{Git tree} is basically Git's representation of a directory. A  tree object contains entries. Each entry records
            \begin{center}
                mode \qquad name \qquad object id
            \end{center}
            For files, the object ID points to a blob
            \begin{center}
                100644      \qquad main.c      \qquad blob abc123
                100644      \qquad README.md   \qquad blob def456
            \end{center}
            For subdirectories, the object ID points to another \textbf{tree}.
        \item \textbf{Object modes}: In Git, object modes are the mode values stored in tree objects to describe what kind of entry each path is.
            \begin{center}
                \begin{tabularx}{\textwidth}{@{}XX@{}}
                    \toprule
                    \textbf{Mode} &	\textbf{Meaning} \\
                    \midrule
                    100644	&Regular file, not executable\\[2ex]
                    100755	&Regular file, executable\\[2ex]
                    040000	&Directory, represented by a tree object\\[2ex]
                    120000	&Symbolic link\\[2ex]
                    160000	&Git submodule, also called a gitlink \\
                    \bottomrule
                \end{tabularx}
            \end{center}

        \item \textbf{Intro}: What Git does is quite rudimentary: it maintains snapshots of a directory’s contents. Much of its internal design can be understood in terms of this basic task.
            \bigbreak \noindent 
            The design of a Git repository in many ways mirrors the structure of a UNIX filesystem: A filesystem begins with a root directory, which typically consists of other directories, most of which have leaf nodes, or files, that contain data. Meta-data about these files’ contents is stored both in the directory (the names), and in the i-nodes that reference the contents of those file (their size, type, permissions, etc). Each i-node has a unique number that identifies the contents of its related file. And while you may have many directory entries pointing to a particular i-node (i.e., hard-links), it’s the i-node which “owns” the contents stored on your filesystem.
            \bigbreak \noindent 
            Internally, Git shares a strikingly similar structure, albeit with one or two key differences. First, it represents your file’s contents in blobs, which are also leaf nodes in something awfully close to a directory, called a tree. Just as an i-node is uniquely identified by a system-assigned number, a blob is named by computing the SHA1 hash id of its size and contents. For all intents and purposes this is just an arbitrary number, like an i-node, except that it has two additional properties: first, it verifies the blob’s contents will never change; and second, the same contents shall always be represented by the same blob, no matter where it appears: across commits, across repositories — even across the whole Internet. If multiple trees reference the same blob, this is just like hard-linking: the blob will not disappear from your repository as long as there is at least one link remaining to it.
            \bigbreak \noindent 
            The difference between a Git blob and a filesystem’s file is that a blob stores no metadata about its content. All such information is kept in the tree that holds the blob. One tree may know those contents as a file named “foo” that was created in August 2004, while another tree may know the same contents as a file named “bar” that was created five years later. In a normal filesystem, two files with the same contents but with such different metadata would always be represented as two independent files. Why this difference? Mainly, it’s because a filesystem is designed to support files that change, whereas Git is not. The fact that data is immutable in the Git repository is what makes all of this work and so a different design was needed. And as it turns out, this design allows for much more compact storage, since all objects having identical content can be shared, no matter where they are.
            \bigbreak \noindent 
            \textbf{Note:} Git tracks commits. A commit points to a tree, the tree maps filenames to blobs, and the blobs store file contents.
        \end{itemize}

            \item \textbf{The blob}: We can create a simple example,
                \bigbreak \noindent 
                \begin{bashcode}
                    mkdir sample && cd sample && echo "Hello, world!" > greeting
                    git hash-object greeting # -> af5626b4a114abcb82d63db7c8082c3c4756e51b
                \end{bashcode}
                \bigbreak \noindent 
                No matter where you run \texttt{git hash-object}, if the file has only the content \textit{Hello, world!}, the hash will be the same.
                \bigbreak \noindent 
                If another repository has a commit whose tree refers to the same blob object, then when we fetch or pull from that repository, Git can recognize that we already have that blob because its object ID is the same. Git only needs to download/store objects that we do not already have.
                \bigbreak \noindent 
                If we now create a repository,
                \bigbreak \noindent 
                \begin{bashcode}
                    git init
                    git add greeting
                    git commit -m "Added greeting"
                \end{bashcode}
                \bigbreak \noindent 
                Our blob should be in the system exactly as we expected, using the hash id de-termined above. As a convenience, Git requires only as many digits of the hash id as are neces-sary to uniquely identify it within the repository. Usually just six or seven digits is enough:
                \bigbreak \noindent 
                \begin{bashcode}
                    git cat-file -t af5626b # -> blob
                    git cat-file blob af5626b # -> Hello, world!
                \end{bashcode}
                \bigbreak \noindent 
                a Git blob represents the fundamental data unit in Git. Really, the whole system is about blob management.
            \item \textbf{Blobs are stored in trees} In order for Git to represent the structure and naming of your files, it attaches blobs as leaf nodes within a tree. Now, I can’t discover which tree(s) a blob lives in just by looking at it, since it may have many, many owners. But I know it must live somewhere within the tree held by the commit I just made:
                \bigbreak \noindent 
                \begin{bashcode}
                    git ls-tree HEAD # -> 100644 blob af5626b4a114abcb82d63db7c8082c3c4756e51b greeting
                \end{bashcode}
                \bigbreak \noindent 
                This commit contains one Git tree, which has a single leaf: the greeting content’s blob.
                \bigbreak \noindent 
                Although I can look at the tree containing my blob by passing HEAD to ls-tree, I haven’t yet seen the underlying tree object referenced by that commit. Here are a few other commands to highlight that difference and thus discover my tree:
                \bigbreak \noindent 
                \begin{bashcode}
                    git rev-parse HEAD # -> 0437733255614428faae0e090712264cf336cd37
                    git cat-file -t HEAD # -> commit
                    git cat-file commit HEAD

                    tree 0563f77d884e4f79ce95117e2d686d7d6e282887
                    author hgoose <noodlecloudcode@gmail.com> 1780326086 -0500
                    committer hgoose <noodlecloudcode@gmail.com> 1780326086 -0500

                    b # -> commit message
                \end{bashcode}
                \bigbreak \noindent 
                The first command decodes the HEAD alias into the commit it references, the second verifies its type, while the third command shows the hash id of the tree held by that commit, as well as the other information stored in the commit object. The hash id for the commit is unique to my repository — because it includes my name and the date when I made the commit — but the hash id for the tree should be common between your example and mine, containing as it does the same blob under the same name.
                \bigbreak \noindent 
                \begin{bashcode}
                    git ls-tree 0563f77 # -> 100644 blob af5626b4a114abcb82d63db7c8082c3c4756e51b greeting
                \end{bashcode}
                \bigbreak \noindent 
                There you have it: my repository contains a single commit, which references a tree that holds a blob — the blob containing the contents I want to record. There’s one more command I can run to verify that this is indeed the case:
                \bigbreak \noindent 
                \begin{bashcode}
                    find .git/objects -type f | sort
                    .git/objects/04/37733255614428faae0e090712264cf336cd37
                    .git/objects/05/63f77d884e4f79ce95117e2d686d7d6e282887
                    .git/objects/af/5626b4a114abcb82d63db7c8082c3c4756e51b
                \end{bashcode}
                \bigbreak \noindent 
                We see three objects. These objects should refer to a tree, a commit, and a blob. Let's take a look
                \bigbreak \noindent 
                \begin{bashcode}
                    git cat-file -t 0437733255
                    commit
                    git cat-file -t 0563f77
                    tree
                    git cat-file -t af5626b
                    blob
                \end{bashcode}
                \bigbreak \noindent 
                Notice the directory name is the MSB of each hash.
                \bigbreak \noindent 
                \begin{bashcode}
                    git show 0437733
                    commit 0437733255614428faae0e090712264cf336cd37 (HEAD -> main)
                    Author: hgoose <noodlecloudcode@gmail.com>
                    Date:   Mon Jun 1 10:01:26 2026 -0500

                    b

                    diff --git a/greeting b/greeting
                    new file mode 100644
                    index 0000000..af5626b
                    --- /dev/null
                    +++ b/greeting
                    @@ -0,0 +1 @@
                    +Hello, world!

                    git show 0563f77
                    tree 0563f77

                    greeting

                    git show af5626b
                    Hello, world!
                \end{bashcode}
            \item \textbf{How trees are made}: Every commit holds a single tree, but how are trees made? We know that blobs are created by stuffing the contents of your files into blobs — and that trees own blobs — but we haven’t yet seen how the tree that holds the blob is made, or how that tree gets linked to its parent commit.
                \bigbreak \noindent 
                Let’s start with a new sample repository again, but this time by doing things manually, so you can get a feeling for exactly what’s happening under the hood:
                \bigbreak \noindent 
                \begin{bashcode}
                    rm -rf greeting .git
                    echo "Hello, world!" > greeting
                    git init
                    git add greeting
                \end{bashcode}
                \bigbreak \noindent 
                It all starts when you first add a file to the index.
                \bigbreak \noindent 
                \begin{bashcode}
                    git log # Fails, no commits

                    git ls-files --stage 
                    100644 af5626b4a114abcb82d63db7c8082c3c4756e51b 0 greeting
                \end{bashcode}
                \bigbreak \noindent 
                What’s this? I haven’t committed anything to the repository yet, but already an object has come into being. It has the same hash id I started this whole business with, so I know it represents the contents of my greeting file. I could use cat-file -t at this point on the hash id, and I’d see that it was a blob. It is, in fact, the same blob I got the first time I created this sample repository.
                \bigbreak \noindent 
                This blob isn’t referenced by a tree yet, nor are there any commits. At the moment it is only referenced from a file named .git/index, which references the blobs and trees that make up the current index. So now let’s make a tree in the repo for our blob to hang off of:
                \bigbreak \noindent 
                \begin{bashcode}
                    git write-tree # Records the contents of the index in a tree
                    0563f77d884e4f79ce95117e2d686d7d6e282887
                \end{bashcode}
                \bigbreak \noindent 
                This number should look familiar as well: a tree containing the same blobs (and sub-trees) will always have the same hash id. I don’t have a commit object yet, but now there is a tree object in that repository which holds the blob. The purpose of the low-level write-tree command is to take whatever the contents of the index are and tuck them into a new tree for the purpose of creating a commit.
                \bigbreak \noindent 
                I can manually make a new commit object by using this tree directly, which is just what the commit-tree command does:
                \bigbreak \noindent 
                \begin{bashcode}
                    git commit-tree 0563f77 -m "Initial commit"
                    d63d26194dd7a5cfbb9fe7365874c2918707f091
                \end{bashcode}
                \bigbreak \noindent 
                The raw commit-tree command takes a tree’s hash id and makes a commit object to hold it. If I had wanted the commit to have a parent, I would have had to specify the parent commit’s hash id explicitly using the -p option.
                \bigbreak \noindent 
                Our work is not done yet, though, since I haven’t registered the commit as the new head of the current branch:
                \bigbreak \noindent 
                \begin{bashcode}
                    echo d63d26194dd7a5cfbb9fe7365874c2918707f091 > .git/refs/head/master
                \end{bashcode}
                \bigbreak \noindent 
                This command tells Git that the branch name “master” should now refer to our recent com- mit. Another, much safer way to do this is by using the command update-ref
                \bigbreak \noindent 
                \begin{bashcode}
                    git update-ref refs/heads/master d63d2619
                \end{bashcode}
                \bigbreak \noindent 
                After creating master, we must associate our working tree with it. Normally this happens for you whenever you check out a branch.
                \bigbreak \noindent 
                \begin{bashcode}
                    git symbolic-ref HEAD refs/heads/master
                \end{bashcode}
                \bigbreak \noindent 
                This command associates HEAD symbolically with the master branch. This is significant be- cause any future commits from the working tree will now automatically update the value of refs/heads/master.
                \bigbreak \noindent 
                The command git symbolic-ref HEAD refs/heads/master manually forces Git's HEAD pointer to point directly to the master branch without altering your working directory or changing your files. It writes the string ref: refs/heads/master straight into .git/HEAD.
                \bigbreak \noindent 
                It’s hard to believe it’s this simple, but yes, I can now use log to view my newly minted commit
                \bigbreak \noindent 
                \begin{bashcode}
                    git log
                \end{bashcode}
                \bigbreak \noindent 
                A side note: if I hadn’t set refs/heads/master to point to the new commit, it would have been considered “unreachable”, since nothing currently refers to it nor is it the parent of a reach- able commit. When this is the case, the commit object will at some point be removed from the repository, along with its tree and all its blobs. (This happens automatically by a command called gc, which you rarely need to use manually). By linking the commit to a name within refs/ heads, as we did above, it becomes a reachable commit, which ensures that it’s kept around from now on.
            \end{itemize}

            \pagebreak 
            \unsect{Commits}
            \begin{itemize}
                \item \textbf{The beauty of commits}: Some version control systems make “branches” into magical things, often distinguishing them from the “main line” or “trunk”, while others discuss the concept as though it were very different from commits. But in Git there are no branches as separate entities: there are only blobs, trees and commits. Since a commit can have one or more parents, and those commits can have parents, this is what allows a single commit to be treated like a branch: because it knows the whole history that led up to it.
                    \bigbreak \noindent 
                    You can examine all the top-level, referenced commits at any time using the branch command:
                    \bigbreak \noindent 
                    \begin{bashcode}
                        git branch -v
                    \end{bashcode}
                    \bigbreak \noindent 
                    Say it with me: A branch is nothing more than a named reference to a commit. In this way, branches and tags are identical, with the sole exception that tags can have their own descriptions, just like the commits they reference. Branches are just names, but tags are descriptive, well, “tags”.
                    \bigbreak \noindent 
                    But the fact is, we don’t really need to use aliases at all. For example, if I wanted to, I could reference everything in the repository using only the hash ids of its commits. Here’s me being straight up loco and resetting the head of my working tree to a particular commit:
                    \bigbreak \noindent 
                    \begin{bashcode}
                        git reset --hard 5f1bc85
                    \end{bashcode}
                    \bigbreak \noindent 
                    The --hard option says to erase all changes currently in my working tree, whether they’ve been registered for a checkin or not (more will be said about this command later). A safer way to do the same thing is by using checkout:
                    \bigbreak \noindent 
                    \begin{bashcode}
                        git checkout 5f1bc85
                    \end{bashcode}
                    \bigbreak \noindent 
                    The difference here is that changed files in my working tree are preserved. If I pass the -f option to checkout, it acts the same in this case to reset --hard, except that checkout only ever changes the working tree, whereas reset --hard changes the current branch's HEAD to ref- erence the specified version of the tree.
                    \bigbreak \noindent 
                    Another joy of the commit-based system is that you can rephrase even the most complicated version control terminology using a single vocabulary. For example, if a commit has multiple parents, it’s a “merge commit” — since it merged multiple commits into one. Or, if a commit has multiple children, it represents the ancestor of a “branch”, etc. But really there is no difference between these things to Git: to it, the world is simply a collection of commit objects, each of which holds a tree that references other trees and blobs, which store your data. Anything more complicated than this is simply a device of nomenclature.
                    \bigbreak \noindent 
                    \fig{.6}{./figures/3.png}
                \item \textbf{A commit by any other name}: If commits are the key, how you name commits is the doorway to mastery. There are many, many ways to name commits, ranges of commits, and even some of the objects held by commits, which are accepted by most of the Git commands. Here’s a summary of some of the more basic usages: 
                    \begin{itemize}
                        \item \textbf{branchname}: As has been said before, the name of any branch is simply an alias for the most recent commit on that “branch”. This is the same as using the word HEAD whenever that branch is checked out.
                        \item \textbf{tagname}: A tag-name alias is identical to a branch alias in terms of naming a commit. The major difference between the two is that tag alias- es never change, whereas branch aliases change each time a new commit is checked in to that branch.
                        \item \textbf{HEAD}: The currently checked out commit is always called HEAD. If you check out a specific commit — instead of a branch name — then HEAD refers to that commit only and not to any branch. Note that this case is somewhat special, and is called “using a detached HEAD” (I’m sure there’s a joke to be told here…).
                        \item \textbf{c82a22c39cbc32}: A commit may always be referenced using its full, 40-character SHA1 hash id. Usually this happens during cut-and-pasting, since there are typically other, more convenient ways to refer to the same commit.
                        \item \textbf{c82a22c}: You only need use as many digits of a hash id as are needed for a unique reference within the repository. Most of the time, six or seven digits is enough.
                        \item \textbf{name$\land$}: The parent of any commit is referenced using the caret symbol. If a commit has more than one parent, the first is used.
                        \item \textbf{name$\land\land$}: Carets may be applied successively. This alias refers to “the parent of the parent” of the given commit name.
                        \item \textbf{name$\land 2$}: If a commit has multiple parents (such as a merge commit), you can refer to the nth parent using name$\land n$.
                        \item \textbf{name$\sim 10$}: A commit’s $n$th ancestor may be referenced using a tilde ($\sim$) followed by the ordinal number. This type of usage is common with \texttt{rebase -i}, for example, to mean “show me a bunch of recent commits”. This is the same as name$\land\land\land\land\land\land\land\land\land\land$.
                        \item \textbf{name:path:} To reference a certain file within a commit’s content tree, specify that file’s name after a colon. This is helpful with show, or to show the difference between two versions of a committed file:
                            \bigbreak \noindent 
                            \begin{bashcode}
                                git diff HEAD^1:Makefile HEAD^2:Makefile
                            \end{bashcode}
                        \item \textbf{name$\land$\{tree\}}: You can reference just the tree held by a commit, rather than the commit itself.
                        \item \textbf{name1..name2}: This and the following aliases indicate commit ranges, which are supremely useful with commands like log for seeing what’s happened during a particular span of time.
                            \bigbreak \noindent 
                            The syntax to the left refers to all the commits reachable from name2 back to, but not including, name1. If either name1 or name2 is omitted, HEAD is used in its place.
                        \item \textbf{name1...name2}: A “triple-dot” range is quite different from the two-dot version above. For commands like log, it refers to all the commits refer- enced by name1 or name2, but not by both. The result is then a list of all the unique commits in both branches.
                            \bigbreak \noindent 
                            For commands like diff, the range expressed is between name2 and the common ancestor of name1 and name2. This differs from the log case in that changes introduced by name1 are not shown.
                        \item \textbf{master..}: This usage is equivalent to “master..HEAD”. I’m adding it here, even though it’s been implied above, because I use this kind of alias constantly when reviewing changes made to the current branch.
                        \item \textbf{..master}: This, too, is especially useful after you’ve done a fetch and you want to see what changes have occurred since your last rebase or merge.
                        \item \textbf{--since="2 weeks ago"}: Refers to all commits since a certain date.
                        \item \textbf{--until=”1 week ago”}: Refers to all commits up to a certain date.
                        \item \textbf{--grep=pattern}: Refers to all commits whose commit message matches the regular expression pattern.
                        \item \textbf{--committer=pattern}: Refers to all commits whose committer matches the pattern.
                        \item \textbf{--author=pattern}: Refers to all commits whose author matches the pattern. The author of a commit is the one who created the changes it represents. For local development this is always the same as the committer, but when patches are being sent by e-mail, the author and the committer usually differ.
                        \item \textbf{--no-merges}: Refers to all commits in the range that have only one parent — that is, it ignores all merge commits.
                    \end{itemize}
                    Most of these options can be mixed-and-matched. Here is an example which shows the fol- lowing log entries: changes made to the current branch (branched from master), by myself, within the last month, which contain the text “foo”:
                    \bigbreak \noindent 
                    \begin{bashcode}
                        git log --grep="foo" --author="johnw" --since="1 month ago" master..
                    \end{bashcode}
                \end{itemize}

                \pagebreak 
                \unsect{Branching, merging, and rebasing}
                \begin{itemize}
                    \item \textbf{Branching and the power of rebase}: One of Git’s most capable commands for manipulating commits is the innocently-named re- base command. Basically, every branch you work from has one or more “base commits”: the commits that branch was born from.
                        \bigbreak \noindent 
                        Take the following typical scenario, for example. Note that the arrows point back in time because each commit references its parent(s), but not its children. Therefore, the $D$ and $Z$ commits represent the heads of their respective branches:
                        \bigbreak \noindent 
                        \fig{.6}{./figures/4.png}
                        \bigbreak \noindent 
                        In this case, running branch would show two “heads”: $D$ and $Z$, with the common parent of both branches being A. The output of show-branch shows us just this information:
                        \bigbreak \noindent 
                        \begin{bashcode}
                            git branch
                            Z
                            * D
                        \end{bashcode}
                        \bigbreak \noindent 
                        \begin{bashcode}
                            git show-branch
                            ! [Z] Z
                            * [D] D
                            --
                            * [D] D
                            * [D^] C
                            * [D~2] B
                            + [Z] Z
                            + [Z^] Y
                            + [Z~2] X
                            + [Z~3] W
                            +* [D~3] A
                        \end{bashcode}
                        \bigbreak \noindent 
                        Reading this output takes a little getting used to, but essentially it’s no different from the diagram above.
                        \begin{itemize}
                            \item The branch we’re on experienced its first divergence at commit $A$ (also known as commit $D\sim 3$, and even $Z\sim 4$ if you feel so inclined). 
                            \item Reading from bottom to top, the first column (the plus signs) shows a divergent branch named $Z$ with four commits: $W, X, Y$ and $Z$.
                            \item The second column (the asterisks) show the commits which happened on the current branch, namely three commits: $B, C$ and $D$.
                            \item The top of the output, separated from the bottom by a dividing line, identifies the branches displayed, which column their commits are labelled by, and the character used for the labeling.
                        \end{itemize}
                        The action we’d like to perform is to bring the working branch $Z$ back up to speed with the main branch, $D$. In other words, we want to incorporate the work from $B, C$, and $D$ into $Z$
                        \bigbreak \noindent 
                        In other version control systems this sort of thing can only be done using a “branch merge”. In fact, a branch merge can still be done in Git, using merge, and remains needful in the case where $Z$ is a published branch and we don’t want to alter its commit history. 
                        \bigbreak \noindent 
                        \begin{bashcode}
                            git checkout Z
                            git merge D
                        \end{bashcode}
                        \bigbreak \noindent 
                        This is what the repository looks like afterward:
                        \bigbreak \noindent 
                        \fig{.6}{./figures/5.png}
                        \bigbreak \noindent 
                        If we checked out the $Z$ branch now, it would contain the contents of the previous $Z$ (now referenceable as $Z\land$), merged with the contents of $D$. (Though note: a real merge operation would have required resolving any conflicts between the states of $D$ and $Z$).
                        \bigbreak \noindent 
                        Although the new $Z$ now contains the changes from $D$, it also includes a new commit to represent the merging of $Z$ with $D$: the commit now shown as $Z^{\prime}$. This commit doesn’t add anything new, but represents the work done to bring $D$ and $Z$ together. In a sense it’s a “meta-commit”, because its contents are related to work done solely in the repository, and not to new work done in the working tree.
                        \bigbreak \noindent 
                        There is a way, however, to transplant the $Z$ branch straight onto $D$, effectively moving it for- ward in time: by using the powerful rebase command. Here’s the graph we’re aiming for:
                        \bigbreak \noindent 
                        \fig{.6}{./figures/6.png}
                        \bigbreak \noindent 
                        This state of affairs most directly represents what we’d like done: for our local, development branch $Z$ to be based on the latest work in the main branch $D$. That’s why the command is called “rebase”, because it changes the base commit of the branch it’s run from. If you run it repeatedly, you can carry forward a set of patches indefinitely, always staying up-to-date with the main branch, but without adding unnecessary merge commits to your development branch
                        \bigbreak \noindent 
                        \begin{bashcode}
                            git checkout Z 
                            git rebase D # Change Z's base commit to point to D
                        \end{bashcode}
                        \bigbreak \noindent 
                        Why is this only for local branches? Because every time you rebase, you’re potentially changing every commit in the branch. Earlier, when $W$ was based on $A$, it contained only the changes needed to transform $A$ into $W$. After running rebase, however, $W$ will be rewritten to contain the changes necessary to transform $D$ into $W^{\prime}$. Even the transformation from $W$ to $X$ is changed, because $A+W+X$ is now $D+W^{\prime}+X^{\prime}$ - and so on. If this were a branch whose changes are seen by other people, and any of your downstream consumers had created their own local branches off of $Z$, their branches would now point to the old $Z$, not the new $Z^{\prime}$
                        \bigbreak \noindent 
                        Generally, the following rule of thumb can be used: Use rebase if you have a local branch with no other branches that have branched off from it, and use merge for all other cases. Merge is also useful when you’re ready to pull your local branch’s changes back into the main branch.
                    \item \textbf{Interactive rebasing}: When rebase was run above, it automatically rewrote all the commits from $W$ to $Z$ in order to rebase the $Z$ branch onto the $D$ commit (i.e., the head commit of the D branch). You can, how- ever, take complete control over how this rewriting is done. If you supply the -i option to rebase, it will pop you into an editing buffer where you can choose what should be done for every commit in the local $Z$ branch:
                        \begin{itemize}
                            \item \textbf{pick} This is the default behavior chosen for every commit in the branch if you don’t use interactive mode. It means that the commit in question should be applied to its (now rewritten) parent commit. For every commit that involves conflicts, the rebase command gives you an opportunity to re- solve them.
                            \item \textbf{squash}: A squashed commit will have its contents “folded” into the contents of the commit preceding it. This can be done any number of times. If you took the example branch above and squashed all of its commits (except the first, which must be a pick in order to squash), you would end up with a new $Z$ branch containing only one commit on top of $D$. Useful if you have changes spread over multiple commits, but you’d like the history rewritten to show them all as a single commit.
                            \item \textbf{edit}: If you mark a commit as edit, the rebasing process will stop at that commit and leave you at the shell with the current working tree set to reflect that commit. The index will have all the commit’s changes registered for inclusion when you run commit. You can thus make whatever changes you like: amend a change, undo a change, etc.; and after committing, and running rebase --continue, the commit will be rewritten as if those changes had been made originally.
                            \item \textbf{(drop)}: If you remove a commit from the interactive rebase file, or if you comment it out, the commit will simply disappear as if it had never been checked in. Note that this can cause merge conflicts if any of the later commits in the branch depended on those changes.
                        \end{itemize}
                        The power of this command is hard to appreciate at first, but it grants you virtually unlimited control over the shape of any branch. You can use it to:
                        \begin{itemize}
                            \item Collapse multiple commits into single ones.
                            \item Re-order commits.
                            \item Remove incorrect changes you now regret.
                            \item Move the base of your branch onto any other commit in the repository.
                            \item Modify a single commit, to amend a change long after the fact.
                        \end{itemize}

                        \bigbreak \noindent

                        git checkout L

                        \end{bashcode}
                        \bigbreak \noindent 
                        Now, we have

                        As you can see, when it comes to local development, rebasing gives you unlimited control over how your commits appear in the repository.
                    \item \textbf{Rebase only with local development}: 
                        Rebasing is recommended mainly for local development because it rewrites commit history.
                        \bigbreak \noindent 
                        A rebase does not move the same commits. It creates new commits with new parent commits, so they get new hashes.
                        \bigbreak \noindent 
                        Suppose we have the history
                        \bigbreak \noindent 
                        \begin{verbatim}
                            A -- B -- C          main
                                  \
                                   D -- E        feature
                        \end{verbatim}
                        \bigbreak \noindent 
                        If we run
                        \bigbreak \noindent 
                        \begin{bashcode}
                        git checkout feature
                        git rebase main
                        \end{bashcode}
                        \bigbreak \noindent 
                        Git replays $D$ and $E$ on top of $C$.
                        \bigbreak \noindent 
                        \begin{verbatim}
                           A -- B -- C -- D' -- E'    feature 
                        \end{verbatim}
                        \bigbreak \noindent 
                        The commits $D^{\prime}$ and $E^{\prime}$ contain similar changes to D and E, but they are not the same commits. Their parent changed, so their hashes changed.
                        \bigbreak \noindent 
                        That is safe if only your local repository knows about $D$ and $E$.
                        \bigbreak \noindent 
                        The problem happens when other people already have the old commits.
                        \bigbreak \noindent 
                        Suppose we push
                        \bigbreak \noindent 
                        \begin{verbatim}
                            A -- B -- C          main
                                  \
                                   D -- E        feature     
                        \end{verbatim}
                        \bigbreak \noindent 
                        Then, someone else creates work on top of $E$
                        \bigbreak \noindent 
                        \begin{verbatim}
                           A -- B -- C          main
                                 \
                                  D -- E -- F   coworker-work 
                        \end{verbatim}
                        \bigbreak \noindent 
                        Now, you rebase and force-push
                        \bigbreak \noindent 
                        \begin{verbatim}
                           A -- B -- C -- D' -- E'   feature 
                        \end{verbatim}
                        \bigbreak \noindent 
                        Their branch is still based on the old E:
                        \bigbreak \noindent 
                        \begin{verbatim}
                            A -- B -- C -- D' -- E'    feature
                                  \
                                   D -- E -- F         coworker-work
                        \end{verbatim}
                        \bigbreak \noindent 
                        Now the project has two versions of the same logical changes:
                        \bigbreak \noindent 
                        \begin{bashcode}
                            D, E      old commits
                            D', E'    rebased replacements
                        \end{bashcode}
                        \bigbreak \noindent 
                        So, rebase freely before you push. Avoid rebasing commits that other people may already have.
                        \bigbreak \noindent 
                        Rebase unpublished/local history. Do not casually rebase published/shared history.
                        \bigbreak \noindent 
                        You can always undo the effect of a merge commit with
                        \bigbreak \noindent 
                        \begin{bashcode}
                        git revert -m 1 M
                        \end{bashcode}
                        \bigbreak \noindent 
                        The -m 1 means: Treat parent 1 as the mainline parent, usually the branch you were on when you merged.
                        \bigbreak \noindent 
                        Merging is safe because it preserves existing history. Being able to revert the merge commit is a useful recovery mechanism, but the core safety property is that merge is additive rather than history-rewriting.

                \end{itemize}
