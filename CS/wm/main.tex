    \unsect{Introduction}
    \begin{itemize}
        \item \textbf{The X11 WM role}: An X11 window manager is not the operating system’s windowing system. It is a normal X client that is granted special control over the root window’s child windows.
            \bigbreak \noindent 
            The X server owns the screen, keyboard, mouse, windows, pixmaps, cursors, and other graphical resources. Programs such as terminals, browsers, editors, and your window manager are X clients. They all talk to the X server through the X11 protocol.
            \bigbreak \noindent 
            A window manager’s job is to manage other clients’ top-level windows.
            \bigbreak \noindent 
            It usually controls:
            \begin{itemize}
                \item where top-level windows appear
                \item how large they are
                \item whether they are mapped or unmapped
                \item stacking order
                \item focus policy
                \item borders/decorations
                \item workspaces/tags/desktops
                \item fullscreen/maximized/minimized state
                \item how user input is interpreted as window-management commands
            \end{itemize}
            It does not normally draw the contents of applications. Firefox draws Firefox’s content. Kitty draws Kitty’s terminal grid. The window manager controls the container around those windows and their position in the global desktop.
            \bigbreak \noindent 
            Your window manager connects to the X server like any other client:
            \bigbreak \noindent 
            \begin{cppcode}
            Display* dpy = XOpenDisplay(NULL);
            \end{cppcode}
            \bigbreak \noindent 
            This gives your program a connection to the running X server.
            \bigbreak \noindent 
            The X server stores the actual window objects. In Xlib, a Window is just an integer ID referring to a server-side resource.
            \bigbreak \noindent 
            \begin{cppcode}
                Window root = DefaultRootWindow(dpy);
            \end{cppcode}
            That root value is not a C object containing the window. It is an ID you send back to the X server when making requests.
            \bigbreak \noindent 
            Xlib calls such as \textbf{XMoveWindow,} \textbf{XResizeWindow,} \textbf{XMapWindow,} and XDestroyWindow send requests to the server. They do not directly manipulate local objects.
        \item \textbf{X Clients}: An X client is any program connected to the X server.
            \bigbreak \noindent 
            Each client may create one or more windows. Applications usually create a top-level window, which is the main window the window manager cares about.
            \bigbreak \noindent 
            A client creates a window using calls such as:
            \bigbreak \noindent 
            \begin{cppcode}
                XCreateWindow(...)
                XCreateSimpleWindow(...)
            \end{cppcode}
            \bigbreak \noindent 
            Then it usually maps the window:
            \bigbreak \noindent 
            \begin{cppcode}
            XMapWindow(dpy, win);
            \end{cppcode}
            \bigbreak \noindent 
            Mapping means “make this window visible.” But because a window manager exists, the server may not immediately honor that mapping request. Instead, it may redirect the request to the window manager.
        \item \textbf{Top level windows}: A top-level window is usually an application window that is a direct child of the root window. These are the windows the window manager manages.
            \bigbreak \noindent 
            Not every window is top-level. Applications often create child windows inside themselves:
            \bigbreak \noindent 
            A basic window manager usually ignores internal child windows. It manages the top-level windows that live directly under the root window.
            \bigbreak \noindent 
            You can check a windows parent using
            \bigbreak \noindent 
            \begin{cppcode}
            XQueryTree(...)
            \end{cppcode}
            \bigbreak \noindent 
            A window whose parent is the root window is usually a top-level client window.
        \item \textbf{Root window}: The root window is the invisible desktop-sized parent window for the whole screen.
            \bigbreak \noindent 
            Every X screen has a root window. It covers the entire display area.
            \bigbreak \noindent 
            For a simple one-screen setup:
            \bigbreak \noindent 
            \begin{cppcode}
                int screen = DefaultScreen(dpy);
                Window root = RootWindow(dpy, screen);
            \end{cppcode}
            \bigbreak \noindent 
            The root window is important because clients create their top-level windows as children of it.
            \bigbreak \noindent 
            The window manager selects events on the root window so it can intercept requests involving the root’s child windows.
            \bigbreak \noindent 
            A minimal window manager usually does something like:
            \bigbreak \noindent 
            \begin{cppcode}
            XSelectInput(
                dpy,
                root,
                SubstructureRedirectMask | SubstructureNotifyMask
            );
        \end{cppcode}
        \bigbreak \noindent 
        This is the critical step that makes your program the active window manager.
        \bigbreak \noindent 
        The window manager is “special” only because it selects SubstructureRedirectMask on the root window.
        \bigbreak \noindent 
        Only one client may select SubstructureRedirectMask on a given root window at a time.
        \bigbreak \noindent 
        That means only one window manager can run per screen.
        \bigbreak \noindent 
        If another window manager is already running, your attempt to select SubstructureRedirectMask fails with a BadAccess error.
        \bigbreak \noindent 
        \textbf{Note:} The X server creates the root window, not the window manager and not normal applications.
        \bigbreak \noindent 
        When Xorg starts, it initializes each screen. As part of that initialization, it creates one root window per screen. That root window is the top-level ancestor for all other windows on that screen.
    \item \textbf{SubstructureRedirectMask}:  The SubstructureRedirectMask is an Xlib event mask that allows a single client (typically a window manager) to intercept structural requests made to the X server on the subwindows of a parent window
        \bigbreak \noindent 
        When this mask is set on a parent window (like the root window), it intercepts requests from other clients to map, configure (resize/move), or circulate child windows. Instead of executing the command, the X server pauses the action and sends the window manager a specific request even
    \item \textbf{SubstructureNotifyMask}: SubstructureNotifyMask is an Xlib event mask used primarily by window managers and system utilities. When selected on a parent window, it alerts the client to any structural or state changes affecting any of its direct child windows. 
        \bigbreak \noindent 
        When the X Server receives SubstructureNotifyMask, it can report the following state-change events for children
        \begin{itemize}
            \item \textbf{CreateNotify:} A new child window was created.
            \item \textbf{DestroyNotify:} A child window was destroyed.
            \item \textbf{MapNotify:} A child window was mapped (made visible).
            \item \textbf{UnmapNotify:} A child window was unmapped (hidden).
            \item \textbf{ConfigureNotify:} A child window changed size, position, border, or stacking order.
            \item \textbf{ReparentNotify:} A child window was reparented to a different window.
            \item \textbf{CirculateNotify:} A child window's stacking order was circulated.
            \item \textbf{GravityNotify:} A child window was moved due to window gravity. 
        \end{itemize}
        You should ideally install a temporary X error handler before this so you can detect whether another window manager already owns the root.
        \bigbreak \noindent 
        \begin{enumerate}
            \item \textbf{normal client:} asks X server to map/configure its window
            \item \textbf{X server:} sees that WM selected SubstructureRedirectMask on root
            \item \textbf{X server:} sends event to WM instead of immediately performing request
            \item \textbf{WM:} decides what to do
            \item \textbf{WM:} sends XMoveResizeWindow / XMapWindow / XConfigureWindow / etc.
        \end{enumerate}
    \item \textbf{Reparenting vs non-reparenting window managers}: A window manager may either manage the application window directly or create a frame window around it.
        \begin{itemize}
            \item \textbf{Non-reparenting WM:} A non-reparenting WM manages the application’s top-level window directly.
            \item \textbf{Reparenting WM}: A reparenting WM creates a frame window, then makes the client window a child of that frame.
                \bigbreak \noindent 
                \begin{verbatim}
                    root window
                    |__ frame window
                        |__ client window
                \end{verbatim}
                The frame belongs to the window manager. The client belongs to the application. This allows the WM to draw title bars, close buttons, resize handles, shadows, and borders independently of the client window.
        \end{itemize}
    \item \textbf{Override-redirect windows}: Some windows are not supposed to be managed by the window manager. These are called \textbf{override-redirect} windows.
        \bigbreak \noindent 
        Your WM should usually ignore override-redirect windows.
    \item \textbf{Workspaces}: In a tiling WM, switching workspaces often means:
        \begin{itemize}
            \item \textbf{Windows on current workspace:} Mapped
            \item \textbf{Windows on hidden workspace:} Unmapped
        \end{itemize}
    \item \textbf{Window lifecycle}: A top-level application window usually goes through this lifecycle:
        \bigbreak \noindent 
        \begin{enumerate}
            \item client connects to X server
            \item client creates top-level window
            \item client sets properties/hints
            \item client asks to map the window
            \item WM receives MapRequest
            \item WM starts managing the window
            \item WM positions/resizes it
            \item WM maps it
            \item user interacts with it
            \item window may be resized/moved/focused/unfocused
            \item client may unmap it
            \item client may destroy it
            \item WM removes it from internal state 
        \end{enumerate}
    \item \textbf{Events to understand}: 
        \begin{itemize}
            \item \textbf{MapRequest}:	A client wants to show a top-level window.
            \item \textbf{ConfigureRequest:}	A client wants to move/resize/restack a top-level window.
            \item \textbf{DestroyNotify:}	A window was destroyed.
            \item \textbf{UnmapNotify}:	A window was hidden/unmapped.
            \item \textbf{MapNotify}:	A window was mapped.
            \item \textbf{EnterNotify}:	Pointer entered a window. Useful for focus-follows-mouse.
            \item \textbf{ButtonPress}:	Mouse button pressed. Useful for moving/resizing/focusing.
            \item \textbf{KeyPress}:	Key pressed. Useful for WM shortcuts.
            \item \textbf{ClientMessage}:	Client sent a protocol message, often EWMH/ICCCM-related.
            \item \textbf{PropertyNotify}:	A window property changed. Useful for titles, hints, states.
        \end{itemize}
    \item \textbf{Desktop backgrounds}: Setting the background of the desktop usually means setting the background of the root window.
        \bigbreak \noindent 
        So, we can convert an image into an X pixmap, set the roots background to that pixmap, and clear / redraw the root window.
    \end{itemize}

    \unsect{Tiling layout model}
    \begin{itemize}
        \item \textbf{Intro}: A tiling layout model is the part of the window manager that decides: Given a monitor rectangle and a set of managed windows, what rectangle should each window occupy?
            \bigbreak \noindent 
            At minimum, a tiling window manager needs to track
            \bigbreak \noindent 
            \begin{cppcode}
                struct Client {
                    Window win;
                    int x, y, w, h;

                    bool is_floating;
                    bool is_fullscreen;
                    bool is_hidden;

                    int min_w, min_h;
                    int max_w, max_h;
                    int base_w, base_h;
                    int resize_inc_w, resize_inc_h;

                    float min_aspect;
                    float max_aspect;
                };
            \end{cppcode}
            \bigbreak \noindent 
            And something like:
            \bigbreak \noindent 
            \begin{cppcode}
                struct Workspace {
                    Client* clients[MAX_CLIENTS];
                    int client_count;

                    LayoutKind layout;

                    int master_count;
                    float master_factor;

                    int gap_inner;
                    int gap_outer;
                    int border_width;
                };
            \end{cppcode}
            \bigbreak \noindent 
            The layout algorithm usually processes only tiled clients, floating and fullscreen clients are layered separately.
        \item \textbf{Master stack layout}: The screen is divided into two areas:
            \bigbreak \noindent 
            \begin{verbatim}
                +-------------------+-------------------+
                |                   |                   |
                |                   |      Stack 1      |
                |                   |                   |
                |      Master       +-------------------+
                |                   |                   |
                |                   |      Stack 2      |
                |                   |                   |
                |                   +-------------------+
                |                   |      Stack 3      |
                +-------------------+-------------------+
            \end{verbatim}
            \begin{itemize}
                \item The focused or first client goes in the master area.
                \item Remaining clients go in the stack area.
                \item master\_factor controls how much width the master area receives.
                \item master\_count controls how many windows are allowed in the master area.
            \end{itemize}
            \bigbreak \noindent 
            With multiple master windows, the master area itself is split vertically.
            \bigbreak \noindent 
            \begin{verbatim}
                +-------------------+-------------------+
                |     Master 1      |      Stack 1      |
                +-------------------+-------------------+
                |     Master 2      |      Stack 2      |
                +-------------------+-------------------+
            \end{verbatim}
        \item \textbf{Monocle / fullscreen layout}: Monocle means every tiled window receives the full usable monitor rectangle.
            \begin{verbatim}
                +-------------------------------+
                |                               |
                |          Window A             |
                |          Window B             |
                |          Window C             |
                |                               |
                +-------------------------------+
            \end{verbatim}
            \bigbreak \noindent 
            They are all the same size, but only the topmost/focused one is visible.
            \begin{itemize}
                \item \textbf{Monocle}: The WM still manages the window normally.
                    \bigbreak \noindent 
                    You may keep borders, gaps, and panels visible depending on your design.
                \item \textbf{Fullscreen}: Fullscreen is usually a special state.
                    \bigbreak \noindent 
                    A fullscreen client should typically:
                    \begin{itemize}
                        cover the entire monitor,
                        ignore gaps,
                        ignore borders or use border width 0,
                        appear above normal tiled clients,
                        possibly cover bars/panels,
                        be excluded from normal tiling.
                    \end{itemize}
                    In EWMH-aware WMs, fullscreen often corresponds to:
                    \bigbreak \noindent 
                    \begin{cppcode}
                    _NET_WM_STATE_FULLSCREEN
                    \end{cppcode}
            \end{itemize}
            \item \textbf{Grid layout}: Grid layout places windows in rows and columns.
                \begin{verbatim}
                    +----------+----------+----------+
                    |    A     |    B     |    C     |
                    +----------+----------+----------+
                    |    D     |    E     |    F     |
                    +----------+----------+----------+
                \end{verbatim}
                \bigbreak \noindent 
                For $n$ clients, you choose
                \begin{align*}
                    \text{cols} &= \left\lceil \sqrt{n} \right\rceil \\
                    \text{rows} &= \left\lceil \frac{n}{\text{cols}} \right\rceil
                .\end{align*}
                Then, for each cell,
                \begin{align*}
                    \text{cell\_w} &= \text{screen\_width} / \text{cols} \\
                    \text{cell\_h} &= \text{screen\_height} / \text{rows}
                .\end{align*}
                A common refinement is to avoid ugly empty cells. For example, with 5 windows:
                \begin{verbatim}
                    +----------+----------+----------+
                    |    A     |    B     |    C     |
                    +----------+----------+----------+
                    |    D     |          E          |
                    +----------+---------------------+
                \end{verbatim}
            \item \textbf{Binary space partitioning}: Binary space partitioning, or BSP, treats the screen as a recursively split rectangle.
                \bigbreak \noindent 
                The monitor begins as one rectangle:
                \begin{verbatim}
                    +-------------------------------+
                    |                               |
                    |            Root               |
                    |                               |
                    +-------------------------------+
                \end{verbatim}
                Then you split it
                \begin{verbatim}
                    +---------------+---------------+
                    |               |               |
                    |       A       |       B       |
                    |               |               |
                    +---------------+---------------+
                \end{verbatim}
                Then one side can split again:
                \begin{verbatim}
                    +---------------+---------------+
                    |               |       B       |
                    |       A       +---------------+
                    |               |       C       |
                    +---------------+---------------+
                \end{verbatim}
                Conceptually, you get a binary tree:
                \begin{verbatim}
                    root
                   /    \
                  A    split
                      /     \
                     B       C
                \end{verbatim}
                \bigbreak \noindent 
                Each internal node is a split:
                \bigbreak \noindent 
                \begin{cppcode}
                    enum SplitKind {
                        SPLIT_HORIZONTAL,
                        SPLIT_VERTICAL
                    };

                    struct Node {
                        bool is_leaf;

                        union {
                            Client* client;

                            struct {
                                Node* first;
                                Node* second;
                                SplitKind split;
                                float ratio;
                            };
                        };
                    };
                \end{cppcode}
            \item \textbf{Manual split trees}: Manual split trees are BSP where the user controls where new windows go. For example,
                \begin{itemize}
                    \item Mod+h  split horizontally
                    \item Mod+v  split vertically
                \end{itemize}
                Then the next window is inserted according to that chosen split.
            \item \textbf{Dynamic tiling}: Dynamic tiling means the layout is generated automatically from the client list and layout mode.
                \bigbreak \noindent 
                In a dynamic tiler, the WM does not store a detailed layout tree for every split. Instead, it stores a list of clients and recomputes positions.
                \bigbreak \noindent 
                \begin{cppcode}
                    clients = [A, B, C, D]
                    layout = MASTER_STACK
                    master_factor = 0.6
                \end{cppcode}
                \bigbreak \noindent 
                Every time something changes, you call
                \bigbreak \noindent 
                \begin{cppcode}
                arrange(workspace);
                \end{cppcode}
                \bigbreak \noindent 
                Events that trigger rearrangement include:
                \begin{itemize}
                    \item new window mapped,
                    \item window closed,
                    \item focus changed,
                    \item layout changed,
                    \item master factor changed,
                    \item monitor resized,
                    \item gap size changed,
                    \item client moved to/from floating state.
                \end{itemize}
                Dynamic tiling is simpler than manual tiling.
            \item \textbf{Floating layer}: Not every window should tile.
                \bigbreak \noindent 
                Some windows should float:
                \begin{itemize}
                    \item dialog boxes,
                    \item file pickers,
                    \item small utility windows,
                    \item splash windows,
                    \item transient windows,
                    \item user-forced floating windows,
                    \item windows with fixed size hints.
                \end{itemize}
                \bigbreak \noindent 
                The important design point is that floating windows are managed by the WM but excluded from tiling.
            \item \textbf{Gaps}: Gaps are empty space between windows
                \bigbreak \noindent 
                There are usually two kinds
                \begin{itemize}
                    \item \textbf{Outer gaps}: Space between the monitor edge and the window layout
                    \item \textbf{Inner gaps}:  Space between adjacent windows
                \end{itemize}
            \item \textbf{Borders}: Border width is part of the windows outer size. If you call
                \bigbreak \noindent 
                \begin{cppcode}
                XMoveResizeWindow(display, win, x, y, w, h);
                \end{cppcode}
                \bigbreak \noindent 
                the $w$ and $h$ refer to the inner window size, not including the border.
            \item \textbf{Resize hints}: Resize hints come from the client application. In X11, you get them with:
                \bigbreak \noindent 
                \begin{cppcode}
                XGetWMNormalHints
                \end{cppcode}
                \bigbreak \noindent 
                They are stored in \textbf{XSizeHints}. Important fields include
                \bigbreak \noindent 
                \begin{cppcode}
                    PMinSize      minimum width/height
                    PMaxSize      maximum width/height
                    PBaseSize     base width/height
                    PResizeInc    resize increment
                    PAspect       aspect ratio constraints
                \end{cppcode}
                \bigbreak \noindent 
                Some applications do not want arbitrary sizes.
            \item \textbf{Aspect-ratio constraints}: Aspect-ratio hints tell the WM that a window wants to stay within a certain width/height ratio.
                \bigbreak \noindent 
                For example, 
                \bigbreak \noindent 
                \begin{cppcode}
                    min aspect = 16/10
                    max aspect = 16/9
                \end{cppcode}
                \bigbreak \noindent 
                The aspect ratio is 
                \begin{align*}
                    \text{ratio} = \text{width} / \text{height}
                .\end{align*}
                If the width is too large for the height, reduce width.
                \bigbreak \noindent 
                If the height is too large for the width, reduce height.

    \end{itemize}

    \unsect{Layout recomputation}
    \begin{itemize}
        \item \textbf{Intro}: A tiling window manager should treat the screen layout as a derived result of its internal state.
            \bigbreak \noindent 
            That is the main idea behind layout recomputation. You do \textbf{not} want the geometry of each window to be the primary thing you manually maintain. Instead, you keep a clean model of the current world, then regenerate window positions from that model whenever something important changes.
            \bigbreak \noindent 
            \begin{center}
                state -> layout algorithm -> window rectangles -> XMoveResizeWindow calls
            \end{center}
            \bigbreak \noindent 
            The window positions are an output, they are not the real source of truth.
        \item \textbf{Why you recompute the whole layout}: Suppose you have three windows $A,B,C$, and window $B$ closes. If you only manually remove $B$, you now need to decide what happens to $C$.
            \bigbreak \noindent 
            The better model is 
            \begin{itemize}
                \item \textbf{Before:}: clients = [A, B, C]
                \item \textbf{After:}: clients = [A, C]
            \end{itemize}
            Then rerun the layout algorithm. 
        \item \textbf{New window appears}: When a new window appears, you usually receive something like MapRequest. You should
            \begin{enumerate}
                \item decide whether to manage the window,
                \item add it to the current workspace’s client list,
                \item set focus if appropriate,
                \item recompute the layout,
                \item map the window.
            \end{enumerate}
        \item \textbf{Window close}: When a window closes, you remove it from your model. Do not just “erase the dead window from the screen.” The X server already destroys/unmaps it. Your job is to update your internal state and then recompute the layout for the remaining windows.

    \end{itemize}

    \unsect{Focus model}
    \begin{itemize}
        \item \textbf{Intro:} A tiling window manager needs a focus model because X11 does not automatically know which client should receive keyboard input. Your WM must decide which window is “active,” tell the X server where keyboard events should go, and keep that decision consistent when windows are created, destroyed, hidden, moved between workspaces, or moved between monitors.
            \bigbreak \noindent 
            In X11 terms, focus mostly means keyboard focus. Pointer/mouse events are separate. A window can receive mouse events because the pointer is over it, while a completely different window receives keyboard events because it has input focus.        
        \item \textbf{Input focus}: Input focus is the X11 mechanism that determines which window receives keyboard events.
            \bigbreak \noindent 
            The core call is usually
            \bigbreak \noindent 
            \begin{cppcode}
                XSetInputFocus(display, window, RevertToPointerRoot, CurrentTime);
            \end{cppcode}
            \bigbreak \noindent 
            The focused window then receives keyboard input, assuming it selected the relevant events or has descendants that do.
            \bigbreak \noindent 
            A window manager usually does not focus the frame window for typing. It normally focuses the actual client window.
            \bigbreak \noindent 
            You may decorate/reparent the client into a frame, but keyboard focus should usually go to the client, not the decoration frame.
            \bigbreak \noindent 
            Focus is not just an X server state. Your WM also needs its own internal state saying which client is focused.
        \item \textbf{Click-to-focus}: Click-to-focus means a window becomes focused only when the user clicks it.
            \bigbreak \noindent 
            This is the easiest policy to implement and reason about.
        \item \textbf{Focus-follows-mouse}: Focus-follows-mouse means the window under the pointer becomes focused when the pointer enters it. The usual X event is \textbf{EnterNotify.}
            \bigbreak \noindent 
            you should usually ignore some EnterNotify events generated by grabs, ungrabs, or window hierarchy changes. X11 crossing events have fields like mode and detail that help distinguish normal pointer motion from synthetic/side-effect transitions.
            \bigbreak \noindent 
            You do not want accidental focus changes caused by:
            \begin{itemize}
                \item Mapping a new window
                \item Unmapping a window
                \item Moving/resizing a window
                \item Grabbing/ungrabbing pointer
                \item Switching workspace
                \item Restacking windows
            \end{itemize}
            So a real implementation often filters enter events.
        \item \textbf{Sloppy focus}: Variant of FFM. The difference is what happens when the pointer leaves a window and enters empty root-window space.
            \begin{itemize}
                \item \textbf{Pointer over window A}: A focused
                \item \textbf{Pointer over root}: A remains focused
            \end{itemize}
            Sloppy focus is often better for tiling WMs because there may be gaps, bars, or empty areas. You usually do not want focus to disappear just because the pointer moved over the root window.
        \item \textbf{Focus direction in a tiling layout}: A tiling WM often supports commands like:
            \begin{itemize}
                \item focus left
                \item focus right
                \item focus up
                \item focus down
            \end{itemize}
            There are two common ways to implement directional focus.
            \begin{itemize}
                \item \textbf{Layout-tree focus}: If your layout is stored as a tree, you navigate the tree
                \item \textbf{Geometry-based focus}: You compare window rectangles. If focused client is $A$, and the user says focus right, find the visible client whose rectangle is to the right of $A$ and closest by distance.
            \end{itemize}
    \end{itemize}

    \unsect{Keyboard control}
    \begin{itemize}
        \item \textbf{Intro}: In a tiling WM, keyboard control is not “text input.” It is a global command interface. You grab certain key combinations on the root window, receive KeyPress events before normal clients do, translate those events into internal commands, then mutate your WM state: focus, layout, workspace, client order, spawning, quitting, and so on.
            \begin{verbatim}
                physical key press
                    -> X11 KeyCode + modifier mask
                    -> WM matches key binding
                    -> dispatches command
                    -> command mutates WM state
                    -> WM rearranges / focuses / spawns / exits
            \end{verbatim}
        \item \textbf{Passive grabs}: A passive key grab means: “when this key combination is pressed, send the event to my WM.”
            \bigbreak \noindent 
            In Xlib, this is usually done with \textbf{XGrabKey.}
            \bigbreak \noindent 
            \textbf{XGrabKey} establishes a passive keyboard grab. When the matching key and modifier combination is pressed, the grab becomes active and the key event is delivered to the grabbing client, usually your WM.
            \bigbreak \noindent 
            For a WM, you usually grab keys on the root window, because the root window is the parent of all top-level client windows. This gives you “global” WM keybindings.
        \item \textbf{Modifier keys}: Modifiers are bitmask attached to key events
            \bigbreak \noindent
            \begin{cppcode}
                ShiftMask
                LockMask       // usually Caps Lock
                ControlMask
                Mod1Mask       // often Alt
                Mod2Mask       // often Num Lock
                Mod3Mask
                Mod4Mask       // often Super / Windows key
                Mod5Mask
            \end{cppcode}
            \bigbreak \noindent 
            Most tiling WMs use Mod4Mask as the main WM modifier:
        \item \textbf{Key symbols}: A KeySym is the symbolic meaning of a key. For example,
            \bigbreak \noindent 
            \begin{cppcode}
                XK_Return
                XK_Tab
                XK_space
                XK_j
                XK_k
                XK_h
                XK_l
                XK_1
                XK_q
            \end{cppcode}
            \bigbreak \noindent 
            These are layout-level symbolic names. You generally write your config using KeySyms because they are readable.
            \bigbreak \noindent 
            Internally you store keysyms, then when registering a grab you convert it to a keycode.
        \item \textbf{Key codes}: A KeyCode is the physical-ish code X11 reports for a key on the keyboard.
            \bigbreak \noindent 
            Your config should usually use KeySym.
            \bigbreak \noindent 
            Your grabs require KeyCode
        \item \textbf{Spawn commands}: A WM does not usually implement a terminal, launcher, browser, or editor itself. It spawns external programs.
            \bigbreak \noindent 
            A spawn command usually forks, closes the X connection in the child, starts a new session, then execs
        

    \end{itemize}

    \unsect{Mouse control}
    \begin{itemize}
        \item \textbf{Intro}: A pure tiling window manager can avoid most mouse logic, but a practical one usually still needs mouse handling for:
            \begin{itemize}
                \item moving floating windows;
                \item resizing floating windows;
                \item interacting with title bars, borders, or decorations;
                \item allowing Mod + mouse controls;
                \item clicking the root window for menus or launchers;
                \item dragging windows between monitors or workspaces.
            \end{itemize}
            In Xlib, mouse control is mostly built out of button events, motion events, and pointer grabs.
        \item \textbf{Button grabs}: A button grab tells the X server: “When this mouse button is pressed with this modifier combination, send the event to the window manager.”
            \bigbreak \noindent 
            This is usually how WMs implement things like:
            \begin{itemize}
                \item Mod + Left Mouse  = move floating window
                \item Mod + Right Mouse = resize floating window
            \end{itemize}
            \bigbreak \noindent 
            We use \textbf{XGrabButton}.
        \item \textbf{Passive grabs}: A passive grab is created ahead of time with \textbf{XGrabButton}. It says “Wake me up when this mouse button/modifier combination happens.” 
        \item \textbf{Active grabs}: An active pointer grab is created with XGrabPointer.
            \bigbreak \noindent 
            It says: “From now until I ungrab, send pointer motion/release events to me.”
            \bigbreak \noindent 
            This is useful during dragging or resizing. Once a move operation starts, you do not want to lose motion events if the pointer leaves the window.
        \item \textbf{Dragging windows}: Dragging a floating window is mostly geometry math.
            \bigbreak \noindent 
            You usually want to move the frame window, not the raw client window, if your WM reparents clients.
            \bigbreak \noindent 
            If your WM is non-reparenting, then you move the client window directly.
        \item \textbf{Resizing floating windows}: Resizing is similar to moving, but instead of changing x and y, you usually change width and height.
        \item \textbf{Root-relative coordinates}: Root-relative coordinates are coordinates relative to the root window.
            \bigbreak \noindent 
            These are usually the best coordinates for dragging and resizing because they are stable globally.
        \item \textbf{Client-relative coordinates}: Client-relative coordinates are relative to the event window.

    \end{itemize}

    \unsect{Borders and visual feedback}
    \begin{itemize}
        \item \textbf{Intro}: In an X11 tiling window manager, borders and visual feedback are how the WM tells the user:
            \begin{itemize}
                \item “this window is focused,”
                \item “this window is urgent,”
                \item “this window is fullscreen,”
                \item “this window belongs to the current layout.”
            \end{itemize}
            Without good visual feedback, a tiling WM feels disorienting even if the layout algorithm works correctly.
        \item \textbf{Borders}: In X11, a top-level application window can have a border controlled through the X server.
            \bigbreak \noindent 
            The classic calls are
            \bigbreak \noindent 
            \begin{cppcode}
                XSetWindowBorderWidth(display, window, width);
                XSetWindowBorder(display, window, pixel_color);  
            \end{cppcode}
            \bigbreak \noindent 
            The border belongs to the X window itself. Your WM does not need to draw a rectangle manually around the client unless you use reparenting.
            \bigbreak \noindent 
            For a simple non-reparenting tiling WM, changing the client window’s border is usually enough.
        \item \textbf{Focused border color}: The focused window should be visually obvious. The focus border is one of the most important parts of the WM. In a tiling environment, the mouse may not clearly indicate which window receives keyboard input, so the border must.
        \item \textbf{Unfocused border color}: Every managed but unfocused tiled window usually gets a neutral border.
        \item \textbf{Border width}: Border width affects both appearance and geometry.
            \bigbreak \noindent 
            If you set
            \bigbreak \noindent 
            \begin{cppcode}
            XSetWindowBorderWidth(dpy, win, 2);
            \end{cppcode}
            \bigbreak \noindent 
            Then, the total visible rectangle is 
            \bigbreak \noindent 
            \begin{cppcode}
                outer_width  = client_width  + 2 * border_width
                outer_height = client_height + 2 * border_width
            \end{cppcode}
        \item \textbf{Urgent window color}: Urgency is usually used when a background window wants attention. For example:
            \begin{itemize}
                \item A terminal bell occurs.
                \item A chat window receives a message.
                \item A dialog appears on another workspace.
                \item A program sets the urgency hint.
            \end{itemize}
            This is commonly exposed through \textbf{WM\_HINTS}. You can query hints with \textbf{XGetWMHints}. You usually clear urgency when the user focuses the window.
            \bigbreak \noindent 
            Urgency is not the same as focus. An urgent window can be unfocused but still have a special border.
        \item \textbf{Fullscreen border suppression}: Fullscreen windows usually should not have borders. When a window is fullscreen, users expect it to occupy the entire monitor:
        \item \textbf{Note:} Do not make focus, layout, and border code all fight each other. A clean model is 
            \begin{itemize}
                \item focus logic decides which client is focused
                \item layout logic decides where clients go
                \item state logic decides fullscreen/urgent/floating
                \item border logic visually reflects that state
            \end{itemize}
            So instead of directly setting borders from everywhere, have one function that says:
            \bigbreak \noindent 
            \begin{cppcode}
            redraw_client_decoration(client);
            \end{cppcode}
            \bigbreak \noindent 
            Or
            \bigbreak \noindent 
            \begin{cppcode}
            update_client_decoration(client);
            \end{cppcode}
            \bigbreak \noindent 
            That function should be the only place where focused, unfocused, urgent, and fullscreen border decisions are made.


    \end{itemize}

    \unsect{Workspaces}
    \begin{itemize}
        \item \textbf{Workspace}: A workspace is a logical collection of windows. Only one workspace might be visible on a given monitor at a time. 
            \bigbreak \noindent 
            In a simple WM, each window belongs to exactly one workspace.
            \bigbreak \noindent 
            So when you switch from workspace 1 to workspace 2, you do something like:
            \bigbreak \noindent 
            \begin{cppcode}
                hide_workspace(old_workspace);
                show_workspace(new_workspace);
                arrange_workspace(new_workspace);
            \end{cppcode}
            \bigbreak \noindent 
            Note that hide usually means unmapping, and show usually means mapping.
        \item \textbf{Tags}: A tag is similar to a workspace, but more flexible.
            \begin{itemize}
                \item \textbf{Workspace model}: A window belongs to one workspace
                \item \textbf{Tag model}: A window may have one or more tags
            \end{itemize}
            For example, in a tag-based WM like dwm, a window can be tagged as both dev and web.
            \begin{itemize}
                \item \textbf{Terminal:}: tags = \{1\}
                \item \textbf{Browser:}:  tags = \{2\}
                \item \textbf{Docs:}:     tags = \{1, 2\}
            \end{itemize}
            If you view tag one you see \textit{Terminal} and \textit{Docs}. If you view tag two you see \textit{Browser} and \textit{Docs}
        \item \textbf{Multi-monitor}: You need to decide whether workspaces are
            \begin{enumerate}
                \item Global, like i3-style workspaces that move between monitors.
                \item Per-monitor, where each monitor has its own independent workspace set.
                \item Tag-based, where each monitor views a tag mask.
            \end{enumerate}
        \item \textbf{Per-workspace layout}: Most tiling WMs allow each workspace to remember its own layout. So, a workspace might store
            \bigbreak \noindent 
            \begin{cppcode}
                enum LayoutKind {
                    LAYOUT_TILE,
                    LAYOUT_MONOCLE,
                    LAYOUT_FLOATING,
                    LAYOUT_GRID
                };

                struct Workspace {
                    Client* clients;
                    Client* focused;
                    enum LayoutKind layout;
                };
            \end{cppcode}
            \bigbreak \noindent 
            When you switch workspaces, you do not globally switch the whole WM’s layout. You use the layout stored inside the workspace:
        \item \textbf{Sticky windows}: A stick window appears on all workspaces. So when switching workspaces, sticky windows remain mapped:
        \item \textbf{Workspaces and EWMH}: If you want your WM to cooperate with panels, pagers, taskbars, and desktop utilities, you should expose workspace state through EWMH properties.
            \bigbreak \noindent 
            Important root window properties include:
            \bigbreak \noindent 
            \begin{cppcode}
                _NET_NUMBER_OF_DESKTOPS
                _NET_CURRENT_DESKTOP
                _NET_DESKTOP_NAMES
                _NET_CLIENT_LIST
                _NET_CLIENT_LIST_STACKING
            \end{cppcode}
            \bigbreak \noindent 
            Important client properties include:
            \bigbreak \noindent 
            \begin{cppcode}
                _NET_WM_DESKTOP
                _NET_WM_STATE
                _NET_WM_STATE_STICKY
            \end{cppcode}
            \bigbreak \noindent 
            For example, 
            \bigbreak \noindent 
            \begin{cppcode}
            _NET_WM_DESKTOP = 0
            \end{cppcode}
            \bigbreak \noindent 
            Means the window is on desktop / workspace 0. Sticky windows are commonly represented with:
            \bigbreak \noindent 
            \begin{cppcode}
            _NET_WM_DESKTOP = 0xFFFFFFFF
            \end{cppcode}
            \bigbreak \noindent 
            or with
            \bigbreak \noindent 
            \begin{cppcode}
            _NET_WM_STATE_STICKY
            \end{cppcode}


    \end{itemize}

    \unsect{Configure requests}
    \begin{itemize}
        \item \textbf{Intro}: In Xorg, configure requests are how a client says: “I would like my top-level window to be moved, resized, raised, lowered, or otherwise reconfigured.”
            \bigbreak \noindent 
            In a normal X session without a window manager, the X server would just do it. With a window manager running, the WM intercepts those requests and decides what actually happens.
            \bigbreak \noindent 
            This is central to a tiling WM because tiling is based on the rule: The layout owns the geometry, not the application.
            \bigbreak \noindent 
            So when a tiled terminal asks to resize itself to 900x600, your WM usually says, “No, your tile is 960x540; stay there.”
        \item \textbf{What a configure request is}: A client can call functions such as:
            \bigbreak \noindent 
            \begin{cppcode}
                XMoveWindow(display, win, x, y);
                XResizeWindow(display, win, width, height);
                XMoveResizeWindow(display, win, x, y, width, height);
                XConfigureWindow(display, win, mask, &changes);
            \end{cppcode}
            \bigbreak \noindent 
            For a top-level application window, this does not immediately happen if your WM has selected SubstructureRedirectMask on the root window.
            \bigbreak \noindent 
            Instead, the X server sends your WM a \textbf{ConfigureRequest} event. The event type is
            \bigbreak \noindent 
            \begin{cppcode}
            XConfigureRequestEvent
            \end{cppcode}
            \bigbreak \noindent 
            If a client is tiled, you should ignore the configure request. For floating windows, dialogs, popups, and special utility windows, you usually do honor the request.
            \bigbreak \noindent 
            A good WM does not blindly honor every request. It still checks:
            \begin{itemize}
                \item minimum size
                \item maximum size
                \item resize increments
                \item aspect ratio
                \item screen bounds
                \item struts/panels
            \end{itemize}
        \item \textbf{Struts and panels}: In the Linux graphical environment, panels (or taskbars) house application launchers and system trays, while struts are the invisible "margins" that tell the window manager to keep applications from covering those panels. They work together to reserve screen space for your desktop UI. 
        \item \textbf{Synthetic ConfigureNotify events}: This is one of the most important practical details.
            \bigbreak \noindent 
            Applications expect to receive \textbf{ConfigureNotify} events when their geometry changes. But if your WM intercepts a configure request and decides not to change the window, the application may still need to be told: “Here is your actual geometry.”
            \bigbreak \noindent 
            This is done with a \textbf{synthetic ConfigureNotify event}. The WM creates the event manually and sends it to the client. Do this when the client requested geometry, the WM denied or modified the request, and the client still needs to know its real position/size.
        \item \textbf{Window gravity}: Window gravity controls how a window’s position should be interpreted when its size changes. The basic question is: If the window changes size, which point should stay fixed?
            \bigbreak \noindent 
            Common gravities include:
            \bigbreak \noindent 
            \begin{cppcode}
                NorthWestGravity
                NorthGravity
                NorthEastGravity
                WestGravity
                CenterGravity
                EastGravity
                SouthWestGravity
                SouthGravity
                SouthEastGravity
                StaticGravity
            \end{cppcode}
    \end{itemize}

    \unsect{Mapping, unmapping, and destruction}
    \begin{itemize}
        \item \textbf{Intro}: In a tiling X11 window manager, this section is about separating X11 window state from your WM’s internal client state.
            \bigbreak \noindent 
            A Window ID can exist, be mapped, be unmapped, be destroyed, be assigned to a workspace, be hidden, or be known to your WM. These are related, but they are not the same thing.
        \item \textbf{Two layers}: You should think in two layers
            \begin{enumerate}
                \item \textbf{X server state}:
                    \begin{itemize}
                        \item Does this X window exist?
                        \item Is it mapped or unmapped?
                        \item Has it been destroyed?
                    \end{itemize}
                \item \textbf{WM state}: 
                    \begin{itemize}
                        \item Do I have a Client struct for it?
                        \item Which workspace is it on?
                        \item Is it visible in the current layout?
                        \item Is it focused?
                        \item Is it floating / tiled / fullscreen?
                    \end{itemize}
            \end{enumerate}
            A common beginner mistake is treating “unmapped” as “dead.” That is wrong.
            \bigbreak \noindent 
            An unmapped window may be:
            \begin{itemize}
                \item a client that exited,
                \item a client that voluntarily withdrew itself,
                \item a window hidden because it is on another workspace,
                \item an iconified/minimized window,
                \item a window your WM temporarily unmapped while rearranging.
            \end{itemize}
            The X server generates \textbf{MapNotify} and \textbf{UnmapNotify} when a window changes between mapped and unmapped states. DestroyNotify is generated when a client destroys a window with XDestroyWindow or related operations.
        \item \textbf{Managing a window}: To manage a window means: your WM creates internal state for it. When you manage a window, you usually:
            When you manage a window, you usually:
            \begin{enumerate}
                \item allocate a Client,
                \item store the Window ID,
                \item select events on the client window,
                \item insert it into your client list,
                \item assign it to the current workspace,
                \item tile it,
                \item map it if it should be visible,
                \item possibly focus it.
            \end{enumerate}
        \item \textbf{Mapping}: For a tiling WM, mapping usually happens when
            \begin{itemize}
                \item a new client appears,
                \item you switch to the workspace containing that client,
                \item a hidden/minimized client is restored,
                \item a client changes state from withdrawn/iconic to normal.
            \end{itemize}
        \item \textbf{Unmapping}: Means to ask the X server to make it not visible. An UnmapNotify can mean very different things depending on who caused it.
            \begin{enumerate}[label=(\alph*)]
                \item \textbf{The client unmapped itself}: The application may be withdrawing its top-level window. In ICCCM terms, newly created top-level windows begin in the Withdrawn state, and transitions between Withdrawn, Normal, and Iconic are driven by mapping, unmapping, and certain WM messages. 
                    \bigbreak \noindent 
                    If a real client window unmaps itself, your WM will often treat that as:
                    \begin{itemize}
                        \item client is no longer normal/manageable
                        \item remove it from tiling data
                        \item clear focus if needed
                    \end{itemize}
                    That is an \textbf{unmanage} operation.
                \item \textbf{Your WM unmapped it to hide it}: Suppose the user switches from workspace 1 to workspace 2.
                    \bigbreak \noindent 
                    \begin{cppcode}
                    for each client on old_workspace:
                        XUnmapWindow(dpy, client->win);

                    for each client on new_workspace:
                        XMapWindow(dpy, client->win);
                    \end{cppcode}
                    \bigbreak \noindent 
                    But those \textbf{XUnmapWindow} calls will also produce \textbf{UnmapNotify.}
                    \bigbreak \noindent 
                    You must not accidentally interpret those as “the client exited.” The client is still managed. It is merely hidden.
                    \bigbreak \noindent 
                    \begin{cppcode}
                        client->hidden = true;
                        client->ignore_unmap++;
                        XUnmapWindow(dpy, client->win);
                    \end{cppcode}
                    \bigbreak \noindent 
                    Then in UnmapNotify:
                    \bigbreak \noindent 
                    \begin{cppcode}
                        void handle_unmap_notify(XUnmapEvent *e) {
                            Client *c = find_client(e->window);
                            if (!c) return;

                            if (c->ignore_unmap > 0) {
                                c->ignore_unmap--;
                                return;
                            }

                            unmanage(c);
                        }
                    \end{cppcode}
                    \bigbreak \noindent 
                    The important point is that you should not blindly unmanage every window that receives \textbf{UnmapNotify.}
            \end{enumerate}
        \item \textbf{Unmanaging a window}: To unmanage a window means to delete your WM's internal record for it. This usually means
            \bigbreak \noindent 
            \begin{cppcode}
                void unmanage(Client *c) {
                    if (!c) return;

                    if (focused == c)
                        focused = NULL;

                    remove_from_workspace(c);
                    remove_from_client_list(c);

                    free(c);

                    arrange();
                    focus_next_valid_client();
                }
            \end{cppcode}
            \bigbreak \noindent 
            Unmanaging should happen when the client is truly gone from the WM’s perspective. Reasons include:
            \begin{itemize}
                \item the client destroyed its top-level window,
                \item the client withdrew itself,
                \item the WM decided this window should no longer be managed,
                \item startup cleanup found stale clients.   
            \end{itemize}
            Unmanage must be idempotent in spirit. In other words, your code should be robust if multiple events arrive close together.
            \bigbreak \noindent 
            \textbf{Note:} Idempotent describes a property of certain actions or operations that yield the same exact result, regardless of whether they are executed once or multiple times. If you repeat the action, it causes no further change beyond the initial application
            \bigbreak \noindent 
            For example, a client may unmap and then destroy. If your \textbf{UnmapNotify} handler already unmanaged it, your \textbf{DestroyNotify} handler must not free the same Client again.
        \item \textbf{Destroyed windows}: A destroyed window is different from an unmapped window. Destroyed means
            \begin{itemize}
                \item The X server object is gone.
                \item The Window ID is no longer valid for normal use.
            \end{itemize}
            After \textbf{DestroyNotify,} you should assume the Window ID is dead. Do not try to focus it, configure it, move it, resize it, restack it, or read properties from it unless you are deliberately handling X errors.
            \bigbreak \noindent 
            Destruction cleanup should clear:
            \begin{itemize}
                focused client,
                last focused client,
                urgent client pointer,
                fullscreen client pointer,
                workspace client references,
                layout stack references,
                any pointer stored in monitor/screen state.
            \end{itemize}
        \item \textbf{Withdrawn state}: In ICCCM terminology, a top-level client window can be in states such as:
            \begin{enumerate}
                \item \textbf{Withdrawn}: Baseline state of a top-level window. It means the window is unmapped, invisible, and completely ignored by the window manager. 
                \item \textbf{Normal}: Application's top-level window should be fully visible, active, and viewable on the screen.
                \item \textbf{Iconic}: Minimized
            \end{enumerate}
            A newly created top-level window starts in the Withdrawn state. It leaves Withdrawn when the client maps it and the WM begins managing it.
            \bigbreak \noindent 
            When the client maps, the WM handles \textbf{MapRequest}, creates a client, and tiles it.
            \bigbreak \noindent 
            When the client withdraws itself, the WM receives an unmap-related event and should remove the client from its managed list. ICCCM specifically notes that WMs should handle the transition to Withdrawn on real UnmapNotify for compatibility.
        \item \textbf{Client exits}: When an application exits, its windows are usually destroyed by the X server. The WM may see 
            \begin{itemize}
                \item UnmapNotify
                \item DestroyNotify
            \end{itemize}
            or just one of them depending on timing and event selection. Your logic should tolerate both. A safe model is 
            \bigbreak \noindent 
            \begin{itemize}
                \item \textbf{UnmapNotify}: If caused by WM hiding/minimizing, ignore as lifecycle removal. Otherwise unmanage.
                \item \textbf{DestroyNotify}: If still managed, unmanage. If not managed, ignore.
            \end{itemize}
    \end{itemize}

    \unsect{Stacking order}
    \begin{itemize}
        \item \textbf{Intro}: In X11, windows are not only arranged in 2D space with $x$, $y$, width, and height. They also have a Z-order, called the stacking order.
            \bigbreak \noindent 
            The stacking order determines which window appears in front of another when they overlap.
            \bigbreak \noindent 
            Even in a tiling window manager, where normal windows usually do not overlap, stacking still matters because of:
            \begin{itemize}
                \item floating windows
                \item fullscreen windows
                \item dialogs
                \item popups
                \item panels/docks
                \item menus
                \item transient windows
                \item focused-window visual behavior
            \end{itemize}
            A tiling window manager still needs a clear stacking policy.
        \item \textbf{Raising}: To raise a window means to move it higher in the stacking order.
            \bigbreak \noindent 
            \begin{cppcode}
            XRaiseWindow(display, window);
            \end{cppcode}
            \bigbreak \noindent 
            This places the window above its siblings. In a tiling WM, you usually do not need to raise every focused tiled window, because tiled windows normally do not overlap. But you may still raise focused floating windows.
        \item \textbf{Lowering}: To lower a window means to move it lower in the stacking order.
            \bigbreak \noindent 
            \begin{cppcode}
            XLowerWindow(display, window); 
            \end{cppcode}
            \bigbreak \noindent 
            Lowering is less common in tiling WMs, but it can be useful when implementing:
            \begin{itemize}
                \item “send to back”
                \item demoting floating windows
                \item keeping tiled windows below floating windows
                \item restoring a normal window after fullscreen mode
            \end{itemize}
        \item \textbf{Restack}: To restack means to explicitly reorder multiple windows.
            \bigbreak \noindent 
            \begin{cppcode}
            XRestackWindows(display, windows, count);
            \end{cppcode}
            \bigbreak \noindent 
            A tiling WM often benefits from having a function like:
            \bigbreak \noindent 
            \begin{cppcode}
            void restack_workspace(Workspace* ws);
            \end{cppcode}
            \bigbreak \noindent 
            That function can enforce your intended stacking layers every time layout changes.
            \bigbreak \noindent 
        \item \textbf{Stacking Policy}: A common stacking policy is
            \begin{verbatim}
                top
                    fullscreen
                    docks/panels
                    floating
                    tiled
                bottom
            \end{verbatim}
            Fullscreen should also interact with panels/docks. Some WMs allow fullscreen windows to cover panels. Others keep panels visible. That is a policy decision.
            \bigbreak \noindent 
            So, another valid policy is 
            \begin{verbatim}
                top
                    docks/panels
                    fullscreen
                    floating
                    tiled
                bottom
            \end{verbatim}
        \item \textbf{Transient windows above parents}: 
            A transient window is a temporary child-like window associated with another top-level window.
            \bigbreak \noindent 
            Examples:
            \begin{itemize}
                \item dialog boxes
                \item file picker windows
                \item confirmation prompts
                \item settings popups
                \item “Save changes?” windows
            \end{itemize}
            In X11, a client can mark a window as transient using:
            \bigbreak \noindent 
            \begin{cppcode}
                XGetTransientForHint(display, win, &parent);
            \end{cppcode}
            \bigbreak \noindent 
            If a window is transient for another window, your WM should usually keep it above its parent.
        \item \textbf{Focused window stacking}: Some WMs raise the focused window. Others do not.
            \bigbreak \noindent 
            In a floating WM, focusing a window usually raises it. For tiled windows, raising on focus usually does not matter because they should not overlap.
        \item \textbf{ConfigureRequest and stacking}: In X11, applications may request changes to their own stacking order. The WM receives these as \textbf{ConfigureRequest} events if it selected \textbf{SubstructureRedirectMask} on the root window.
            \bigbreak \noindent 
            A client might request:
            \begin{itemize}
                \item raise me
                \item lower me
                \item place me above another window
                \item place me below another window
                \item change my size
                \item change my position
            \end{itemize}
            A WM is not required to blindly obey these requests. The WM owns global window policy.
            \bigbreak \noindent 
            For example, if a tiled client requests to raise itself above a fullscreen window, you probably reject or reinterpret that request.
        \item \textbf{Sibling windows}: Stacking order is among sibling windows. In X11, top-level application windows are usually children of the root window. Since they share the same parent, they are siblings.



    \end{itemize}


    \pagebreak 
    \unsect{Error handling}
    \begin{itemize}
        \item \textbf{Intro}: In Xlib, error handling is different from normal C error handling because many X requests do not fail immediately. You often send a request to the X server, continue executing, and only later receive an error event saying that an earlier request was invalid. 
        \item \textbf{Errors are asynchronous}: When you call something like
            \bigbreak \noindent 
            \begin{cppcode}
            XMoveResizeWindow(...)
            \end{cppcode}
            \bigbreak \noindent 
            That function usually does not wait for the X server to confirm success. It writes a request into Xlib’s output buffer and returns.
            \bigbreak \noindent 
            The actual error, if any, may occur later when the server processes the request.
        \item \textbf{The X error handler}: Xlib lets you install a global error handler.
            \bigbreak \noindent 
            \begin{cppcode}
                int x_error_handler(Display* display, XErrorEvent* error) {
                    char msg[1024];

                    XGetErrorText(display, error->error_code, msg, sizeof(msg));

                    fprintf(stderr,
                        "X error: %s\n"
                        "  request_code: %d\n"
                        "  minor_code: %d\n"
                        "  resourceid: 0x%lx\n",
                        msg,
                        error->request_code,
                        error->minor_code,
                        error->resourceid
                    );

                    return 0;
                }
            \end{cppcode}
            \bigbreak \noindent 
            Then, install it 
            \bigbreak \noindent 
            \begin{cppcode}
            XSetErrorHandler(x_error_handler);
            \end{cppcode}
            \bigbreak \noindent 
            The handler is called when the X server reports an error for one of your requests.
            \bigbreak \noindent 
            Important: the error handler is \textbf{global per process}, not per request. You cannot easily attach a normal callback to one X request.
        \item \textbf{Bad window}: BadWindow means you tried to use a window ID that the X server no longer considers valid.
            \bigbreak \noindent 
            This is extremely common in window managers.
            \bigbreak \noindent 
            For example, in 
            \bigbreak \noindent 
            \begin{cppcode}
            XMoveWindow(display, win, 100, 100);
            \end{cppcode}
            \bigbreak \noindent 
            A \textbf{BadWindow} might occur if the client destroyed the window before your request reached the server. A tiling WM should treat many \textbf{BadWindow} errors as normal race conditions, not fatal bugs.
            \bigbreak \noindent 
            But do not blindly ignore everything. A \textbf{BadWindow} can also indicate that your WM state is stale or corrupted.
        \item \textbf{BadAccess}: BadAccess means your request attempted to access something exclusively controlled by another client.
            \bigbreak \noindent 
            For a window manager, the most important case is this: 
            \bigbreak \noindent 
            \begin{cppcode}
                XSelectInput(display, root, SubstructureRedirectMask | SubstructureNotifyMask);
            \end{cppcode}
            \bigbreak \noindent 
            Only one client can select \textbf{SubstructureRedirectMask} on the root window. If another WM is already running, your call will generate \textbf{BadAccess.}
        \item \textbf{Detecting another WM already running}: Because errors are asynchronous, this is not enough:
            \bigbreak \noindent 
            \begin{cppcode}
                XSelectInput(display, root, SubstructureRedirectMask);
                printf("I am the WM now\n");
            \end{cppcode}
            \bigbreak \noindent 
            The error may not have arrived yet. You need to force the server to process the request using \textbf{XSync}.
            \bigbreak \noindent 
            \begin{cppcode}
                static int wm_detected = 0;

                int wm_error_handler(Display* display, XErrorEvent* error) {
                    if (error->error_code == BadAccess) {
                        wm_detected = 1;
                        return 0;
                    }

                    return 0;
                }

                int main() {
                    Display* display = XOpenDisplay(NULL);
                    if (!display) {
                        fprintf(stderr, "Could not open display\n");
                        return 1;
                    }

                    XSetErrorHandler(wm_error_handler);

                    Window root = DefaultRootWindow(display);

                    XSelectInput(display, root,
                        SubstructureRedirectMask |
                        SubstructureNotifyMask
                    );

                    XSync(display, False);

                    if (wm_detected) {
                        fprintf(stderr, "Another window manager is already running\n");
                        return 1;
                    }

                    printf("Window manager started\n");

                    // Main event loop...
                }
            \end{cppcode}
            \bigbreak \noindent 
            The key line is 
            \bigbreak \noindent 
            \begin{cppcode}
            XSync(display, False);
            \end{cppcode}
            \bigbreak \noindent 
            That forces pending requests to be sent to the server and waits until the server has processed them. The second argument controls whether pending events are discarded. Do not use True casually because it discards pending events.
        \item \textbf{Good handler}: A good error handler should print at least
            \bigbreak \noindent 
            \begin{cppcode}
                int x_error_handler(Display* display, XErrorEvent* e) {
                    char buffer[1024];

                    XGetErrorText(display, e->error_code, buffer, sizeof(buffer));

                    fprintf(stderr, "X error: %s\n", buffer);
                    fprintf(stderr, "  error_code: %d\n", e->error_code);
                    fprintf(stderr, "  request_code: %d\n", e->request_code);
                    fprintf(stderr, "  minor_code: %d\n", e->minor_code);
                    fprintf(stderr, "  resourceid: 0x%lx\n", e->resourceid);
                    fprintf(stderr, "  serial: %lu\n", e->serial);

                    return 0;
                }
            \end{cppcode}
            \begin{center}
                \begin{tabularx}{\textwidth}{@{}lX@{}}
                    \toprule
                    \textbf{Field}&	\textbf{Meaning} \\
                    \midrule
                    error\_code	&The actual error, such as BadWindow or BadAccess.\\[2ex]
                    request\_code	&The major X protocol request that failed.\\[2ex]
                    minor\_code	&Extra request-specific code, especially useful for extensions.\\[2ex]
                    resourceid	&The X resource involved, often a Window, Pixmap, GC, etc.\\[2ex]
                    serial	&The request serial number associated with the error. \\
                    \bottomrule
                \end{tabularx}
            \end{center}
            The serial is useful because Xlib assigns sequence numbers to requests. Advanced debugging can compare the error serial to request serials.
        \item \textbf{Common errors}:
            \begin{itemize}
                \item \textbf{BadWindow}:  You tried to operate on a destroyed or invalid window.
                \item \textbf{BadAccess}: You requested access that another client already owns.
                \item \textbf{BadMatch}: The resources are valid, but incompatible for that request.
                \item \textbf{BadDrawable}: You passed an invalid Drawable.
                \item \textbf{BadValue}: One of your numeric arguments was invalid.
                \item \textbf{BadAlloc}: Server could not allocate resource.
            \end{itemize}



    \end{itemize}

    \pagebreak 
    \unsect{Startup sequence / minimal implementation order}
    \begin{itemize}
        \item \textbf{Startup sequence}: A basic WM startup usually does this:
            \begin{enumerate}
                \item Open display.
                \item Get root window.
                \item Install X error handler.
                \item Select events on root window.
                \item Grab keys/buttons.
                \item Scan existing windows.
                \item Manage existing windows.
                \item Enter event loop.
            \end{enumerate}
        \item \textbf{Minimal implementation order}: A good build order is 
            \begin{enumerate}
                \item Start the WM and claim the root window
                \item Handle MapRequest
                \item Track managed clients
                \item Tile all windows in a simple vertical stack
                \item Add focus
                \item Add keybindings
                \item Add master-stack layout
                \item Add window closing
                \item Add workspaces
                \item Add floating windows
                \item Add fullscreen
                \item Add ICCCM/EWMH support
                \item Add multi-monitor suppor
            \end{enumerate}
    \end{itemize}
