    \unsect{Writing makefiles}
    \begin{itemize}
        \item \textbf{What a rule looks like}:
            \bigbreak \noindent 
            \begin{bashcode}
            target ... : prerequisites ...
                recipe 
                ...
                ...
            \end{bashcode}
            \bigbreak \noindent 
            A target is usually the name of a file that is generated by a program; examples of targets are executable or object files. A target can also be the name of an action to carry out, such as ‘clean’
            \bigbreak \noindent 
            A prerequisite is a file that is used as input to create the target. A target often depends on several files.
            \bigbreak \noindent 
            A recipe is an action that make carries out. A recipe may have more than one command, either on the same line or each on its own line. Please note: you need to put a tab character at the beginning of every recipe line! This is an obscurity that catches the unwary. If you prefer to prefix your recipes with a character other than tab, you can set the \textbf{.RECIPEPREFIX} variable to an alternate character
            \bigbreak \noindent 
            Usually a recipe is in a rule with prerequisites and serves to create a target file if any of the prerequisites change. However, the rule that specifies a recipe for the target need not have prerequisites. For example, the rule containing the delete command associated with the target ‘clean’ does not have prerequisites.
            \bigbreak \noindent 
             rule, then, explains how and when to remake certain files which are the targets of the particular rule. make carries out the recipe on the prerequisites to create or update the target. A rule can also explain how and when to carry out an action. 
        \item \textbf{The prereqs}: In a normal Make rule, prerequisites are used by make for timestamp-based rebuild decisions and build ordering.
            \bigbreak \noindent 
            \begin{bashcode}
                main.o : main.c defs.h
                    cc -c main.c
            \end{bashcode}
            \bigbreak \noindent 
            Make checks
            \begin{itemize}
                \item Does main.o exist?
                \item Is main.c newer than main.o?
                \item Is defs.h newer than main.o?
            \end{itemize}
            If main.o does not exist, or if any prerequisite is newer than main.o, then make runs the recipe.
        \item \textbf{There are also order-only prerequisites:}: 
            \bigbreak \noindent 
            \begin{bashcode}
            main.o : main.c | build
                cc -c main.c -o build/main.o
            \end{bashcode}
            The build prerequisite after | means: Make sure build exists before building main.o, but do not rebuild main.o just because build is newer.
            \bigbreak \noindent 
            Before building main.o, make sure there is a target or file-system entry named build.
            \bigbreak \noindent 
            Make treats targets as pathnames. A directory can be a target because it is also a file-system entry.
            \bigbreak \noindent 
            you would normally ad a rule telling make how to create it.
            \bigbreak \noindent 
            \begin{bashcode}
            build:
                mkdir -p build
            \end{bashcode}
        \item \textbf{Target with no prereqs}: In make, whether a target runs depends mainly on whether the target file exists and whether it is out of date.
            \bigbreak \noindent 
            \begin{bashcode}
            clean: 
                rm -rf *.o editor
            \end{bashcode}
            \bigbreak \noindent 
            Make asks: "Does a file named clean exist?". If no file named clean exists, the recipe runs. So,
            \bigbreak \noindent 
            \begin{bashcode}
            make clean
            \end{bashcode}
            \bigbreak \noindent 
            Usually runs every time. But if a file named clean exists, then make may say
            \bigbreak \noindent 
            \begin{bashcode}
                make: 'clean' is up to date
            \end{bashcode}
            \bigbreak \noindent 
            and not run the recipe. This is why command style targets are usually marked .PHONY
            \bigbreak \noindent 
            \begin{bashcode}
            .PHONY: clean

            clean: 
                rm -r *.o editor
            \end{bashcode}
            \bigbreak \noindent 
            \texttt{.PHONY} tells make: "clean is not a real file target. Always run its recipe when requested.".
        \item \textbf{Goals}: Make does not run every rule in the makefile. When you run
            \bigbreak \noindent 
            \begin{bashcode}
            make
            \end{bashcode}
            \bigbreak \noindent 
            make builds the \textbf{default goal}, which is normally the \textbf{first target} in the Makefile.
            \bigbreak \noindent 
            \begin{bashcode}
                editor: main.o kbd.o command.o display.o insert.o search.o files.o utils.o
                    cc -o editor main.o kbd.o command.o display.o insert.o search.o files.o utils.o

                main.o: main.c defs.h
                    cc -c main.c

                clean:
                    rm -f *.o editor
            \end{bashcode}
            \bigbreak \noindent 
            Here, the first target is \textbf{editor}. So, running \texttt{make} means
            \bigbreak \noindent 
            \begin{bashcode}
            make editor
            \end{bashcode}
            \bigbreak \noindent 
            make then builds editor and anything editor depends on. 
        \item \textbf{all}: all is just a conventional target name used to mean: Build the main thing, or build everything important.
            \bigbreak \noindent 
            \begin{bashcode}
            all: editor
            \end{bashcode}
            \bigbreak \noindent 
            means to build all, first build editor.
            \bigbreak \noindent 
            \begin{bashcode}
                .PHONY: all clean

                all: editor

                editor: main.o kbd.o command.o
                    cc -o editor main.o kbd.o command.o

                main.o: main.c defs.h
                    cc -c main.c

                kbd.o: kbd.c defs.h command.h
                    cc -c kbd.c

                command.o: command.c defs.h command.h
                    cc -c command.c

                clean:
                    rm -f *.o editor
            \end{bashcode}
        \item \textbf{Ignoring errors}: 
            \bigbreak \noindent 
            \begin{bashcode}
            clean: 
                -rm -rf *.o editor
            \end{bashcode}
            \bigbreak \noindent 
            The leading - before rm is Makefile syntax, not part of the shell command. It tells make: Run this command, but ignore errors if it fails.
        \item \textbf{What makefile contains}: Makefiles contain five kinds of things: explicit rules, implicit rules, variable definitions, directives, and comments. Rules, variables, and directives are described at length in later chapters 
            \begin{enumerate}
                \item \textbf{An explicit rule} says when and how to remake one or more files, called the rule’s targets. It lists the other files that the targets depend on, called the prerequisites of the target, and may also give a recipe to use to create or update the targets. 
                \item \textbf{An implicit rule} says when and how to remake a class of files based on their names. It describes how a target may depend on a file with a name similar to the target and gives a recipe to create or update such a target
                \item \textbf{A variable definition} is a line that specifies a text string value for a variable that can be substituted into the text later. T
                \item \textbf{A directive} is an instruction for make to do something special while reading the makefile. These include: 
                    \begin{itemize}
                        \item reading another makefile
                        \item Deciding (based on the values of variables) whether to use or ignore a part of the makefile
                        \item Defining a variable from a verbatim string containing multiple lines (
                    \end{itemize}
            \end{enumerate}
        \item \textbf{Comments}: ‘\#’ in a line of a makefile starts a comment. It and the rest of the line are ignored, except that a trailing backslash not escaped by another backslash will continue the comment across multiple lines. A line containing just a comment (with perhaps spaces before it) is effectively blank, and is ignored. If you want a literal \#, escape it with a backslash (e.g., \textbackslash\#). Comments may appear on any line in the makefile, although they are treated specially in certain situations.
            \bigbreak \noindent 
            You cannot use comments within variable references or function calls: any instance of \# will be treated literally (rather than as the start of a comment) inside a variable reference or function call.
            \bigbreak \noindent 
            Within a define directive, comments are not ignored during the definition of the variable, but rather kept intact in the value of the variable. When the variable is expanded they will either be treated as make comments or as recipe text, depending on the context in which the variable is evaluated.
        \item \textbf{Splitting without adding whitespace}: If you need to split a line but do not want any whitespace added, you can utilize a subtle trick: replace your backslash/newline pairs with the three characters dollar sign, backslash, and newline:
            \bigbreak \noindent 
            \begin{bashcode}
                var := one$\
                word
            \end{bashcode}
            After make removes the backslash/newline and condenses the following line into a single space, this is equivalent to:
            \bigbreak \noindent 
            \begin{bashcode}
                var := one$ word
            \end{bashcode}
            Then make will perform variable expansion. The variable reference ‘\$ ’ refers to a variable with the one-character name “ ” (space) which does not exist, and so expands to the empty string, giving a final assignment which is the equivalent of
        \item \textbf{Names make looks for}: When make looks for a makefile, it trues the following names, in order
            \begin{itemize}
                \item GNUmakefile
                \item makefile
                \item Makefile
            \end{itemize}
            GNUmakefile will not be understood by other versions of make. You can specify another name with \texttt{-f (--file)} 
        \item \textbf{Including makefiles}: The include directive tells make to suspend reading the current makefile and read one or more other makefiles before continuing. The directive is a line in the makefile that looks like this:
            \bigbreak \noindent 
            \begin{bashcode}
            include filenames...
            \end{bashcode}
            \bigbreak \noindent 
            Which can contain shell file name patterns. If no files are supplied, nothing is included and no error is printed.
        \item \textbf{The MAKEFILES env variable}: If the environment variable MAKEFILES is defined, make considers its value as a list of names (separated by whitespace) of additional makefiles to be read before the others. This works much like the include directive:
        \item \textbf{How make reads a makefile}: GNU make does its work in two distinct phases. During the first phase it reads all the makefiles, included makefiles, etc. and internalizes all the variables and their values and implicit and explicit rules, and builds a dependency graph of all the targets and their prerequisites. During the second phase, make uses this internalized data to determine which targets need to be updated and run the recipes necessary to update them.
            \bigbreak \noindent 
            It’s important to understand this two-phase approach because it has a direct impact on how variable and function expansion happens; this is often a source of some confusion when writing makefiles. Below is a summary of the different constructs that can be found in a makefile, and the phase in which expansion happens for each part of the construct.
            \bigbreak \noindent 
            We say that expansion is immediate if it happens during the first phase: make will expand that part of the construct as the makefile is parsed. We say that expansion is deferred if it is not immediate. Expansion of a deferred construct part is delayed until the expansion is used: either when it is referenced in an immediate context, or when it is needed during the second phase.
            \bigbreak \noindent 
            A rule is always expanded the same way, regardless of the form:
            \bigbreak \noindent 
            \begin{bashcode}
                immediate : immediate ; deferred
                    deferred
            \end{bashcode}
            \bigbreak \noindent 
            That is, the target and prerequisite sections are expanded immediately, and the recipe used to build the target is always deferred. This is true for explicit rules, pattern rules, suffix rules, static pattern rules, and simple prerequisite definitions.
        \item \textbf{One liners}: In a Makefile, a semicolon after the prerequisites lets you put the first recipe command on the same line as the target rule.
            \bigbreak \noindent 
            \begin{bashcode}
            target: prerequisites ; recipe-command
            \end{bashcode}
            \bigbreak \noindent 
            For example, 
            \bigbreak \noindent 
            \begin{bashcode}
            clean: ; rm -f *.o editor
            \end{bashcode}
            \bigbreak \noindent 
            you can still add more recipe lines afterward:
            \bigbreak \noindent 
            \begin{bashcode}
                clean: ; rm -f *.o editor
                    rm -rf build
            \end{bashcode}
        \item \textbf{Forcing}: FORCE: is a common Makefile trick used to make another target run unconditionally.
            \bigbreak \noindent 
            \begin{bashcode}
            FORCE:

            clean: FORCE
                rm -f *.o editor
            \end{bashcode}
            \bigbreak \noindent 
            FORCE has no prerequisites, and no recipe. Usually no file named FORCE EXISTS. So, when another target depends on it: 
            \bigbreak \noindent 
            \begin{bashcode}
            clean: FORCE
            \end{bashcode}
            \bigbreak \noindent 
            make tries to build FORCE. Since there is normally no real file named FORCE, make treats FORCE as something that is always out of date. That makes clean out of date too, so clean runs.
        \item \textbf{Special built in target names}: 
            \begin{itemize}
                \item \textbf{.PHONY}: The prerequisites of the special target .PHONY are considered to be phony targets. When it is time to consider such a target, make will run its recipe un- conditionally, regardless of whether a file with that name exists or what its last-modification time is. 
                \item \textbf{.SUFFIXES}: The prerequisites of the special target .SUFFIXES are the list of suffixes to be used in checking for suffix rules.
                \item \textbf{.DEFAULT}: The recipe specified for .DEFAULT is used for any target for which no rules are found (either explicit rules or implicit rules).
                    \bigbreak \noindent 
                     If a .DEFAULT recipe is specified, every file mentioned as a prerequi- site, but not as a target in a rule, will have that recipe executed on its behalf.
                \item \textbf{.PRECIOUS}: The targets which .PRECIOUS depends on are given the following special treat- ment: if make is killed or interrupted during the execution of their recipes, the target is not deleted.
                \item \textbf{.INTERMEDIATE}: The targets which .INTERMEDIATE depends on are treated as intermediate files.
                \item \textbf{.NONINTERMEDIATE}: Prerequisites of the special target .NOTINTERMEDIATE are never considered intermediate files. 
                    \bigbreak \noindent 
                    .NOTINTERMEDIATE with no prerequisites causes all targets to be treated as not intermediate.
                    \bigbreak \noindent 
                    If the prerequisite is a target pattern then targets that are built using that pattern rule are not considered intermediate.
                \item \textbf{.SILENT}: If you specify prerequisites for .SILENT, then make will not print the recipe used to remake those particular files before executing them. The recipe for .SILENT is ignored.
                    \bigbreak \noindent 
                    If mentioned as a target with no prerequisites, .SILENT says not to print any recipes before executing them.
                    \bigbreak \noindent 
                    you want to silence all recipes for a particular run of make, use the ‘-s’ or ‘\texttt{--}silent’ option (
                \item \textbf{.IGNORE}: If you specify prerequisites for .IGNORE, then make will ignore errors in execution of the recipe for those particular files. The recipe for .IGNORE (if any) is ignored.
                    \bigbreak \noindent 
                    If mentioned as a target with no prerequisites, .IGNORE says to ignore errors in execution of recipes for all files. This usage of ‘.IGNORE’ is supported only for historical compatibility. Since this affects every recipe in the makefile, it is not very useful; we recommend you use the more selective ways to ignore errors in specific recipes.
                \item \textbf{.ONESHELL}: If .ONESHELL is mentioned as a target, then when a target is built all lines of the recipe will be given to a single invocation of the shell rather than each line being invoked separately.
                \item \textbf{.NOTPARALLEL}: If .NOTPARALLEL is mentioned as a target with no prerequisites, all targets in this invocation of make will be run serially, even if the ‘-j’ option is given. Any recursively invoked make command will still run recipes in parallel 
            \end{itemize}
        \item \textbf{Independent and grouped targets}: A normal rule uses a single colon
            \bigbreak \noindent 
            \begin{bashcode}
                a b: input.txt
                    ./build-one-output $@
            \end{bashcode}
            \bigbreak \noindent 
            This means
            \bigbreak \noindent 
            \begin{bashcode}
                a: input.txt
                    ./build-one-output $@

                b: input.txt
                    ./build-one-output $@
            \end{bashcode}
            \bigbreak \noindent 
            So a and b are treated as separate targets that merely happen to share the same prerequisites and recipe. These are \textbf{independent targets}. The automatic variable \$@ is different for each target:
            \bigbreak \noindent 
            \textbf{Grouped targets} use \&:
            \bigbreak \noindent 
            \begin{bashcode}
                a b &: input.txt
                    ./generate-both input.txt
            \end{bashcode}
            \bigbreak \noindent 
            This means $a$ and $b$ are produced by one single execution of the recipe.
            \bigbreak \noindent 
            Make understands that the recipe creates both files together.
            \bigbreak \noindent 
            So if either $a$ or $b$ is missing or out of date, Make runs the recipe once, and expects both outputs to be updated.

    \end{itemize}

    \unsect{Variables}
    \begin{itemize}
        \item \textbf{Variables}: In a Makefile, variables are names that expand to text. They are mostly used to avoid repeating compiler names, flags, filenames, directories, and commands.
            \bigbreak \noindent 
            \begin{cppcode}
                CC = cc
                CFLAGS = -Wall -Wextra -g
                OBJS = main.o kbd.o command.o display.o insert.o search.o files.o utils.o

                editor: $(OBJS)
                    $(CC) $(CFLAGS) -o editor $(OBJS)
            \end{cppcode}
        \item \textbf{Recursive assignment (=)}: 
            \bigbreak \noindent 
            \begin{bashcode}
                A = $(B)
                B = hello

                all:
                    echo $(A)
            \end{bashcode}
            \bigbreak \noindent 
            With =, the variable is expanded later, when it is used.
        \item \textbf{Intermediate assignment (:=)}: This prints nothing for A, because B had no value when A := \$(B) was evaluated.
            \bigbreak \noindent 
            \begin{bashcode}
                A := $(B)
                B = hello

                all:
                    echo $(A)
            \end{bashcode}
            \bigbreak \noindent 
            \begin{bashcode}
            A = $(B)
            \end{bashcode}
            \bigbreak \noindent 
            Means expand $\$(B)$ later.
            \bigbreak \noindent 
            \begin{bashcode}
            A := $(B)
            \end{bashcode}
            \bigbreak \noindent 
            Means expand $\$(B)$ now. In ordinary Makefiles, := is often safer and easier to reason about.
        \item \textbf{Appending (+=)}: You can add more text to a variable.
            \bigbreak \noindent 
            \begin{cppcode}
                CFLAGS = -Wall
                CFLAGS += -Wextra
                CFLAGS += -g
            \end{cppcode}
        \item \textbf{Conditional assignment (?=)}: 
            \bigbreak \noindent 
            \begin{bashcode}
            CC ?= cc
            \end{bashcode}
            \bigbreak \noindent 
            Sets CC to cc only if CC does not already have a value. This lets users override it from the command line
            \bigbreak \noindent 
            \begin{bashcode}
            make CC=clang
            \end{bashcode}
        \item \textbf{Automatic variables}: Make also has special variables that are only meaningful inside recipes.
            \begin{center}
                \begin{tabularx}{\textwidth}{@{}lX@{}}
                    \toprule
                    \textbf{Variable}&	\textbf{Meaning} \\
                    \midrule
                    \$@	&The target name.\\[2ex]
                    \$<	&The first prerequisite.\\[2ex]
                    \$$\land$	&All prerequisites, with duplicates removed.\\[2ex]
                    \$+	&All prerequisites, preserving duplicates and order.\\[2ex]
                    \$?	&Prerequisites newer than the target.\\[2ex]
                    \$*	&The stem matched by a pattern rule.\\[2ex]
                    \$\%	&The member name, only for archive targets like lib.a(file.o) \\
                    \bottomrule
                \end{tabularx}
            \end{center}
            \bigbreak \noindent 
            \begin{bashcode}
                editor: main.o kbd.o command.o
                    cc -o $@ $^
            \end{bashcode}
            \bigbreak \noindent 
            Expands roughly to 
            \bigbreak \noindent 
            \begin{bashcode}
            cc -o editor main.o kbd.o command.o
            \end{bashcode}
        \item \textbf{Directory file variants}: 
            \begin{center}
                \begin{tabularx}{\textwidth}{@{}lX@{}}
                    \toprule 
                    \textbf{Variable}&	\textbf{Meaning} \\
                    \midrule
                    \$(@D)	&Directory part of \$@.\\[2ex]
                    \$(@F)	&File part of \$@.\\[2ex]
                    \$(<D)	&Directory part of \$<.\\[2ex]
                    \$(<F)	&File part of \$<.\\[2ex]
                    \$($\land$D)	&Directory parts of \$$\land$.\\[2ex]
                    \$($\land$F)	&File parts of \$$\land$. \\
                    \bottomrule
                \end{tabularx}
            \end{center}




    \end{itemize}

    \unsect{Patterns and substitutions}
    \begin{itemize}
        \item \textbf{Patterns}: A pattern is a way to describe a family of filenames using \%
            \bigbreak \noindent 
            The \% character means: Match some nonempty or empty part of the filename, then reuse that matched part elsewhere.
            \bigbreak \noindent 
            That matched part is called the stem.
            \bigbreak \noindent 
            For example,
            \bigbreak \noindent 
            \begin{bashcode}
            %.o: %.c
                gcc -c $< -o $@
            \end{bashcode}
            \bigbreak \noindent 
            So if Make needs main.o, it matches:
            \bigbreak \noindent 
            \begin{bashcode}
            %.o
            \end{bashcode}
            \bigbreak \noindent 
            against 
            \bigbreak \noindent 
            \begin{bashcode}
            main.o
            \end{bashcode}
            \bigbreak \noindent 
            The stem is therefore \textbf{main}. Make then substitutes that stem in the prerequisite pattern \%.c
        \item \textbf{Pattern rules}: A pattern rule uses \% in the target. This is an implicit rule.
        \item \textbf{Static pattern rules}: A static pattern rule also uses \%, but only for a fixed list of targets:
            \bigbreak \noindent 
            \begin{bashcode}
                targets ...: target-pattern: prereq-patterns ...
                    recipe
                    ...
            \end{bashcode}
            \bigbreak \noindent 
            For example, 
            \begin{bashcode}
            objects = main.o lexer.o parser.o
                $(objects): %.o: %.c
                gcc -c $< -o $@
            \end{bashcode}
            \bigbreak \noindent 
            This applies only to the object files listed in the variable \texttt{objects}.
        \item \textbf{Substitutions}: In Make, substitution usually means transforming one list of words into another list of words.
            \bigbreak \noindent 
            The most common case is:
            \bigbreak \noindent 
            \begin{bashcode}
                sources = main.c lexer.c parser.c
                objects = $(sources:.c=.o)
            \end{bashcode}
            \bigbreak \noindent 
            This changes 
            \bigbreak \noindent 
            \begin{bashcode}
            main.c lexer.c parser.c
            \end{bashcode}
            \bigbreak \noindent 
            Into
            \bigbreak \noindent 
            \begin{bashcode}
            main.o lexer.o parser.o
            \end{bashcode}
            \bigbreak \noindent 
            So \$(sources:.c=.o) means: In each word of \$(sources), replace the suffix .c with .o.
        \item \textbf{Substitution references}: The basic form is
            \bigbreak \noindent 
            \begin{bashcode}
            $(variable:from=to)
            \end{bashcode}
            \bigbreak \noindent 
            It only replaces \textbf{from} when it appears at the \textbf{end} of a word.
        \item \textbf{Pattern substitution references}: There is also a pattern form using \%
            \bigbreak \noindent 
            \begin{bashcode}
            $(variable:pattern=replacement)
            \end{bashcode}
            \bigbreak \noindent 
            \begin{bashcode}
                sources = src/main.c src/lexer.c src/parser.c
                objects = $(sources:src/%.c=build/%.o)
            \end{bashcode}
            \bigbreak \noindent 
            Produces
            \bigbreak \noindent 
            \begin{bashcode}
            build/main.o build/lexer.o build/parser.o
            \end{bashcode}
        \item \textbf{patsubst}: The full function form is 
            \bigbreak \noindent 
            \begin{bashcode}
            $(patsubst pattern,replacement,text)
            \end{bashcode}
            \bigbreak \noindent 
            \begin{bashcode}
                sources = src/main.c src/lexer.c src/parser.c
                objects = $(patsubst src/%.c,build/%.o,$(sources))
            \end{bashcode}
            \bigbreak \noindent 
            Also produces
            \bigbreak \noindent 
            \begin{bashcode}
            build/main.o build/lexer.o build/parser.o
            \end{bashcode}
        \item \textbf{subst}: There is also a different function called subst:
            \bigbreak \noindent 
            \begin{bashcode}
            $(subst from,to,text)
            \end{bashcode}
            \bigbreak \noindent 
            Unlike patsubst, this performs a plain textual replacement
            \bigbreak \noindent 
            \begin{bashcode}
                text = hello.c old.c.backup
                result = $(subst .c,.o,$(text))
            \end{bashcode}
            \bigbreak \noindent 
            Result:
            \bigbreak \noindent 
            \begin{bashcode}
            hello.o old.o.backup
            \end{bashcode}
            \bigbreak \noindent 
            This replaces .c wherever it appears, not just at the end of a word.

    \end{itemize}

    \unsect{Make functions}
    \begin{itemize}
        \item \textbf{filename functions}: 
            \bigbreak \noindent 
            \begin{bashcode}
            $(dir names...)
            \end{bashcode}
            \bigbreak \noindent 
            Extracts the directory part.
            \bigbreak \noindent 
            \begin{bashcode}
            $(dir src/main.c) # src/
            \end{bashcode}
            \bigbreak \noindent 
            \begin{bashcode}
            $(notdir names...)
            \end{bashcode}
            \bigbreak \noindent 
            Extracts the file name without the directory.
            \bigbreak \noindent 
            \begin{bashcode}
            $(suffix names...)
            \end{bashcode}
            \bigbreak \noindent 
            Extracts the file suffix
            \bigbreak \noindent 
            \begin{bashcode}
            basename names...
            \end{bashcode}
            \bigbreak \noindent 
            Removes the suffix.
            \bigbreak \noindent 
            \begin{bashcode}
            $(addsuffix suffix, names...)
            \end{bashcode}
            \bigbreak \noindent 
            Adds a suffix to each word.
            \bigbreak \noindent 
            \begin{bashcode}
            $(addprefix prefix,names...)
            \end{bashcode}
            \bigbreak \noindent 
            Adds a prefix to each word.
            \bigbreak \noindent 
            \begin{bashcode}
            $(wildcard pattern...)
            \end{bashcode}
            \bigbreak \noindent 
            Finds existing files matching a shell-style pattern.
        \item \textbf{List functions}: Make treats many variables as whitespace-separated word lists.
            \bigbreak \noindent 
            \begin{bashcode}
            $(word n,text)
            \end{bashcode}
            \bigbreak \noindent 
            gets the $n$th word.
            \bigbreak \noindent 
            \begin{bashcode}
            $(words text)
            \end{bashcode}
            \bigbreak \noindent 
            counts words.
            \bigbreak \noindent 
            \begin{bashcode}
            $(firstword text)
            $(lastword text)
            \end{bashcode}
            \bigbreak \noindent 
            gets the first or last word
            \bigbreak \noindent 
            \begin{bashcode}
            wordlist start,end,text
            \end{bashcode}
            \bigbreak \noindent 
            gets a range of words
            \bigbreak \noindent 
            \begin{bashcode}
            $(filter pattern...,text)
            \end{bashcode}
            \bigbreak \noindent 
            keeps only words matching a pattern
            \bigbreak \noindent 
            \begin{bashcode}
            $(filter-out pattern..., text)
            \end{bashcode}
            \bigbreak \noindent 
            removes words matching a pattern.
            \bigbreak \noindent 
            \begin{bashcode}
            $(sort list)
            \end{bashcode}
            \bigbreak \noindent 
            sorts words and removes duplicates
        \item \textbf{Conditional functions}: Useful for configurable builds
            \bigbreak \noindent 
            \begin{bashcode}
            $(if condition, then-part, else-part)
            \end{bashcode}
            \bigbreak \noindent 
            \begin{bashcode}
            $(or a,b,c)
            \end{bashcode}
            \bigbreak \noindent 
            returns the first nonempty argument
            \bigbreak \noindent 
            \begin{bashcode}
            $(and a,b,c)
            \end{bashcode}
            \bigbreak \noindent 
            returns the last argument if all are nonempty; otherwise empty
        \item \textbf{Shell function}:
            \bigbreak \noindent 
            \begin{bashcode}
            $(shell command)
            \end{bashcode}
            \bigbreak \noindent 
            runs a shell command and captures its output. Be careful with \$(shell ...): it runs during Makefile expansion, not as a normal recipe command.
        \item \textbf{Strip}
            \bigbreak \noindent 
            \begin{bashcode}
            $(strip text)
            \end{bashcode}
            \bigbreak \noindent 
            removes leading / trailing whitespace and collapses internal whitespace.
        \item \textbf{eval}:
            \bigbreak \noindent 
            \begin{bashcode}
            $(eval text
            \end{bashcode}
            \bigbreak \noindent 
            takes expanded text and reads it as makefile code.



    \end{itemize}

    \pagebreak 
    \unsect{Conditionals}
    \begin{itemize}
        \item \textbf{Conditionals}: Make conditionals let the makefile itself choose which text to include before make runs recipes.
            \bigbreak \noindent 
            They are evaluated by make, not by the shell. That means they affect which variables, rules, prerequisites, and recipe lines exist in the parsed makefile.
            \bigbreak \noindent 
            \begin{bashcode}
                ifeq ($(VAR),value)
                    # used if VAR expands to value
                else
                    # used otherwise
                endif
            \end{bashcode}
            \bigbreak \noindent 
            \begin{bashcode}
                ifneq ($(VAR),value)
                    # used if VAR does not expand to value
                endif
            \end{bashcode}
            \bigbreak \noindent 
            \begin{bashcode}
                ifdef VAR
                    # used if VAR has a non-empty value
                endif
            \end{bashcode}
            \bigbreak \noindent 
            \begin{bashcode}
                ifndef VAR
                    # used if VAR is empty or undefined
                endif
            \end{bashcode}
    \end{itemize}

    \pagebreak 
    \unsect{User-defined functions}
    \begin{itemize}
        \item \textbf{functions}: 
            \bigbreak \noindent 
            \begin{bashcode}
            define function_name
                ...
            endef
            \end{bashcode}
            \bigbreak \noindent 
            Call with
            \bigbreak \noindent 
            \begin{bashcode}
            $(call function_name,arg1,arg2,...)
            \end{bashcode}
            \bigbreak \noindent 
            \begin{bashcode}
            $(i)
            \end{bashcode}
            \bigbreak \noindent 
            References the $i$th argument given to the function.
    \end{itemize}


    \pagebreak 
    \unsect{More on make}
    \begin{itemize}
        \item \textbf{Silencing commands}: In a make recipe, putting @ before a command tells make: Run this command, but do not print the command itself before running it.
            \bigbreak \noindent 
            \begin{bashcode}
            hello:
                @echo "hello"
            \end{bashcode}
        \item \textbf{More prefixes}: 
            \begin{bashcode}
                @command   # do not print the command
                -command   # ignore errors from the command
                +command   # run even under some recursive/dry-run conditions
            \end{bashcode}
            \bigbreak \noindent 
            They can be combined
            \bigbreak \noindent 
            \begin{bashcode}
            clean:
                @-rm missing-file
            \end{bashcode}
    \end{itemize}
